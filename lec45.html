Part 1 — Wireshark: Packet Capture & Analysis

01:15 What is Wireshark? A traffic analysis tool that captures packets on a live network interface and lets you inspect them layer by layer. The interface lists all available network adapters — the demo machine is connected to a campus Wi-Fi (WLAN0 interface).

Capturing Packets

Capture is started on the WLAN0 interface. 02:48 After opening Gmail, capture is stopped and the packet list is inspected. Most packets show protocol GVSP (GIGE Vision Streaming Protocol) — an artefact of the TeamViewer recording session running in the background. TCP packets are also present.

Inspecting a TCP Packet 04:11

Selecting one packet opens a three-pane view: packet list (top), packet details by layer (middle), raw bytes (bottom). This middle pane is the important one — it shows the full TCP/IP stack top-down, which is exactly how the course has been structured.

Wireshark's detail pane is a live demonstration of the encapsulation model. Each expandable section corresponds to one layer of the protocol stack.
Layer-by-layer breakdown of the captured packet
L7 App
TLS / SSL (Application Data)
Google sends data encrypted using TLS 1.2 over TCP. The payload is split into three TLS record blocks. The actual HTTP content is inside the encrypted data — Wireshark can see the TLS envelope but not the plaintext. The HTTP CONNECT header is visible because it is proxy tunnel metadata, not encrypted content.
Protocol: HTTP over TLS 1.2 | Encrypted application data
L4 TCP
Transmission Control Protocol
Visible fields: source port (8080 — campus HTTP proxy), destination port (54768), stream index, TCP segment length, sequence number, next sequence number, acknowledgement number, header length, flag bits, receiver advertised window size (rwnd), calculated window size, checksum, urgent pointer, TCP options.
Source port 8080 = IIT KGP campus HTTP proxy
L3 IP
Internet Protocol v4
Visible fields: version (4), header length, differentiated services field, total length, identification, flag bits (Don't Fragment set), fragment offset, TTL (Time to Live), protocol field (TCP = 6), header checksum, source IP, destination IP.
Src: 172.16.2.30 (campus HTTP proxy) → Dst: 10.146.58.130 (this machine's WLAN IP)
L2 Data Link
Ethernet / LLC + MAC
The data link layer has two sublayers visible in Wireshark. The Frame section (from MAC) contains: packet arrival time, epoch time, frame number, frame length, capture length, frame markers. The Ethernet section (from LLC) contains: source and destination MAC addresses, and the EtherType field identifying the upper-layer protocol (IPv4).
Encapsulation type: Ethernet (1)

TCP Three-Way Handshake Observable in Capture 09:24

Filtering for SYN packets reveals the handshake sequence: a SYN with sequence number 0 is sent, followed by a SYN-ACK from the server, followed by the client's ACK. The SYN flag bit is visibly set in the TCP header flags field of the first packet.

Part 2 — Mininet: Network Emulation Theory

Why Emulation? 10:29

The ideal way to learn networking is to run experiments on a real network. In practice this is unavailable (campus networks are restricted, private setups are expensive). The two available alternatives are simulation and emulation — and they are not the same thing.

Simulation
What it does
Models network behaviour in a hypothetical/virtual environment
Protocol stack
Synthetic — not the real kernel implementation
Accuracy
Lower — results may not reflect real-world performance
Examples
ns-3, OMNET++
Emulation (Mininet)
What it does
Runs actual kernel TCP/IP stack in virtualized instances
Protocol stack
Real — same implementation that runs on physical hardware
Accuracy
Higher — results are meaningful for real deployments
Examples
Mininet (uses Linux kernel namespaces + Open vSwitch)

How Mininet Maps Physical Components to Virtual Ones 12:53

Each physical component of a real network has a virtual counterpart inside Mininet:

Physical
Router / Switch
Mininet Equivalent
Virtual namespace (legacy) / Open vSwitch (SDN)
Physical
Host / Server
Mininet Equivalent
Virtual namespace (isolated protocol stack instance)
Physical
Network link (cable)
Mininet Equivalent
Virtual link (veth pair)
Namespace = virtual protocol stack instance. Creating a virtual host in Mininet means taking the full 5-layer TCP/IP stack implementation from the Linux kernel and spawning a virtualized instance of it. That instance behaves as an independent machine. This is analogous to how VirtualBox lets you run multiple OSes on one physical machine — Mininet lets you run multiple network stacks on one physical machine.

Open vSwitch (OVS) is the open-source tool that implements switching (L2) or routing (L3) functionality on top of the kernel's protocol stack. It is what enables SDN-capable virtual switches inside Mininet.

Mininet Topology Commands 17:16

Mininet is invoked from the terminal as mn (must be run as root via sudo). Topology is specified with the --topo flag:

mn --topo single,2 1 switch, 2 hosts (star)
mn --topo single,3 1 switch, 3 hosts (star)
mn --topo linear,3 3 switches in a line, 1 host per switch
mn --topo linear,2,3 --controller=remote --switch ovsk SDN topology: 2 switches, 3 hosts each, remote Ryu controller, OVS switches

The --controller=remote flag tells Mininet the SDN controller is running as a separate process (locally or remotely). The controller listens on port 6653 or 6633 (OpenFlow standard ports). If no controller is running when Mininet starts, it reports a connection failure but still creates the topology — packets just won't be forwarded until the controller connects.

Part 3 — Live SDN Demo: Ryu Controller + Mininet
🎬 Highly visual section. This portion is a live terminal demo. Watch from 20:04 to follow along. The notes below summarise the sequence of commands and observations.

Step 1 — Start Mininet Topology 20:09

Command run: sudo mn --topo single,3 --mac --controller remote --switch ovsk

This creates: 3 hosts (h1, h2, h3) connected to 1 OVS switch (s1) in a star topology. Mininet reports it cannot contact the remote controller at 127.0.0.1:6653 because no controller has been started yet. The topology is created anyway.

At the mininet> prompt, running h1 ping h2 returns "Destination Host Unreachable" — because without a controller, the SDN switch has no flow rules and does not know how to forward packets.

Step 2 — Configure OVS Switch for OpenFlow 1.3 23:46

In a separate terminal, the switch is configured to use OpenFlow 1.3:

sudo ovs-vsctl set bridge s1 protocols=OpenFlow13

This tells OVS switch s1 to speak OpenFlow version 1.3 with whatever controller connects to it.

Step 3 — Start Ryu Controller with simple_switch_13.py 23:46

./bin/ryu-manager --verbose ryu/app/simple_switch_13.py

Ryu connects to the OVS switch. The verbose log shows the controller consuming various OpenFlow events: EventOFPSwitchFeatures, EventOFPPacketIn, etc. Once connected, h1 ping h2 now succeeds.

Key Observation: First-Packet Latency 27:23

In the ping output, the first packet has significantly higher latency (~4.84 ms) compared to subsequent packets (~0.03–0.16 ms). This directly demonstrates the SDN packet-in → controller → flow rule installation cycle:

First packet path: h1 → switch (no matching rule) → OpenFlow PacketIn event generated → event sent to Ryu controller → controller computes forwarding rule → installs rule in switch via FlowMod → packet forwarded (was buffered at switch during this). All subsequent packets hit the installed rule directly — no controller involvement, hence much lower latency.

Every new host pair that hasn't communicated before triggers this first-packet overhead. The demo confirms this by running h2 ping h3 — again the first packet is slow, subsequent ones are fast.

Step 4 — The Ryu Application: simple_switch_13.py 30:06

This Python script is Ryu's default L2 learning switch application. It is what runs as the "network application" on the controller. Its structure:

Function / Handler
switch_features_handler
Purpose
Installs a default table-miss flow entry when switch connects. Sends unmatched packets to controller.
Function / Handler
add_flow
Purpose
Installs a new forwarding rule (FlowMod message) in the switch's flow table for a given match condition and action.
Function / Handler
_packet_in_handler
Purpose
Handles PacketIn events. Extracts src/dst MAC from ethernet header, learns src MAC → port mapping, determines output port (flood if unknown, specific port if known), calls add_flow to install rule, sends PacketOut to forward the buffered packet.
MAC learning logic: The controller maintains a table mac_to_port[dpid][src_mac] = in_port. When a packet arrives, it records which port the source MAC was seen on. If the destination MAC is in the table, it sends to that port. If not, it floods (sends to all ports except the in-port). This is exactly how a traditional L2 switch learns — the difference is the logic runs in the controller, not in the switch hardware.
Concept Summary

Wireshark

A live packet capture tool that shows every field of every layer of the TCP/IP stack for any captured packet. Useful for verifying protocol behaviour, debugging connections, and observing headers you've only seen in diagrams. TLS-encrypted traffic shows the TLS envelope but not the plaintext payload. The TCP 3-way handshake (SYN → SYN-ACK → ACK) is directly observable in a capture.

Mininet

A network emulator that runs inside a single Linux machine using kernel namespaces (virtual hosts) and Open vSwitch (virtual switches). Unlike simulators, it uses the real Linux TCP/IP stack, so behaviour matches real networks. Topologies are created with simple CLI commands and can be as complex as needed.

SDN in Mininet: The Flow

Mininet creates the data plane (virtual switches and hosts). A separate SDN controller (Ryu, OpenDaylight, Floodlight, etc.) connects to the OVS switches via OpenFlow. On first packet arrival, the switch sends a PacketIn event to the controller; the controller runs its application logic and installs a FlowMod rule in the switch; the switch forwards the buffered packet. Subsequent packets matching that flow are forwarded locally at the switch without controller involvement. The first-packet penalty is the measurable cost of the reactive control plane.
