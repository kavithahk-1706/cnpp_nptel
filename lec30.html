Lecture 30 HIGH PRIORITY

IPv6 Addressing

Why IPv4 ran out of steam, how IPv6 fixes it, header format, address types, neighbor discovery, mobility support, and migration strategies.

Motivation

Why Do We Need IPv6?

IPv4 uses a 32-bit address space. Theoretically that's 232 addresses, but a huge chunk are reserved (loopback, broadcast, private ranges, subnet addresses). The usable pool is far smaller. With desktops, laptops, phones, IoT sensors, smart appliances all demanding unique addresses, the IPv4 pool hit saturation. Workarounds like NAT and CIDR bought time but didn't solve the core problem.

IoT tipping point: Every sensor on a smart fridge, AC unit, light switch, or hospital monitor needs a globally reachable address. IPv4 simply cannot accommodate this scale.

Problems with IPv4 (Beyond Address Space)

Address Exhaustion
Even with CIDR, the 32-bit space is saturated. The drop in address growth in recent years isn't declining demand — it's a hard ceiling.
QoS Vaguely Defined
IPv4 was designed for plain data traffic. Modern multimedia (VoIP, live streaming, video calls) needs delay-sensitive, prioritised delivery — not supported natively.
Mobility Unmanageable
IPv4 assumes fixed devices. Mobile IP was added as a patch but performs poorly in real deployments. Not a native feature.
No Native Security
Security was a secondary concern at IPv4's inception. TLS, IPSec etc. are bolt-ons layered above IPv4 — not integrated. Adds complexity and management overhead.
Because QoS, mobility, and security all had to be patched on externally, the protocol became bloated and unmanageable at internet scale.
IPv6 Design

IPv6 Features

The IPv6 draft proposal arrived in December 1998, but the protocol was only formally standardised in July 2017 — nearly 20 years of iteration and refinement.

128-bit Address Space
2128 possible addresses — enough for every grain of sand on Earth to have billions of IPs. Exhaustion is effectively off the table.
Global Hierarchical Addressing
Unlike IPv4's broken hierarchy (forced to abandon classful structure for CIDR), IPv6 preserves hierarchical structure globally, enabling optimised prefix-based routing tables.
Auto-configuration
DHCP functionality is built directly into IPv6 via stateless address autoconfiguration (SLAAC). No separate DHCP protocol required — a device can self-assign an address upon connecting.
Built-in Security
Authentication and encryption are native IPv6 features via extension headers, not patched on via separate protocols.
QoS Support
Traffic class and flow label fields in the IPv6 header enable native classification and prioritisation of traffic flows (e.g., video vs. background data).
IPv4 Compatibility
Backward compatibility mechanisms (dual stack, tunneling, header translation) allow gradual migration — you don't have to flip a switch on the entire internet overnight.
Header Structure

IPv6 Header Format

The mandatory IPv6 header is fixed and simpler than IPv4's variable-length header. Optional functionality is offloaded to extension headers, keeping the base header lean for fast processing.

Field Size Purpose
Version 4 bits Always 6 for IPv6
Traffic Class 8 bits QoS — marks packet priority/type
Flow Label 20 bits Identifies a specific traffic flow for QoS handling at routers
Payload Length 16 bits Length of data following the header
Next Header 8 bits Type of the next header (extension header or transport protocol)
Hop Limit 8 bits Equivalent to IPv4 TTL — decremented at each hop; packet dropped at 0
Source Address 128 bits Sender's IPv6 address
Destination Address 128 bits Recipient's IPv6 address

Extension Headers

Every IPv6 packet has exactly one mandatory base header. Beyond that, optional extension headers can be chained together using the Next Header pointer — each header points to the next. This keeps the base header fixed-size while allowing arbitrary extensibility.

Chain structure: Base Header → (optional) Hop-by-Hop Options → Routing Header → Fragment Header → Authentication Header → TCP Header + Data
Hop-by-Hop Options
Instructions for every router the packet passes through — how should each hop handle this packet?
Routing Header
Enables source routing — the sender can embed the exact route the packet must follow to reach the destination.
Fragment Header
Carries fragmentation info when an IPv6 packet is split into pieces. Note: only the source can fragment in IPv6 — routers cannot.
Authentication Header
Security data for verifying the sender's identity — authentication built natively into the packet structure.
Address Format

IPv6 Address Notation

A 128-bit IPv6 address is written as 8 groups of 16-bit hexadecimal numbers, separated by colons.

Full form → FE80:0000:0000:0000:0001:0800:23E:F5DB
Drop leading zeros → FE80:0:0:0:1:800:23E:F5DB
Compress consecutive zero groups with :: → FE80::1:800:23E:F5DB
:: can only appear once. If you use it twice, it becomes impossible to determine how many zero groups each :: represents. The missing groups are inferred from the total count — only unambiguous if :: is used exactly once.
Address Types

Address Space Allocation by Prefix

IPv6 divides its address space by examining the leading bits of an address (the prefix), similar to CIDR.

Allocation Prefix (binary) Start (hex) Prefix Length Fraction of Space
Reserved 0000 0000 0::/8 8 bits 1/256
Reserved for NSAP 0000 001 200::/7 7 bits 1/128
Reserved for IPX 0000 010 400::/7 7 bits 1/128
Aggregatable Global Unicast 001 2000::/3 3 bits 1/8
Link-Local Unicast 1111 1110 10 FE80::/10 10 bits 1/1024
Site-Local Unicast 1111 1110 11 FEC0::/10 10 bits 1/1024
Multicast 1111 1111 FF00::/8 8 bits 1/256
IPv6 has no broadcast. Broadcast is replaced by multicast (FF00::/8) and anycast. This is a significant efficiency improvement — no device is forced to process traffic not destined for it.

Global Unicast Address Format

The globally routable unicast addresses (2000::/3) are structured in three parts:

Global Routing Prefix n bits
Subnet ID m bits
Interface ID 128 − n − m bits

The Global Routing Prefix is assigned to a site (e.g., an ISP or organisation) by routing authorities, structured hierarchically so the entire internet's routing table can be kept manageable. The Subnet ID identifies a specific subnet within that site. The Interface ID identifies the specific network interface (analogous to the host part in IPv4).

In CIDR notation, the prefix length covers Global Routing Prefix + Subnet ID combined. So /48 means 48 bits of global+subnet prefix, leaving 80 bits for interface IDs within that subnet.
ICMPv6 Feature

Neighbor Discovery Protocol (NDP)

Neighbor Discovery is the IPv6 replacement for ARP. In IPv4, ARP resolves an IP address to a MAC address via a broadcast — every device on the subnet has to process that broadcast. IPv6 replaces this with a targeted mechanism using ICMPv6.

ARP (IPv4)
Method
Broadcasts query to every device on the subnet
Scope
All devices must receive and process
Protocol
ARP (separate from ICMP)
NDP (IPv6)
Method
Sends Neighbor Solicitation to a specific solicited-node multicast address
Scope
Only the target node (and others sharing that multicast group) processes it
Protocol
ICMPv6 (integrated into IPv6)

Neighbor Solicitation (ICMPv6 Type 135)

When node A wants to find the MAC address of node B, it sends a Neighbor Solicitation message. The destination is not a broadcast address — it's the solicited-node multicast address derived from B's IPv6 address. Only node B (and any other node sharing that solicited-node multicast group) receives and processes it.

The message contains: the target IPv6 address (whose MAC we want) and the source link-layer address (A's own MAC).

Neighbor Advertisement (ICMPv6 Type 136)

Node B responds with a Neighbor Advertisement containing its IPv6 address and its MAC address. Nodes also periodically send unsolicited Neighbor Advertisements to maintain link connectivity.

R flag Sender is a router
S flag This advertisement is a response to a solicitation
O flag Override — source must update its neighbor cache with this new info
Hint — visualise this: The solicited-node multicast concept is best understood by tracing a packet exchange between two nodes on a link. See the worked example at ~25:18 in the lecture video.
Mobility

IPv6 Mobility Support

Unlike IPv4 where mobile support required an external foreign agent and performed poorly, IPv6 builds mobility support into the protocol natively via extension headers.

Temporary Care-of Address
When a mobile node leaves its home network, it gets a temporary address at the new location. Its home address is stored in the IPv6 Destination Options extension header, so traffic destined for the home address can still reach it.
No Foreign Agent Needed
IPv4 mobility required a designated foreign agent router to map old and new addresses. IPv6 uses NDP and address autoconfiguration instead — a node connects to any subnet and self-configures. Clean and protocol-native.

The mobile station lists routing information in the Routing extension header so packets can be tunneled and forwarded to the correct location without involving an external agent.

Migration Strategies

Migrating from IPv4 to IPv6

A hard cutover from IPv4 to IPv6 is impossible — the majority of existing infrastructure runs IPv4. Three mechanisms enable gradual coexistence and migration:

1. Dual Stack
A single machine runs both IPv4 and IPv6 protocol stacks simultaneously. Communicating with an IPv4 host? Use the IPv4 stack. Communicating with an IPv6 host? Use the IPv6 stack. No translation needed — just choose the right stack at runtime.
2. Tunneling
IPv6 packets are encapsulated inside IPv4 packets (or vice versa) when traversing a network that only supports the other version. The IPv4/IPv6 header wraps the other as payload — the inner header is read by the final destination.
3. Header Translation
An IPv4 header is translated into an IPv6 header (or vice versa) by converting field values. Requires address translation as well — specific rules govern how IPv4 and IPv6 addresses map to each other.

Address Translation Rules (Header Translation)

IPv6 → IPv4 → Take the low-order 32 bits of the IPv6 address as the IPv4 address
IPv4 → IPv6 → Prepend 80 zero bits + 16 one-bits (::FFFF/96 prefix) to the 32-bit IPv4 address
Example: IPv4 202.141.80.20 → IPv6 ::FFFF:CA8D:5014 (202=CA, 141=8D, 80=50, 20=14 in hex). Going the other way: IPv6 FE80:2381::1:0800:23E:F5DB — take the last 32 bits (FE=254, 80=128, 23=35, 81=129) → IPv4 254.128.35.129.
Summary

Key Concepts at a Glance

IPv4 Problems
Address exhaustion (32-bit), no native QoS, no native mobility, no native security. All three added as external patches, making the protocol unmanageable at scale.
IPv6 Address Format
128 bits, written as 8 colon-separated hex groups. Leading zeros droppable. Consecutive zero groups collapsible to :: (exactly once).
Header Design
Fixed mandatory header + chained optional extension headers. Traffic Class + Flow Label = native QoS. Next Header pointer enables unlimited extension chaining.
Neighbor Discovery
IPv6's ARP replacement. Uses ICMPv6 Neighbor Solicitation (type 135) and Advertisement (type 136). Targets a solicited-node multicast address instead of broadcasting — far more efficient.
Address Types
Global unicast (2000::/3, 1/8th of space), link-local (FE80::/10), site-local (FEC0::/10), multicast (FF00::/8). No broadcast in IPv6.
Migration
Three strategies: dual stack (run both), tunneling (encapsulate one inside the other), header translation (convert between formats with address mapping rules).
