Internet QoS – II: Basic QoS Architecture

Prof. Sandip Chakraborty  ·  IIT Kharagpur

▶ Watch Lecture
Application Classes Based on QoS

Different applications have fundamentally different QoS requirements. Four broad application classes are recognised based on their tolerance for delay, jitter, loss, and bandwidth variation.

Strict Requirements
CBR Constant Bit Rate — data must arrive at a fixed, steady rate. Receiver processes at that same rate. Examples: VoIP, telephone applications. Any rate mismatch causes degradation.
RT-VBR Real-Time Variable Bit Rate — bit rate may fluctuate, but a hard deadline on maximum delay must be met. Missing the deadline counts as a transfer failure. Examples: video conferencing, live streaming.
Relaxed Requirements
NRT-VBR Non-Real-Time Variable Bit Rate — no strict real-time deadline; buffering is supported. A loose bound on delay and packet loss still applies. Examples: on-demand video streaming, IPTV.
ABR / BE Available Bit Rate / Best Effort — utilises whatever bandwidth the network currently has; no strict QoS guarantees required. Examples: file transfer, general data downloads.
Basic QoS Architecture — Pipeline Overview

The end-to-end QoS architecture is a pipeline of five sequential processing stages applied to every packet as it traverses the network. All five stages are implemented inside Layer 3 devices (routers). Each stage inspects packet properties and applies corrective measures to uphold QoS guarantees.

Visual reference: The Cisco Basic QoS Architecture diagram (Refer Slide Time: 04:06) is the canonical illustration of this pipeline. Cross-reference it for the full end-to-end flow.
Stage 1 → Admission Control — decide whether to admit a new flow.
Stage 2 → Classification & Marking — identify the traffic class; tag the packet.
Stage 3 → Policing & Markdown — monitor flows; drop or re-mark violating packets.
Stage 4 → Scheduling (Queuing & Dropping) — prioritise packets into output queues.
Stage 5 → Traffic Shaping — regulate output rate; smooth jitter.
Stage 6 → Link-Specific Mechanisms — apply technology-specific QoS (e.g., Wi-Fi channel access prioritisation).
Stage-by-Stage Detail

Stage 1 — Admission Control

Admission control is the gatekeeper of the network. A new flow is admitted only if the network can satisfy the QoS requirements of all existing flows plus the new flow simultaneously. If the network cannot guarantee this, the new flow is blocked.

Real-world analogy: The "all lines are busy, please dial after sometime" message on a cellular network is admission control in action — the provider has exhausted resources available to meet minimum call quality, so new calls are rejected rather than admitted and degraded.

Stage 2 — Classification & Marking

Each incoming packet is examined and assigned to one of the defined QoS traffic classes. The assignment is then written into the packet header so all downstream nodes can act on it without re-inspecting payload content.

IP Type of Service (ToS) Field 18:13

Classification is encoded using the 8-bit Type of Service (ToS) field in the IPv4 header. The field is structured as follows:

Bits Field Purpose
3 bits IP Precedence Identifies the broad traffic class (8 levels, 0–7)
4 bits Priority within class Relative priority among packets of the same class (e.g., voice vs. video within the Critical class)
1 bit Reserved —

Stage 3 — Policing & Markdown

Traffic policing monitors flows in transit and takes corrective action when packets are found to violate their QoS contract. The primary action is early packet drop: if an intermediate router can determine that a packet cannot be delivered within its deadline, it drops the packet immediately rather than forwarding it uselessly.

Example (Refer Slide Time: 08:56): End-to-end delay requirement = 10 ms. A packet arrives at an intermediate router having already accumulated 9 ms of delay. The router determines that delivering the packet to the destination within the remaining 1 ms is impossible. The packet is dropped immediately — forwarding it would consume bandwidth and processing resources at the destination with no benefit.

Service Level Agreements (SLAs)

A Service Level Agreement (SLA) is a formal contract between a customer and a network service provider that specifies the guaranteed QoS parameters for a given traffic type (e.g., maximum delay, available bandwidth, packet loss tolerance). SLAs are configured into edge/gateway routers and govern how packets are treated end-to-end.

Familiar example: Broadband subscription packages are SLAs. A "1 Mbps downlink / 256 Kbps uplink" plan is a service level agreement — the provider commits to those rates and the policing mechanisms enforce adherence.

Traffic policing enforces the SLA by monitoring flows and marking violating packets for reduced priority or immediate drop.

Stage 4 — Traffic Scheduling (Queuing & Dropping)

Scheduling determines the order in which packets are transmitted on an outgoing link, based on the markings assigned during classification and the policies applied during policing. The scheduler maintains multiple queues at each interface, one per traffic class, and services them according to a scheduling policy.

Example: Priority Queuing 11:53

Packets are sorted into queues by class on arrival. The scheduler always services the highest-priority non-empty queue first, descending to lower-priority queues only when all higher queues are empty.

Red Queue — Priority 1 (Highest) → Voice traffic (CBR). Served first unconditionally.
Blue Queue — Priority 2 → Video traffic (RT-VBR). Served only when red queue is empty.
Yellow Queue — Priority 3 (Lowest) → Data / best-effort traffic. Served only when both red and blue queues are empty.
Priority queuing is one of several scheduling disciplines. Others include Custom Queuing, Weighted Fair Queuing (WFQ), and Fair Weighted Fair Queuing. These are discussed in subsequent lectures.

Stage 5 — Traffic Shaping

Traffic shaping regulates the outgoing traffic rate, decoupled from the incoming rate. A shaper absorbs bursty input traffic into a buffer and releases packets at a controlled, constant rate, thereby smoothing jitter on the outgoing link.

Traffic Policing
Drops packets that exceed the agreed rate or deadline. Acts reactively — violating packets are discarded immediately. Rate after policing can still be irregular, but peaks are clipped.
Traffic Shaping
Buffers and re-times packets to produce a smooth, constant output rate. Does not drop by default — delays instead. Effective only when the mean input rate is at or below the target rate.
Why both are needed: If the mean traffic rate already exceeds the SLA bound, shaping alone cannot enforce compliance — there is simply too much data. Policing must first drop the excess; then shaping can smooth the remainder into a well-regulated output stream.
Concept Summary

Key Takeaways

Four QoS application classes — CBR, RT-VBR, NRT-VBR, ABR/Best-Effort — differ in their strictness on delay, jitter, loss, and bandwidth. Understanding which class an application belongs to determines which QoS mechanisms must be applied.

The Basic QoS Architecture is a six-stage pipeline implemented inside routers: Admission Control → Classification & Marking → Policing & Markdown → Scheduling → Traffic Shaping → Link-Specific Mechanisms. Every packet traverses this pipeline at each router hop.

Admission Control is a hard gate: it blocks new flows if existing flows' QoS cannot be maintained. It does not degrade existing flows to accommodate new ones.

The IP ToS field (8 bits) carries the classification in-band: 3 bits encode IP Precedence (traffic class, 8 levels) and 4 bits encode intra-class relative priority.

SLAs are the contractual basis for policing. Traffic policing enforces SLAs by monitoring flows and dropping or re-marking packets that violate the agreed parameters.

Policing vs. Shaping: Policing drops violating packets (hard enforcement); shaping delays and buffers them to produce a smooth output rate (soft regulation). Both are required together in real deployments.
