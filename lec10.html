Week 10 HIGH YIELD

Application Layer — V
SMTP & SNMP

Two more application-layer protocols: SMTP for email delivery and MIME for non-ASCII content, plus POP3/IMAP4 for mail access. Then SNMP — the protocol that silently keeps your entire network from dying.

Part 1 — SMTP

Simple Mail Transfer Protocol — Basics

MCQ anchor: SMTP runs over TCP, default port 25. Defined in RFC 821 (1982, Jon Postel). Message format defined in RFC 822/2822.

SMTP is a client-server protocol. The server listens at port 25. The goal: transfer mail reliably between mail clients across the internet.

Two Core Components

User Agent (UA)
Prepares the message and encloses it in an envelope. Examples: Thunderbird, Eudora. Sits at the user-facing end.
Mail Transfer Agent (MTA)
Transfers mail across the internet. Examples: Sendmail, Exim, Postfix. Acts as both SMTP client and server. Analogous to the postal service.

MTAs can also act as relays — receive a mail and forward it onward toward the destination MTA. A mail gateway accepts mail in non-SMTP formats and converts it to SMTP before forwarding.

Email Structure

Envelope → Sender address + receiver address + routing metadata
Header → From, To, Subject, Date — human-readable metadata
Body → Actual message content (the payload)

SMTP Keywords (Commands)

Keyword Purpose Priority
HELO Sender identifies itself — sends its host domain name core
MAIL FROM Email address of the sender core
RCPT TO Email address of the intended recipient core
DATA Start of message body; terminated by a lone dot on a line core
QUIT Formal connection termination by client core
VRFY Verify a username extra
EXPN Expand a mailing list extra
RSET / NOOP / TURN / HELP Reset session / no-op / reverse roles / help extra

SMTP Status Codes

Pattern to remember: 2xx = OK, 3xx = more info needed, 4xx = temporary error (retry), 5xx = permanent failure (bad user/address)
Code RangeMeaning
2xxSuccess
3xxCommand accepted, more information required
4xxCommand rejected — temporary error (client should retry)
5xxCommand rejected — permanent failure (bad user etc.)

SMTP Session Flow

Step 1→TCP connection established (client → server port 25)
Step 2→Server sends 220 service ready
Step 3→Client sends HELO → server responds 250 OK
Step 4→Client: MAIL FROM:<addr> → server: 250 OK
Step 5→Client: RCPT TO:<addr> → server: 250 OK
Step 6→Client: DATA → server: 354 start input → client sends headers, blank line, body, ends with lone .
Step 7→Server: 250 mail accepted
Step 8→Client: QUIT → server: 221 service closing
MCQ trap: You can replicate an SMTP session manually using Telnet (e.g., telnet mailserver 25). Same idea as doing HTTP via Telnet on port 80.
Part 2 — MIME

Multipurpose Internet Mail Extensions

Basic SMTP only handles 7-bit ASCII. MIME is an extension that enables sending non-ASCII data (images, audio, video, executables) by encoding it into 7-bit NVT ASCII that SMTP can carry.

Where MIME lives: MIME headers sit between the email header and the body. They describe what the body contains and how it's encoded.

MIME Header Fields

FieldPurpose
MIME-VersionIdentifies message as MIME-formatted
Content-TypeType/subtype of the data (text/plain, image/jpeg, etc.)
Content-Transfer-EncodingHow data is encoded for transmission
Content-IDUnique identifier for the content
Content-DescriptionHuman-readable description

MIME Content Types

text application image audio video multipart

Transfer Encodings

7-bit 8-bit binary base64 quoted-printable
Note: Some mail servers block certain content types (executables, large videos). Those are application-level restrictions on top of SMTP — MIME itself just defines the encoding format.
Part 3 — Mail Access

POP3 vs IMAP4

SMTP delivers mail to the mail server. But how does the user retrieve it? That's what mail access protocols are for. MTA delivers to the mailbox; user uses POP3 or IMAP4 to pull it out.

POP3 — Post Office Protocol v3
StyleSimple download model
DownloadPulls all messages from server to client
After DLUser can delete or keep on server
HeadersMust download full message to read
LocationPrimarily single-device use
GoalMinimize server resource usage
IMAP4 — Internet Message Access Protocol v4
StyleFull mailbox management on server
DownloadCan download partial emails
Pre-DLCan check headers before downloading
SearchSearch email for a string before downloading
LocationAccess from any device/location
MailboxesCreate, delete, rename mailboxes on server
MCQ anchor: IMAP4 key differentiators — check headers before download, search before download, partial email download, multi-device access, server-side mailbox management.
Part 4 — SNMP

Simple Network Management Protocol

While most protocols are about moving data, SNMP is about managing the network itself. Two core functions: monitor (collect info to ensure health) and control (act on that info to fix/change things).

RFC 1157 defines the components. SNMP runs on UDP (not TCP — manager queries are fast, stateless).

SNMP Architecture — Components

SNMP Agent → Software running on a network device (router, host, printer). Maintains a local database of config/state. Reports to the manager.
SNMP Manager → Application program (management station) that queries or modifies the agent's database. Network admin uses this.
MIB → Management Information Base — text file on the agent describing what managed objects exist and their structure, using ASN.1 syntax.
SNMP Protocol → The application-layer protocol used by agents and managers to exchange data. Runs on UDP port 161.

MIB — Management Information Base

MIB is a text file that describes managed objects using ASN.1 (Abstract Syntax Notation 1) — a formal language for describing data structures. Each managed object is assigned an OID (Object Identifier) — a sequence of integers separated by dots (e.g., 1.3.6.1.2.1.1.1) that uniquely identifies it. MIB-2 (RFC 1213) defines the standard managed objects for TCP/IP networks.

Manager sends the OID to the agent to request a specific piece of data. Think of OIDs as a unique address for each piece of network info.

SNMP Operations

Operation Direction Purpose
Get-request Manager → Agent Request value of one or more objects
Get-next-request Manager → Agent Request next object in lexicographical OID order
Set-request Manager → Agent Modify the value of one or more objects
Get-response Agent → Manager Response to Get-request, Get-next-request, or Set-request
Trap Agent → Manager Unsolicited notification — agent pushes alert on a specific event without waiting for a query
MCQ trap: Traps are the only SNMP messages initiated by the agent without a request from the manager. Everything else is request-response. Port 161 for request/response; traps go to port 162.

SNMP Versions

Version Year PDU Counter Security
SNMPv1 1990 32-bit Community string as plain text password — no encryption. Essentially none.
SNMPv2c 1996 64-bit Attempted to fix security, failed. Still community-based (the "c").
SNMPv3 2002 — Full security: Integrity (packet not tampered), Authentication (valid source), Privacy/Confidentiality (encrypted). CIA triad.
MCQ anchor — SNMPv3 security triad: Integrity + Authentication + Privacy (confidentiality). Many agents and managers support all 3 versions simultaneously.
