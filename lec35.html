Lecture 35 MEDIUM PRIORITY

Internet QoS — V: IntServ & DiffServ Architectures

Prof. Sandip Chakraborty  ·  IIT Kharagpur  ·  CNPP Week 7

▶ Watch Lecture
Overview

This lecture concludes the Internet QoS series by examining the two architectural approaches for providing QoS over the internet: Integrated Service (IntServ), which offers hard guarantees on a per-flow basis, and Differentiated Service (DiffServ), which offers class-based best-effort guarantees. The lecture covers the Internet Service Architecture (ISA), the Resource Reservation Protocol (RSVP), the DiffServ domain model, bandwidth brokers, traffic conditioning, and per-hop behaviours (PHBs).

Part 1 — Integrated Service Architecture (IntServ / ISA)

Internet Service Architecture (ISA)

The Internet Service Architecture (ISA) is the framework for delivering IntServ QoS. It operates on four principal mechanisms within each router.

Admission Control
Determines whether the network can support a new flow's QoS requirements before allowing it entry. New flows are admitted only if resources can be reserved end-to-end via RSVP. If not, the flow is rejected — analogous to a telephone exchange playing "all lines are busy."
QoS-Aware Routing
Routing decisions factor in the current load on each router. Overloaded routers are avoided in favour of alternate paths. This is an extension of standard routing algorithms to incorporate QoS parameters.
Queuing
Different queuing strategies (priority queuing, WFQ, custom queuing) are applied per outgoing interface to honour diverse flow requirements. Queues are interface-specific — each outgoing interface maintains its own set of queues.
Discard Policy
Congestion avoidance mechanisms such as Random Early Detection (RED) are applied to proactively drop packets before queues fill, maintaining the required QoS levels.

ISA Inside a Router

The ISA implementation in a router integrates routing and QoS into a unified forwarding pipeline.

Control Plane: Routing protocols populate the Routing Database. The Reservation Protocol (RSVP) and Admission Control Protocol interact to populate the Traffic Control Database. The Management Agent oversees traffic shaping and policing policies.
Data Plane (Forwarding): An incoming packet first passes through the Classifier and Route Selection module, which assigns the packet to a traffic class and selects an outgoing interface. The Packet Scheduler then places the packet into the appropriate queue — either a QoS queue or the best-effort queue — and transmits it according to the active scheduling policy.

Resource Reservation Protocol (RSVP) 10:42

RSVP is a network control protocol, not a routing protocol, though it operates in conjunction with IP routing. It enables a data receiver to request end-to-end QoS for a flow across all routers along the path.

Key distinction: RSVP is receiver-initiated. The destination host sends the reservation request, which propagates upstream toward the source, reserving resources at each hop along the way.

An RSVP daemon runs at the source host, the destination host, and every intermediate router in the end-to-end path. These daemons coordinate with each other to reserve bandwidth and buffer space at each node. This inter-daemon coordination is precisely what makes IntServ difficult to scale to the open internet.

RSVP Reservation Procedure

When a reservation request arrives at a router, it is evaluated by two decision modules in parallel.

Admission Control Module
Checks whether the router has sufficient free resources (bandwidth, buffer) to satisfy the requested QoS. If resources are available, the flow is admitted; otherwise it is dropped.
Policy Control Module
Checks whether the requesting user has administrative permission (i.e., a valid Service Level Agreement with the ISP) to receive elevated QoS. Without an SLA, packets are treated as best-effort regardless of the request.

If both checks succeed, the reservation parameters are written into the packet classifier and link layer interface. If either check fails, RSVP returns an error notification to the requesting application process.

RSVP Reservation Model — Flow Descriptor

Each RSVP reservation request consists of a flow descriptor comprising two components.

Component Configures Purpose Contains
flowspec Packet Scheduler Specifies desired QoS level Rspec (desired QoS) + Tspec (data flow description)
filterspec Packet Classifier Defines which packets belong to the flow Classifier rules + queuing policy selection

flowspec Structure

The flowspec is a C struct containing the following numeric parameters, all of type ULONG.

Rspec fields (scheduler parameters): TokenRate, TokenBucketSize
Tspec fields (flow profile): PeakBandwidth, Latency, DelayVariation, ServiceType, MaxSduSize, MinimumPolicedSize

These numeric values together encode the SLA, and are used to configure intermediate router queues end-to-end.

Problems with RSVP / IntServ 21:15

Scalability failure: The RSVP daemon must maintain per-flow state at every intermediate router. On the global internet — millions of routers, billions of short-lived flows — this per-flow state and per-flow processing overhead is computationally infeasible. This fundamental limitation motivates the DiffServ architecture.
Part 2 — Differentiated Service Architecture (DiffServ)

DiffServ Overview 21:46

DiffServ is a coarse-grained, class-based mechanism for traffic management. Instead of per-flow reservations, it classifies traffic into a small, fixed set of network-wide service classes and applies different forwarding treatments per class.

Key design shift: In IntServ, classifier behaviour is user-specific and configured via filterspec. In DiffServ, traffic classes are standardised and fixed; classification is performed by reading a 6-bit DSCP (Differentiated Services Code Point) field embedded in the 8-bit DS field of the IP header. No filterspec is needed.
Per-Hop Behaviour (PHB)
DiffServ-aware routers implement per-hop behaviours — standardised forwarding rules associated with each DSCP class. Each router independently applies the PHB for whatever class a packet belongs to, requiring no global coordination.
DiffServ (DS) Domain
A group of routers that implement a common, administratively defined set of DiffServ policies. The internet is partitioned into DS domains (e.g., local ISPs, tier-1 ISPs), each enforcing DiffServ within its boundary.

DiffServ Architecture — Multi-Domain Model 24:21

A packet traversing the internet crosses multiple DS domains (e.g., DS-1 local ISP → DS-2 tier-1 ISP → DS-3 local ISP). At each domain boundary, the edge router evaluates how much of the end-to-end SLA has already been consumed and treats the packet accordingly.

Example: SLA specifies end-to-end delay ≤ 30 ms. A packet arrives at DS-2 having already incurred 20 ms of delay. DS-2 now knows it must deliver the packet within 10 ms and raises its priority relative to packets with looser remaining budgets.

Bandwidth Broker 26:54

Inter-domain QoS coordination in DiffServ is handled not by per-router RSVP daemons but by Bandwidth Brokers — one per DS domain.

A Bandwidth Broker (RFC 2638) is an agent with knowledge of an organisation's QoS priorities and policies. It allocates QoS resources within its domain and communicates with adjacent Bandwidth Brokers to negotiate end-to-end resource allocation across domain boundaries via purely bilateral agreements.

This is the key scalability advantage of DiffServ: instead of all-router coordination (IntServ), only domain-level brokers coordinate, massively reducing signalling overhead.

SLA and TCA 28:43

Service Level Agreement (SLA)
A set of parameters defining the service offered to a traffic stream by a DS domain. Specifies the QoS the customer will receive (e.g., delay bounds, loss bounds) for a given traffic class.
Traffic Conditioning Agreement (TCA)
A set of parameters specifying classifier rules and a traffic profile. Defines which of the fixed service classes a customer's packets will be assigned to, and what that customer must pay. Higher class = higher cost = higher priority.

Traffic Classification and Conditioning 29:54

At each DS domain boundary, boundary nodes (edge routers) perform traffic classification and conditioning. They map incoming packets to forwarding classes and verify that the traffic conforms to the customer's SLA.

Traffic conditioning is implemented via four functional components applied to classified packet streams.

Component Function
Meter Measures the traffic stream against its traffic profile; estimates how much QoS has already been consumed and how much remains to be satisfied in subsequent domains.
Marker Dynamically assigns or re-assigns the DSCP priority value based on the meter's output. Packets behind on their delay budget are promoted; packets with slack are demoted.
Shaper Delays packets that exceed the traffic profile to smooth the traffic rate and bring it into compliance with the TCA.
Dropper Discards excess packets that cannot be shaped and that exceed the agreed profile. Excess packets may alternatively be re-marked as low priority rather than dropped outright.
Why DiffServ is "best effort" not "guaranteed": Excess packets are marked as low priority or dropped. The network makes no absolute commitment that every packet will meet its SLA target. If a domain is overloaded, QoS degrades gracefully rather than being enforced unconditionally.

Per-Hop Behaviours (PHBs) 32:53

DiffServ defines four standardised PHBs, each associated with a queuing mechanism.

PHB Full Name Purpose Typical Queue
Default Default PHB Best-effort service for all traffic not classified otherwise Priority Queue
EF Expedited Forwarding Low-loss, low-latency forwarding for real-time traffic (e.g., VoIP) Priority Queue
AF Assured Forwarding Assured delivery under prescribed bandwidth conditions Custom Queue
CS Class Selector Backward-compatible with legacy IP Precedence field; ensures fairness WFQ

Working Steps of a DS Domain 33:53

  1. The source negotiates an SLA with its ISP.
  2. The source sends a request to its first-hop router.
  3. The first-hop router forwards the request to the local Bandwidth Broker.
  4. The Bandwidth Broker accepts or rejects the request based on available resources and the SLA.
  5. On acceptance, the source or first-hop router marks the DSCP field and begins transmitting packets.
  6. At each DS domain boundary, the edge router verifies SLA compliance and performs policing; excess packets are discarded or re-marked as low priority.
  7. Core (interior) routers simply read the DSCP field and apply the corresponding PHB — no admission logic required.
Concept Summary

IntServ vs DiffServ — The Core Trade-off

IntServ provides mathematically hard QoS guarantees by reserving resources per-flow on every router along the path. This is theoretically ideal but operationally unscalable for the global internet due to the per-flow state that every router must maintain. DiffServ sacrifices the granularity of per-flow guarantees in exchange for scalability: routers only need to know a packet's class (encoded in the 6-bit DSCP field), not the identity of its flow. Domain-level Bandwidth Brokers handle inter-ISP coordination rather than per-router RSVP daemons.

RSVP in Three Sentences

RSVP is a receiver-initiated, network control protocol that runs as a daemon on every host and router in the end-to-end path. A reservation request carries a flow descriptor (flowspec + filterspec); the flowspec configures the packet scheduler (via Rspec and Tspec parameters) while the filterspec configures the packet classifier. Admission is contingent on both resource availability (admission control) and user authorisation (policy control); failure in either results in an error notification to the application.

DiffServ in Three Sentences

DiffServ classifies all traffic into a small, standardised set of classes identified by the DSCP field in the IP header; routers apply per-hop behaviours to each class without maintaining any per-flow state. Domain boundaries host traffic conditioners (meter → marker → shaper/dropper) that enforce SLA compliance and re-mark packet priorities based on remaining delay budget. Bandwidth Brokers coordinate QoS allocation between adjacent DS domains via bilateral agreements, replacing the global router-level coordination of IntServ.
