Lecture 44 HIGH PRIORITY

Software Defined Networking — II (OpenFlow)

OpenFlow protocol mechanics, flow tables, match-action rules, controller-switch messaging, and the shift from closed to open networking innovation.

Closed vs Open Innovation

Closed Innovation in Traditional Networking

In traditional networking, every device (router, switch) is a single vendor-locked box containing three tightly coupled components:

Packet Forwarding Hardware
Implemented in TCAM (Ternary Content-Addressable Memory) or older CAM. Performs the actual data plane forwarding at hardware speeds.
Network Operating System
Vendor-specific OS (e.g., Cisco IOS). Implements routing and control protocols. Tightly coupled to the vendor's hardware.
Applications
Firewall, packet forwarding, QoS applications — all bundled inside the same device by the same vendor.
Problems with Closed Innovation: Poor interoperability (Cisco router vs Netgear router require separate manual configuration), difficult network manageability, and vendor lock-in. Network admins must read 1000-page vendor-specific manuals for each device type.

Open Innovation — The SDN Approach

SDN breaks the vendor box apart. Vendors supply only dumb / blind / open switches — programmable packet forwarding hardware. A single central network-wide operating system (the SDN controller) programs all hardware across all vendors.

Closed Innovation
Hardware Vendor-locked, proprietary TCAM/CAM
Software Vendor OS (Cisco IOS, etc.), bundled inside device
Apps Pre-installed, no external control
Config Per-device, vendor-specific CLI/manual
Innovation Slow — all changes need vendor involvement
Open Innovation (SDN)
Hardware Programmable open switches, any vendor
Software Open-source NOS (ONOS, Ryu, OpenDaylight)
Apps Written in any language (Python, Java) on top of controller
Config Centralized — one controller manages all switches
Innovation Rapid — anyone can write and test new protocols
Key SDN Principle: Data plane functionality (TCAM-based forwarding) stays in the hardware. Control plane logic (routing decisions, protocol implementation) moves to a logically centralized software controller.
SDN Architecture

Three-Layer Architecture

Application Layer Flow optimizer, load balancer, topology viewer, policy enforcement, bandwidth mgmt, automation
Control Layer (SDN Controller) GUI, L2/L3 module, cluster module, VPN module, QoS/ACL module, DHCP module
Infrastructure Layer ToR switches, DC gateways, servers, open switches (OpenFlow-enabled)

Controller Interfaces

Northbound Interface
Connects control layer to the application layer. Exposes a programming API so developers can write network apps in their preferred language (Python → POX/Ryu, Java → OpenDaylight). The northbound interface compiles application logic into network protocol instructions and maps them to controller modules.
Southbound Interface
Connects control layer to the infrastructure layer. Converts network protocol rules into forwarding entries that get programmed into the TCAM of dumb switches. OpenFlow is the southbound interface protocol. Plugins like Dovs, NetConf, OpenFlow implement this interface.

Three Requirements for SDN to Work

Open interface to hardware (vendor-agnostic) Open API for application development Extensible OS to convert programs → forwarding rules

OpenFlow

What is OpenFlow?

OpenFlow is an open-source protocol that implements the southbound interface — it controls the forwarding behaviour of Ethernet switches in an SDN network.

Originally released by Clean Slate Program, Stanford Specifications maintained by Open Networking Forum (ONF) Current stable version: 1.5.1

How OpenFlow Works

Inside each OpenFlow (OF) switch, there are two planes:

Control Path (Software)
Runs a minimal OpenFlow Client — a lightweight stub that can receive, parse, and apply configuration messages from the controller. No routing protocols run here anymore. The switch does not need to think — it just obeys the controller.
Data Path (Hardware — TCAM)
The actual packet forwarding engine. TCAM is programmable memory — the controller installs forwarding rules into it dynamically via OpenFlow messages. When a packet arrives, its header fields are matched against TCAM entries and the corresponding action is executed at hardware speed.
Controller ↔ Switch communication uses the OpenFlow Messaging API over SSL/TCP. The controller is logically centralized. The switch is just a dumb hardware executor.
OpenFlow Flow Table

Flow Table Structure

Each entry in the OpenFlow flow table has four components:

Component Description Details
Rule Set of header fields to match against incoming packets Switch Port, VLAN ID, MAC Src, MAC Dst, ETH Type, IP Src, IP Dst, IP ToS, TCP Sport, TCP Dport — wildcards (*) allowed
Action What to do when rule matches Forward to port(s), encapsulate and forward, modify fields, drop, custom extensions
Statistics Counters maintained per rule Packet counter, byte counter — enables traffic monitoring and bandwidth throttling per flow
Priority Numeric priority of the rule When a packet matches multiple rules simultaneously, the highest priority rule is executed

OpenFlow Flow Table — Rule Fields

L2 Fields
Switch Port  ·  VLAN ID  ·  MAC SRC  ·  MAC DST  ·  ETH Type
L3 Fields
IP SRC  ·  IP DST  ·  IP ToS (for QoS)
L4 Fields
TCP SPORT  ·  TCP DPORT
Wildcard
* (any value) can be used for any field not being matched
OpenFlow's power: Rules can span L2, L3, and L4 headers simultaneously. A traditional router only looks at L3 (IP destination). OpenFlow can match any combination of fields, enabling incredibly fine-grained per-flow control.
Flow Table Examples

Use Cases as Match-Action Pairs

Layer 2 Switching

Match only on MAC DST; all other fields are wildcards. If MAC DST = 12:3F:…, forward to eth2. Behaves exactly like a normal L2 switch.

Firewall

Match on TCP DPORT. If TCP destination port = 22, action = drop. All other fields wildcarded. This implements an SSH-blocking firewall rule with a single table entry.

L3 Forwarding

Match on IP DST subnet. If IP DST belongs to 202.2.*.*, forward to eth2. Standard IP routing behavior implemented as a match-action rule.

Source Routing

Match on both IP SRC and IP DST. If SRC is from 16.2.3.* and DST is 202.2.*.*, forward to eth2. Enables path selection based on source, not just destination.

Flow Switching Interesting

Match on all fields simultaneously — MAC SRC, MAC DST, ETH Type, IP SRC, IP DST, TCP SPORT, TCP DPORT. This uniquely identifies a process-to-process flow. For that specific flow, forward to a specific port. This is circuit-switching-like behaviour achieved within a packet-switched network — SDN can bridge both paradigms.

VLAN Switching

Match on VLAN ID + MAC DST. Action = forward to multiple ports (e.g., eth2 and eth3). Ports in the same VLAN receive the packet — implements virtual LAN segmentation.

Power of OpenFlow

Per-User Forwarding Policies

Each user (Bob, Alice) writes their own forwarding application on top of the same controller. The controller compiles the application and installs forwarding rules only on the specific switches that lie along the desired path. Other switches are unaffected. Multiple users' rules can coexist on the same physical network infrastructure simultaneously.

Bob wants R1 → R2 → R4: rules installed only on R1, R2, R4.
Alice wants R1 → R4 → R3: rules installed only on R1, R4, R3.
Completely independent forwarding policies on a shared physical network.
OpenFlow Messages

Controller ↔ Switch Message Types

Message Direction Purpose
HELLO Bidirectional Exchanged after TCP connection; negotiate OpenFlow version (highest common version is used)
Feature Request / Reply Controller → Switch Controller learns the datapath ID of the switch and determines which features it supports (e.g., does it support QoS-based forwarding?)
Flow Modification Controller → Switch Install, modify, or delete flow entries in the switch's TCAM flow table. Core mechanism for programming the switch.
Echo Request / Reply Bidirectional Keepalive — checks that the TCP connection between controller and switch is still alive
Group Configuration Controller → Switch Groups multiple flow entries together into group tables inside the switch for bulk rule management
Stats Request (flow/port/queue/group/table) Controller → Switch Retrieves per-flow and per-port counters from the switch. Enables traffic monitoring, anomaly detection, and dynamic bandwidth throttling.
Async Messages (flow removal, config error, port up/down) Switch → Controller Event-driven notifications. If a port goes down, the switch notifies the controller so it can reroute traffic with a failsafe rule.
Packet Processing Flow: Packet arrives at OF switch → IP header extracted → fields matched against TCAM flow table → highest-priority matching rule found → corresponding action executed (forward, drop, modify, encapsulate).
Concept Summary

Key Takeaways

CLOSED vs OPEN
Vendor-lock to Open Innovation
Traditional networking bundles hardware + OS + apps from one vendor per device. SDN separates these: any vendor's hardware + open-source controller + custom apps. The packet forwarding hardware becomes programmable and vendor-agnostic.
INTERFACES
Northbound vs Southbound
Northbound: controller ↔ applications (programming API, your Python/Java code lives here). Southbound: controller ↔ hardware (OpenFlow lives here — converts protocol logic to TCAM rules installed in dumb switches).
OPENFLOW
Match-Action Paradigm
Every networking behaviour — switching, routing, firewall, QoS, NAT, load balancing — can be expressed as a match + action pair in the flow table. The rule specifies which header fields to look at; the action says what to do on a match. Wildcards (*) allow partial matching.
FLOW TABLE
Rule · Action · Statistics · Priority
Four components per entry. Priority resolves conflicts (multiple matching rules → highest priority wins). Statistics give real-time per-flow counters for monitoring and control. Rules are stored in TCAM — programmable hardware, updated dynamically by the controller.
MESSAGING
TCP-based Controller ↔ Switch Protocol
OpenFlow runs over SSL/TCP. Handshake: HELLO (version negotiation) → Feature Request (capability discovery) → Flow Modification (rule installation). Ongoing: Echo (keepalive), Stats (monitoring), Async messages (event notifications like port-down). Switch only needs a minimal OpenFlow client — no routing intelligence needed on the switch itself.
Future relevance: SDN is not a niche topic. The networking industry is actively moving toward SDN-controlled infrastructure. OpenFlow, ONOS, Ryu, and OpenDaylight are real production tools used by cloud providers and ISPs today.
