ARP / RARP / BOOTP / DHCP (Contd.)

Continuation of companion protocols to the data link layer — covering RARP, BOOTP, and DHCP in detail, including packet formats, operational flows, and lease management.

▶ Watch Lecture
Protocol Evolution

These protocols sit at the boundary between the network layer (IP) and the data link layer (MAC). They collectively solve the problem of bootstrapping a host's network identity from nothing but a hardware address.

DHCP Full dynamic config — current standard
BOOTP UDP/IP — richer config than RARP
RARP MAC → IP only; data link layer; LAN-only
ARP IP → MAC (previous lecture)
RARP — Reverse Address Resolution Protocol

What it does

RARP performs the reverse mapping of ARP: given a known physical (MAC) address, resolve it to a logical (IP) address. This is primarily needed by diskless / thin-client workstations that have no persistent storage to remember their IP configuration across reboots.

ARP: IP → MAC  |  RARP: MAC → IP

Operation

The thin client broadcasts a RARP request announcing its MAC address. A RARP server on the network holds a static MAC-to-IP mapping table. On finding a match, it replies unicast directly back to the requesting machine with the assigned IP.

a. → RARP Request — broadcast (MAC known, IP = ?)
b. ← RARP Reply — unicast (IP filled in by server)

RARP Packet Format 06:37

The packet format mirrors ARP. Key difference in field population between request and reply:

Field In Request In Reply
Operation 3 (Request) 4 (Reply)
Sender Hardware Address Known (6 bytes, e.g. Ethernet) Known
Sender Protocol (IP) Address Unknown — not filled Filled by server
Target Hardware Address Known (server's MAC) Known
Target Protocol Address Known (server's IP) Known

Encapsulation

RARP packets are encapsulated directly in Ethernet frames. The Ethernet Type field is set to 0x8035 to identify RARP payload.

Limitations of RARP

RARP only delivers an IP address. A booting host also requires subnet mask, default gateway address, and DNS server addresses to fully participate in the network. RARP cannot provide these — leading to the development of BOOTP, and later DHCP.

BOOTP — Bootstrap Protocol

Purpose

BOOTP addresses RARP's limitations by operating over UDP/IP (not raw Ethernet) and providing a richer set of configuration parameters at boot time. It runs at the application layer, relying on UDP transport.

BOOTP provides three services: (1) IP address assignment, (2) detection of the IP address of the serving machine, (3) the name of a boot file to be loaded and executed on the client.

Key Properties

UDP-based (not raw L2) Never fragmented Richer config than RARP Static MAC→IP table (no dynamic pools) LAN scope only (without relay)

Well-Known Port Numbers

1. → Client broadcasts BOOTREQUEST containing its MAC address (SP:68, DP:67, SA:all-0s, DA:all-1s)

3. ← Server responds with BOOTREPLY (SP:67, DP:68, SA:server unicast, DA:all-1s or client unicast)

Note: BOOTP assumes packets are never fragmented — the entire response arrives in a single unfragmented UDP datagram.

BOOTP became the direct predecessor and basis for DHCP. Its extensions and limitations motivated DHCP's design.

DHCP — Dynamic Host Configuration Protocol

Purpose and Motivation

DHCP supersedes RARP and BOOTP as the standard for dynamic network configuration. Unlike BOOTP's static MAC→IP table, DHCP maintains pools (scopes) of IP addresses and allocates them dynamically on demand. It is not limited to diskless clients — virtually all modern networked devices use DHCP for configuration at startup.

What DHCP provides
IP address · Subnet mask · Default gateway · DNS server(s) · Lease duration
Transport
UDP — port 67 (server) / port 68 (client) — same as BOOTP

Key Concept: IP Address Leasing

DHCP does not assign addresses permanently. It leases an address for a configurable time period controlled by the administrator. When the lease expires, the address returns to the pool unless the client renews it. This allows fine-grained IP address management across large networks.

DHCP Scope: The administrator-defined pool of IP addresses available for allocation on a given subnet. Not all addresses in the scope are necessarily handed out simultaneously.

DHCP Message Sequence (Initial Binding) 29:34

1. DHCPDISCOVER → Client broadcasts to find available DHCP servers (SP:68, DP:67, SA:0.0.0.0, DA:255.255.255.255)
2. DHCPOFFER ← One or more servers offer an IP address and config parameters
3. DHCPREQUEST → Client selects one server, broadcasts request for the offered address (notifies rejected servers)
4. DHCPACK ← Server confirms; client binds the IP and full network configuration

Lease Renewal and Rebinding State Machine 23:20

After binding, the client manages its lease autonomously through three renewal thresholds:

Lease Time ElapsedClient ActionServer ACKServer NACK / No Response
50% Sends DHCPREQUEST (unicast to original server) — Renewing state DHCPACK → new lease starts, return to Bound No response → wait until 87.5%
87.5% Sends DHCPREQUEST (broadcast) — Rebinding state DHCPACK → bind again with new lease DHCPNACK → client must restart from DHCPDISCOVER
100% (expired) Lease expired — must release address — Returns to Initializing state
Lease cancellation: A client can also voluntarily release its address early by sending DHCPRELEASE, which returns the IP to the pool and resets the client to the Initializing state.

DHCP State Diagram Summary

Initializing
Boot / DHCPNACK / lease cancelled
Selecting
DHCPDISCOVER sent → collect offers
Requesting
DHCPREQUEST sent to chosen server
Bound
DHCPACK received — fully configured
Renewing / Rebinding
50% / 87.5% lease expiry handling

DHCP and Cross-Subnet Operation

DHCP operates across most IP routers. Since the initial DHCPDISCOVER is a broadcast (limited to the local subnet), a DHCP relay agent (configured on the router) forwards it as unicast to a DHCP server on a remote subnet. This allows a single DHCP server to serve multiple subnets, with the server allocating from the appropriate scope based on which relay agent forwarded the request.

Concept Summary
RARP — Reverse of ARP. MAC → IP. Broadcast request, unicast reply. Data link layer only. No config beyond IP. Superseded by BOOTP/DHCP.
BOOTP — UDP/IP based. Ports 67/68. Provides IP + subnet + gateway + DNS + boot filename. Static MAC-to-IP table on server. Packet never fragmented. Basis for DHCP.
DHCP — Dynamic address pools (scopes). Leases IPs for configurable time. Full config delivered. State machine: Init → Selecting → Requesting → Bound → Renewing/Rebinding. Renewal at 50% and 87.5% lease expiry. DHCPNACK forces full rediscovery. Relay agents enable cross-subnet operation.
Layer note: These protocols sit at the boundary of network and data link layers. Some literature places ARP/RARP at L2, while BOOTP/DHCP operate at the application layer over UDP/IP — but all involve both MAC and IP address spaces.
