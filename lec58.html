Security — Basic Components (CIA Triad)

Every security discussion — whether for networks, computers, or information systems — revolves around three core properties. A breach of any one constitutes a security failure.

Confidentiality
Keeping data and resources hidden from unauthorised parties. A message sent between two parties should not be readable by anyone else. Focuses on secrecy.
Integrity
Ensuring data is not tampered with in transit (data integrity) and that the claimed sender is actually who they say they are (origin integrity / authentication).
Availability
Enabling legitimate access to data and resources. A system can have perfect confidentiality and integrity but still be under attack if it is made inaccessible — e.g., via flooding or congestion.
Four Types of Security Attacks

Any action that compromises the security of information is a security attack. All attacks map to a violation of one or more CIA properties, framed against a basic source-to-destination message model.

Attack Type What Happens CIA Property Violated
Interruption Message is blocked and never reaches destination (physically or via congestion). Also known as Denial of Service (DoS). Availability
Interception An unauthorised party (I) gains access to the message flowing from S to D — eavesdropping / wiretapping. Confidentiality
Modification Attacker intercepts the message, alters it, then forwards the tampered version to the destination. Integrity
Fabrication Attacker (I) sends a message to D while impersonating S — the destination believes it is receiving from a legitimate source. Authenticity
Classes of Threats
Disclosure
Snooping — passive observation of data in transit without altering it.
Deception
Modification, spoofing, repudiation of origin, denial of receipt — causing a party to believe something false.
Disruption
Modification of system components to interrupt or degrade the service.
Usurpation
Modification, spoofing, delay, denial of service — gaining unauthorised control over system resources.
Passive vs Active Attacks 02:35
Passive Attacks
GoalObtain/gather information only — no data alteration.
TypesRelease of message content (eavesdropping); Traffic analysis (inferring communication patterns, frequency, length).
DetectionVery difficult — attacker does not manifest on the system; operates silently.
PurposeOften a precursor to active attacks; gathered intelligence enables targeted strikes.
Active Attacks
GoalModify the data stream or create a false data stream.
TypesMasquerade (identity spoofing); Replay (retransmit captured data units); Modification; Denial of Service.
DetectionEasier to detect since they produce observable effects on the network or system.
DoSFlooding communication facilities to prevent legitimate use — everything is in place but the resource is inaccessible.
Policies and Mechanisms 09:57

Security is enforced through the interaction of policies (what is and is not allowed) and mechanisms (technical means to enforce those policies).

Policy conflict → vulnerability. In large organisations, policies are often defined piecemeal by different teams. When those partial policies are integrated or implemented in a distributed manner, they can conflict — creating exploitable gaps. A classic example: a rule restricting hall-to-internet access that does not also restrict hall-to-lab-to-internet hops effectively leaves a bypass open.
Trust and assumptions underlie every security decision. Policies must unambiguously partition system states into secure and non-secure sets. Mechanisms are assumed to correctly enforce the policy — if either assumption breaks, the system becomes vulnerable.
Goals of Security 13:00
Prevention
Stop attackers from violating security policies in the first place. The primary and most desirable goal.
Detection
Identify when and where a policy violation has occurred. Necessary because prevention alone is never fully achievable.
Recovery
Stop the attack, assess and repair damage, and continue to function correctly even after a successful attack. Ensures resilience.
Types of Mechanisms — Secure / Precise / Broad 15:49

A system transitions through security states as it evolves (new software installed, patches applied, devices added). The goal is to keep all reachable states within the set of secure states.

Secure
All reachable states are a subset of secure states. Ideal — the system never enters an insecure state.
Precise
Reachable states map exactly onto secure states — a one-to-one correspondence. Theoretically ideal but hard to achieve in practice.
Broad security occurs when some reachable states fall outside the secure perimeter. Security exists but with gaps. Security also introduces cost: monetary overhead and performance degradation (authentication checks, encryption, filtering all add latency).
Assurance 19:23

After deploying a security infrastructure, assurance answers the question: up to what level is this system actually secured? It requires:

Specification
Requirements analysis — what desired functionality must the security system provide?
Design
How will the system meet the specification — architectural decisions, component choices.
Implementation
Programs and systems that carry out the design — the actual deployed software/hardware.
Operational Issues 20:32
Cost-Benefit Analysis
Is it cheaper to prevent attacks or recover from them? Not every system needs the same level of protection — prioritise critical assets (e.g., a few hundred critical servers out of thousands).
Risk Analysis
Should we protect a particular asset? How much? Tiered security levels are common — strong protection for critical infrastructure, lighter protection elsewhere.
Laws and Customs
Are the desired security measures legal in the jurisdiction? Will users comply with them? Regulatory and human compliance are part of the operational reality.
Human Issues
Organisational problems (power, responsibility, financial incentives) and people problems (outsiders vs insiders, social engineering). Insiders are often the most difficult threat to prevent technically.
Security Services 25:06
Service What It Provides Example Mechanism
Confidentiality Protection against loss of privacy; data hidden from unauthorised parties. Encryption
Authentication Identification of the legitimate creator or sender of data. Digital certificates, passwords
Integrity Assurance that data has not been altered in transit. Message digests, checksums
Non-repudiation Proof that an action occurred — the sender cannot deny having sent it. The order is final. Digital signatures, OTP confirmation
Access Control Prevent misuse of resources. Defines what each user/role may access. MAC, DAC, RBAC (Role-Based Access Control)
Availability Ensure data/resources remain accessible to authorised parties; prevent erasure or denial. Redundancy, anti-DoS filtering
Safety Protection against tampering, physical damage, and theft. Physical security, backups
Non-repudiation in practice: When you swipe a card at a shop and sign the slip, that slip is kept as proof — not sent to the bank immediately, but used if you later dispute the transaction. The same principle is applied digitally via OTPs, digital signatures, and transaction confirmation codes. Once you confirm, the order is final.

Step 1 — Determine Security Policy 31:12

A security policy is the full roadmap. It must cover: usage policies for networks and servers; user training (password hygiene, social engineering awareness); privacy policies for stored data; schedules for updates and audits. The network design itself must reflect the policy — where database/file servers are placed, the location of Demilitarized Zones (DMZs), firewall placement and rules, and deployment of Intrusion Detection Systems (IDS).

DMZ (Demilitarized Zone): The area between the secure internal network and the external internet. Services that must be publicly accessible (e.g., web servers) are placed in the DMZ — exposed to the outside but isolated from the internal network. Everything else stays behind the firewall.

Step 2 — Implement Security Policy 34:02

Implementation involves installing and configuring firewalls and IDS systems. iptables is a widely-used free firewall configuration tool on Linux. Rules for incoming packets should be created (drop by default; allow specific services like HTTP). Rules for outgoing packets may also be created depending on policy. Snort is a common free and upgradeable IDS; most IDS tools log alerts to files and can trigger paging, email, or telephone notifications on serious events.

Tying It Together — The Security Lifecycle

Security is a continuous, dynamic process — not a one-time deployment. The lifecycle flows as: Threats → Policy → Specification → Design → Implementation → Operation and then loops back. New threats emerge from operation, prompting policy revision. This cycle must be actively managed.

TCP/IP and security: The TCP/IP and OSI models do not natively include security provisions. Security mechanisms are layered on top. Any security additions must be designed carefully — intermediate devices (routers, switches) must still be able to forward packets correctly. If a security mechanism interferes with standard protocol behaviour, packet forwarding breaks.
Concept Summary
CIA Triad: Confidentiality (secrecy), Integrity (data + origin), Availability (access). Every security mechanism ultimately protects one or more of these.
4 attack types: Interruption (→ availability), Interception (→ confidentiality), Modification (→ integrity), Fabrication (→ authenticity). All are variations of an attacker interfering with a source→destination message flow.
Passive vs Active: Passive = eavesdrop only, hard to detect. Active = alter/create data, detectable but damaging.
Security goals: Prevention → Detection → Recovery. In order of preference; in practice all three are needed.
Non-repudiation is the property that prevents a sender from denying they sent a message. Implemented via digital signatures, OTPs, and confirmation flows.
Network security workflow: Policy → Implementation (firewalls + IDS) → Reconnaissance → Vulnerability scan → Penetration testing → Post-attack investigation.
