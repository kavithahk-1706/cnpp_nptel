Network Layer I — Introduction

▶ watch lecture
Lecture Overview

This lecture opens the network layer module. It covers what the network layer does, why it exists, and what makes it fundamentally different from the layers below and above it. The core topics are: best-effort datagram delivery, the routing problem in a decentralised graph network, the hierarchical architecture of the internet (LAN → AS → ISP tiers), and the two foundational requirements of IP addressing — uniquely identifying a network and a host within it.

Context: We've covered application layer and transport layer. Network layer (Layer 3) is the third from the top. This lecture is purely conceptual groundwork — actual IP addressing and routing protocols come in subsequent lectures.
Network Layer — Role and Services ↗ ~00:44
Primary objective: Deliver a data packet from any source host to any destination host across a multi-hop network. The network layer provides best-effort, unreliable datagram delivery — it tries its hardest, but guarantees nothing.

The data unit at the network layer is called a datagram. The word "unreliable" is not a design flaw — it is an intentional choice. In a packet-switched network, intermediate buffers can fill up and drop packets. Reliability (retransmission, ordering, flow control) is deliberately pushed up to the transport layer (TCP), keeping the network layer simple and fast.

Services the Network Layer Sits Between

The data link layer (L2) provides single-hop delivery — it gets a frame from one node to the immediately adjacent node that is directly connected. The network layer uses this service repeatedly across multiple hops to achieve end-to-end delivery across an arbitrarily large graph of routers.

The Routing Problem — Network as a Graph ↗ ~05:27

Model the entire internet as a graph: nodes are routers (L3 switches), edges are the links (wired or wireless) between them. A source and destination are connected via multiple possible paths through this graph.

Routing is the process of deciding, at every intermediate router, what the correct next hop should be for a packet with a given destination address. This decision is made in a completely decentralised way — there is no central coordinator that can see the whole graph.

Why not just use shortest-path (minimum hops)?

Minimum hop count is the most obvious metric but not always optimal. The minimum-hop path might have very low link capacity. Routing all traffic down one narrow path causes congestion and degrades performance. Other metrics — bandwidth, latency, load — can serve as link weights instead.

Why can't we just run Dijkstra centrally?

In theory, if you had the entire topology and link characteristics as a central graph, you could run Dijkstra's or Bellman-Ford to compute optimal paths. In practice this is impossible at internet scale — the internet is decentralised, no single entity owns or monitors every router. Each router must compute its own next-hop independently using routing protocols that exchange partial topology information with neighbours.

Routing protocol = the distributed algorithm that lets individual routers collectively agree on paths without any central coordinator. The design of these protocols is the central problem of the network layer.

Two primary requirements for network layer to work:

1. Addressing
Every host must be uniquely identifiable by a globally meaningful address. The address must encode both which network the host belongs to and which host within that network — analogous to a postal address with city + house number.
2. Routing
Given a destination address, every intermediate router must be able to decide the next hop. Routing protocols distribute topology information so each router can make this decision locally without global knowledge.
Postal Mail Analogy — Hierarchical Forwarding ↗ ~14:15

The entire packet delivery model in the network mirrors how physical postal mail works. A postal address is hierarchical: name → building → locality → city → pin code → state → country. At each level, the postal system only needs to understand the next coarser level to forward correctly. The letter does not need a complete map of every route — it just moves toward increasingly specific destinations.

Internet routing follows the same principle. An IP address encodes a network identifier and a host identifier. Routers in the middle of the internet only need to understand the network part — once the packet reaches the right network, the local router delivers it to the specific host.

Example: Sandip Chakraborty, CSE-331K, Dept. of CSE, IIT Kharagpur, West Bengal, India — the postal system forwards India → West Bengal → Kharagpur → IIT KGP → CSE dept → office number. Exact same principle, exact same hierarchy.

Internet Architecture — Hierarchical Organisation ↗ ~16:09

The internet is organised in a strict hierarchy of networks. Understanding this hierarchy is essential for understanding why IP addressing looks the way it does.

Bottom-up build: from lab to global internet

LAN (Local Area Network)
A set of devices sharing a common L3 gateway. Devices within a LAN are directly connected via L2 switches. Example: SW Lab 1 at IIT KGP — all machines connected through one L2 switch, forming a single broadcast domain.
Department Network
Multiple LANs (e.g. SW Lab 1, SW Lab 2, faculty network, research labs) connected via L3 routers. Each LAN is its own broadcast domain; the L3 router connects them. Example: CSE Network or EEE Network at IIT KGP.
Institution Network
Multiple department networks connected via L3 routers form the institution's complete network. Example: IIT KGP Network = CSE + EEE + Mechanical + Admin + Hostel networks all connected.
ERNET / Regional Network
Multiple institution networks connected together. ERNET (Education and Research Network of India) is the government initiative connecting all IITs, central universities, and research institutes. IIT KGP, IIT Bombay, IIT Kanpur etc. all connect here.

Above ERNET, you have national-level ISPs (Airtel, BSNL, Vodafone in India). Multiple national ISPs are connected globally via Internet Exchange Points (IXPs) — physical interconnects where traffic passes between national backbones. The transatlantic fiber cables under the Atlantic Ocean are a real-world example of an IXP connecting European and US national networks.

Autonomous Systems (AS) ↗ ~20:55
Autonomous System (AS): A set of LANs under a single administrative domain, identified by a globally unique AS number, where routing policy is controlled by a single administrator.

The key property of an AS is that the routing policy — the metric used to define "best path" — is chosen by the AS administrator and can differ from one AS to another. Inside an AS, a consistent routing policy is applied. Between ASes, a separate inter-domain routing protocol (BGP) is used.

Why routing policy matters

If you could represent the whole internet as one graph and run Dijkstra, the metric you choose as edge weight determines everything. Different network operators have different priorities — some optimise for latency, some for cost, some for bandwidth. The AS model allows each operator to independently choose their internal metric while still connecting to the global internet. This is not a technical limitation — it is a deliberate design for administrative autonomy.

India's AS graph is maintained and visualised at labs.apnic.net/vizas/#IN. Central nodes (high-connectivity ASes like ERNET, Airtel backbone) provide transit to edge ASes (individual institutions, small ISPs). Worth checking out once.
ISP Tier Architecture ↗ ~24:12

Internet Service Providers are organised into three tiers. Each tier gets connectivity from the tier above and provides it to the tier below.

Tier 3 — Local ISP
The ISP that directly connects end users. Gets service from Tier 2. Examples: IIT KGP campus network, Airtel home broadband, a company's internal network. The end user's last-mile connection.
Tier 2 — Regional ISP
Connects multiple Tier 3 networks. Gets service from Tier 1. Examples: ERNET India, state-level BSNL networks. Regional ISPs can peer with each other directly (private peering) to reduce costs.
Tier 1 — National/Transit ISP
Country-level backbone. Connects to other national ISPs via IXPs (Internet Exchange Points). No one above them — they peer globally with each other. Examples: Bharti Airtel backbone, major US telecom carriers.

Private Peering

When two ISPs (typically at the same tier) establish a direct connection between themselves and agree to exchange traffic directly without going through an upstream provider. This reduces transit costs and can improve routing efficiency. The reduced pricing you sometimes see when transferring data between two users on the same carrier (e.g. Airtel-to-Airtel) often reflects a private peering relationship.

Point of Presence (PoP)

A special-purpose edge network that consumes service but does not provide it to others. Examples include military networks and private government networks — they connect to the backbone but are not transit providers for anyone else.

Packet path across the hierarchy (example: IIT KGP → Google USA)

Your machine → IIT KGP LAN → IIT KGP AS → ERNET India → Bharti Airtel → (IXP / transatlantic cable) → US national ISP → Google's regional ISP → Google's AS → Google server.

Exact mirror of: your local post office → regional office → central post office India → international → US central → US regional → local → door.
IP Addressing — Basic Principles ↗ ~30:13

For routing to work in this hierarchical architecture, every host's address must encode two things simultaneously:

Network Identifier + Host Identifier = IP Address
Why this two-part structure is essential: A router in the middle of the internet does not need to know about every individual host — that would require storing billions of entries. It only needs to know which network to forward toward. Once the packet reaches the destination network, the local router handles the final hop to the specific host. Two-level lookup, just like postal delivery.

Routing decisions use the network part; local delivery uses the host part

Intermediate routers read the network identifier from the destination IP and forward accordingly — they don't care about the host part at all. Only the final-hop router (directly connected to the destination network) looks at the host identifier to deliver to the specific machine.
