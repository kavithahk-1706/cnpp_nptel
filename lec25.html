Socket Programming — II

Continuation of socket programming: TCP server and client implementation, iterative vs. concurrent (fork-based) servers, and the select() system call for I/O multiplexing.

Note: This lecture is heavily demo-driven. The entire second half involves live C code walkthroughs and terminal demos. Understanding the concepts is more important than memorising the code syntax, but you should be able to reason through what each system call does.

Part 1 — TCP Server & Client

TCP vs UDP Socket Types

The core difference when switching from UDP to TCP at the socket creation step is the socket type argument:

UDP Socket
TypeSOCK_DGRAM
ProtocolConnectionless
Noconnect/accept calls needed
TCP Socket
TypeSOCK_STREAM
ProtocolConnection-oriented
Yesconnect() and accept() required

Both use AF_INET as the protocol family and 0 as the protocol field (letting the OS choose the appropriate protocol for the socket type).

TCP Server — Step-by-Step Flow

The TCP server setup follows a fixed sequence of system calls. Each step has a distinct purpose.

socket(AF_INET, SOCK_STREAM, 0)
Create socket fd
serverAddr.sin_family / sin_addr / sin_port
Init server address struct
bind(socketfd, &serverAddr, sizeof(serverAddr))
Attach socket to port
listen(socketfd, 5)
Mark socket as passive; set backlog
accept(socketfd, &clientAddr, &cliLen)
Block until client connects; returns new fd
Key insight — two file descriptors on the server: The original socketfd (the listening socket) remains open to accept future connections. accept() returns a new child file descriptor (childfd) that is used exclusively for send/recv with that specific client. This separation is what makes concurrent handling possible.

listen() — The Backlog Parameter

listen(socketfd, 5) — the second argument (here, 5) sets the backlog: the maximum number of incoming connection requests that can queue up while the server is busy handling an existing connection. If more connections arrive than the backlog limit, they are dropped.

setsockopt — SO_REUSEADDR

The setsockopt(socketfd, SOL_SOCKET, SO_REUSEADDR, ...) call allows a port to be reused immediately after a previous process releases it. Without this, the OS may hold the port in TIME_WAIT state and reject the bind call on restart.

INADDR_ANY

Setting serverAddr.sin_addr.s_addr = htonl(INADDR_ANY) instructs the socket to accept connections on any available network interface of the local machine, rather than binding to one specific IP.

TCP Client — Step-by-Step Flow

The client is simpler — it does not bind or listen. It takes the server hostname and port from command-line arguments, resolves the hostname using gethostbyname(), and initiates the connection.

socket(AF_INET, SOCK_STREAM, 0)
Create socket fd
gethostbyname(hostname)
Resolve hostname → IP
connect(socketfd, &serverAddr, sizeof(serverAddr))
Initiates TCP 3-way handshake
send(socketfd, buf, bufLen+1, 0)
Send data to server
recv(socketfd, recvBuf, bufLen, 0)
Receive server's response

Part 2 — Iterative vs Concurrent Servers

Iterative Server

The basic TCP server implementation from Part 1 is called an iterative server. The server runs a single while(1) loop: accept one connection → complete all send/recv for that connection → close it → only then accept the next.

Problem: The accept() call is a blocking call — the process halts there until a client connects. Once a connection is accepted, the process is further blocked inside the send/recv loop. All other incoming clients queue up in the backlog. If the server is slow (or the current client keeps the connection open), every other client starves. This does not scale.

How Iterative Server Works — Summary

listen() puts the socket in a passive/listening state and sets the backlog size. accept() blocks until a queued connection exists and returns a new connected socket fd. All other connections arriving during send/recv processing are backlogged. Once the current client is fully handled, the loop comes back to accept(). Client requests are executed one at a time, serially.

Concurrent Server using fork()

To handle multiple clients in parallel, the server creates a child process for each accepted connection using the fork() system call (an OS-level call that duplicates the current process).

fork() behaviour: In the parent process, fork() returns the child's PID (a non-zero value). In the child process, fork() returns 0. This allows an if (fork() == 0) check to split execution paths.

Concurrent Server Logic

After accept() returns the new child file descriptor, the server immediately calls fork(). The child process inherits the child fd, closes the original listening socket (it doesn't need it), and handles all send/recv for that client. The parent process closes the child fd (it doesn't need it) and immediately loops back to call accept() again — without waiting for the child to finish. The send/recv blocking now happens inside the child process, not the parent. Result: the parent can accept new connections in parallel with child processes handling existing ones.

Iterative Server
ConcurrencyNone — one client at a time
BlockingEntire process blocks on send/recv
ComplexitySimple
Use caseLow-traffic, simple servers
Concurrent (fork) Server
ConcurrencyFull — each client gets a child process
BlockingOnly child blocks; parent free to accept
ComplexityModerate (OS process management)
Use caseHigh-traffic servers (e.g. web servers)
Zombie problem with fork(): If a parent process terminates or is killed while child processes are still running, those children become zombie processes — they've finished execution but their entry remains in the process table. This is a resource management concern with fork-based servers.

Part 3 — The select() System Call

Motivation — Peer-to-Peer Chat Application

Consider a P2P chat application where every user runs their own TCP server. A user must simultaneously listen on a socket (incoming messages from peers) and read from standard input (keyboard — their own typed messages). In UNIX, both are file descriptors: a socket is a file descriptor, and standard input (STDIN) is also a file descriptor. The problem is that both inputs are asynchronous — a message from a peer can arrive while you are mid-sentence. How do you watch both at once without blocking on either?

select() — Synchronous I/O Multiplexing

select() is an OS-level system call that monitors multiple file descriptors simultaneously and notifies the caller which ones are ready for I/O. It provides synchronous I/O multiplexing over asynchronous inputs.

Key advantage over fork(): No child processes are created. No zombie problem. A single process can manage many connections efficiently using select().

select() Signature

int select(int nfds, fd_set *readfds, fd_set *writefds, fd_set *exceptfds, struct timeval *timeout);

Parameters

nfds
Highest-numbered fd in any of the three sets, plus 1
readfds
Set of fds to watch for readability
writefds
Set of fds to watch for write availability
exceptfds
Set of fds to watch for exceptional conditions
timeout
Max time to block. NULL = block indefinitely

The Three File Descriptor Sets

readfds
Watched to detect when data is available to read. A fd in this set "fires" when recv() would not block.
writefds
Watched to detect when space is available to write. A fd fires when send() would not block.
exceptfds
Watched for exceptional/error conditions on the file descriptor.
fd_set type
An internal bitmap structure. Each bit corresponds to one file descriptor number. Manipulated with macros.

The Timeout (struct timeval)

select() blocks until one of three things happens: a watched fd becomes ready, a signal interrupts the call, or the timeout expires. The timeout is specified in seconds and microseconds:

struct timeval { long tv_sec; /* seconds */ long tv_usec; /* microseconds */ };

Passing NULL for timeout means block indefinitely until an fd becomes ready.

fd_set Macros

FD_ZERO(&set)
Initialise / clear all bits in the fd_set
FD_SET(fd, &set)
Add file descriptor fd to the set
FD_ISSET(fd, &set)
Test if fd is set (i.e. has an event). Use after select() returns.
FD_CLR(fd, &set)
Remove fd from the set (e.g. when closing a connection)

Return Values of select()

Returns -1
Error occurred — check errno
Returns 0
Timeout expired — no fd became ready
Returns > 0
Number of file descriptors with pending events (ready to read/write or have exceptions)

How select() Works Internally

Internally, select() loops over all file descriptors in the set and calls each fd's poll method. The poll method adds the calling thread to that fd's wait queue and reports which events (readable, writable, exception) currently apply. If any fd already matches the requested condition, select() returns immediately after updating the fd sets. If none match, select() sleeps until either the timeout expires or an event wakes it up via the wait queue mechanism.

select()-based Server Logic (simplified)

The select-based server maintains an array of active file descriptors (fds[]). On each iteration of the outer while loop: initialize the fd_set with FD_ZERO, add the listening socket and all active client fds using FD_SET, call select(). When select() returns, check using FD_ISSET: if the listening socket is set, call accept() to accept the new connection and add it to fds[]. If any existing client fd is set, call recv() on it to read available data and then send() back. If recvLen == 0, the client has closed the connection — call FD_CLR to remove it from the set and close the fd.

select() vs fork() — the tradeoff: fork() creates true OS-level parallelism (multiple processes running simultaneously) but has overhead and the zombie problem. select() handles multiple connections from a single process using event-driven I/O — no child processes, no zombies, more resource-efficient. Both break the blocking problem of iterative servers.

Concept Summary

Three TCP Server Variants

Iterative Server
Accept → recv/send loop → close → repeat. One client at a time. Simple but does not scale. All other clients block in the backlog queue.
Concurrent (fork) Server
Accept → fork() → child handles send/recv, parent immediately loops back to accept(). True parallelism via OS processes. Zombie process management required.
select() Server
Single process monitors all fds simultaneously via the select() multiplexer. No forking, no zombies. Event-driven: only processes a fd when data is actually available.

Key Terms

SOCK_STREAM
TCP socket type — connection-oriented, reliable, ordered byte stream
backlog
Max number of pending (unaccepted) connections the OS will queue
blocking call
accept(), recv() — process halts until the operation completes
fork()
OS syscall to duplicate process. Returns child PID to parent, 0 to child
select()
Synchronous I/O multiplexer — monitors multiple fds from one process
fd_set
Bitmap structure representing a set of file descriptors to monitor
STDIN
Standard input file descriptor (keyboard input) — also an fd in UNIX
SO_REUSEADDR
Socket option to allow reuse of a port immediately after process exit

Know what fork() returns in parent vs. child and how that enables concurrent handling. Know what select() is, what its three fd sets do, what the macros (FD_ZERO, FD_SET, FD_ISSET) mean, and how to interpret its return value.
