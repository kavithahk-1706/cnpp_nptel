Software Defined Networking — I
Basics

Introduction to SDN: motivation, architecture, control/data plane separation, OpenFlow overview, and the shift from distributed to centralised control.

Definition

What is Software Defined Networking?

SDN is a network framework that separates a network's control functions from its data forwarding functions, centralises its intelligence, and abstracts the underlying hardware from applications and services.

The two key separable components inside any router are:

Control Plane
The decision-making module. Runs routing algorithms, constructs and manages routing tables. Acts as the "brain" — an instructor that tells the data plane what to do.
Data Plane
The execution module. Matches incoming packets against the Forwarding Information Base (FIB) and forwards them out the appropriate interface. Does only what it is told.

In a traditional router these two planes are implemented together in the same device. SDN pulls them apart.

Motivation

Why Move Away from Traditional Distributed Control?

Traditional distributed routing has several compounding problems at scale:

Scalability
Routing Protocol Limitations
Distance Vector routing suffers from the count-to-infinity problem. Link State routing requires maintaining very large LSP databases across the entire network. Both are bounded to use within a single subnet/AS; BGP handles inter-AS but is policy-driven and complex to update.
Consistency
Distributed Inconsistency
When a policy changes, every individual router's control plane must be updated. In a distributed architecture this takes time, and during that window routers can hold conflicting routing tables — particularly damaging in large networks with 1000+ routers.
Vendor Lock-in
Proprietary Hardware + Software
Vendors (Cisco, Arista, Juniper, Huawei) bundle hardware (data plane) and software (control plane) together. There is no standardisation across vendors. Cisco IOS alone is a ~5000 page manual. Mixing vendors in one network creates severe interoperability and management complexity.
Manageability
Incremental Deployment Drift
Networks are built incrementally over time. Routers added later may come from different vendors or different models. Reconciling configuration across these heterogeneous devices is extremely difficult.
Core SDN idea: extract all control planes from individual routers and consolidate them into a single, centralised SDN Controller. The routers become "dumb" forwarding hardware — vendors deliver only the data plane (TCAM + forwarding engine), and network managers write the control logic themselves.
Advantages

Benefits of Separating Control from Data

Vendor Independence
Features are no longer limited to what a vendor ships. Any protocol or policy can be written as a software application on the controller.
Community Development
Open-source community can design and deploy new network protocols without waiting for vendor hardware cycles or proprietary firmware updates.
Longer Product Lifetime
The same physical hardware can be repurposed for new use-cases by updating only the controller software. No need to replace switches for new features.
Architecture

How SDN Works — Devices

An SDN network contains two categories of devices:

SDN Controller
A general-purpose computer running controller software. Acts as the brain of the entire network. Holds all routing logic and policies. Dynamically installs forwarding rules into switches. Example open-source controllers: Ryu, POX, NOX, OpenDaylight.
SDN Switches (Dumb Switches)
Contain only TCAM hardware + forwarding engine + a microcontroller for communication. No built-in routing logic. Must be instructed by the controller before they know how to handle any packet. Example: Zodiac FX (4-port SDN switch).
The controller runs on a standard computer — no specialised hardware required, because the route processor is just a general-purpose CPU. The entire brain of a large network can literally run on a laptop during development.
Packet Flow

Step-by-Step: Forwarding in an SDN Environment

Scenario: host h1 wants to send a packet to host h3 via switch s1.

Step 1 → h1 sends packet to switch s1.
Step 2 → Table MISS — s1 has no rule for this packet. It buffers the packet and sends a Packet-IN event to the controller c0, including the packet metadata.
Step 3 → Controller c0 runs its routing logic (software program) and decides what to do with the packet.
Step 4 → Controller sends a Packet-OUT event back to s1 along with a flow rule to install in its TCAM.
Step 5 → Rule is installed in the TCAM of s1. The buffered packet is forwarded to h3.
Step 6 → TCAM Hit for all subsequent packets from h1→h3. No controller involvement — switch forwards directly using the cached rule.
Controller communication is only needed on the first packet of a new flow (a "table miss"). All subsequent packets of the same flow hit the TCAM cache and bypass the controller entirely — keeping per-packet latency low.
Flow Rules

Match-Action Pairs — The Core SDN Primitive

Every rule installed in a switch TCAM is a simple match → action pair. The match field can target any header field; the action says what to do with a matching packet.

Match : DIP = 10.2.0.0/16 → Action: Forward to interface eth0
Match : DIP = 172.16.20.0/24 → Action: Drop (firewall rule)
Match : Transport Port = 80 → Action: Drop (block HTTP)
Match : IP = 172.16.20.0/24, Port = 80 → Action: Drop (combined rule)

Because match fields can span all layers (L2 MAC, L3 IP, L4 transport port, VLAN tags), the same rule mechanism can implement a wide variety of network functions:

Packet Forwarding Firewall NAT Forwarding Gateway Packet Gateway VLAN Management Transport-Layer Filtering Load Balancing
Architecture

The Three-Layer SDN Architecture

Application Layer Firewall / NAT / Custom Forwarding Apps
↕ Northbound API App ↔ Controller interface
Control Layer Network Controller (Ryu / NOX / OpenDaylight)
↕ Southbound API (OpenFlow) Controller ↔ Switch interface
Infrastructure Layer Programmable (Dumb) Switches — TCAM only

The Southbound API (OpenFlow) standardises how the controller communicates with switches — this is covered in detail in Lecture 44. The Northbound API is how network applications talk to the controller to install or query policies.

Protocol

OpenFlow — Brief Overview

OpenFlow is the open-source standard protocol that defines the message format for controller-to-switch communication. It specifies how rules are expressed, installed, and removed from a switch's TCAM. Because of OpenFlow, switches from any vendor can be controlled by any standards-compliant controller.

In SDN terminology, the device is not called a "router" — it is called an SDN switch or OpenFlow switch, because routing logic no longer lives inside it.

OpenFlow protocol internals and rule structure are covered in depth in Lecture 44.

Summary

Concept Summary

SDN in One Line
Pull the routing brain out of every switch and put it in one place. Switches become dumb hardware; a single controller software makes all decisions.
Flow = Match + Action
All network behaviour — forwarding, blocking, NAT, filtering — reduces to match-action rules installed in switch TCAMs by the controller. This is the universal primitive of SDN.
First Packet vs Subsequent
First packet of a new flow triggers Packet-IN → controller decision → Packet-OUT → rule installation. Every subsequent packet in the same flow is handled locally via TCAM hit with no controller involvement.
Key insight: SDN doesn't change what a network does — it changes where the decision-making lives. By centralising it, you gain programmability, consistency, vendor independence, and the ability to enforce global policy from a single point.
