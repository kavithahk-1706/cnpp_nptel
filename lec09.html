Lecture 09 Application Layer

HTTP Proxy, HTML & TELNET

Prof. Soumya Kanti Ghosh · IIT Kharagpur · CNPP

Part 1 — HTTP Proxy

What is an HTTP Proxy?

An HTTP proxy server sits between the client and the actual HTTP server and acts on behalf of one or both sides. It is an intermediate server that intercepts HTTP requests and responses.

Proxy ≠ server. The client talks to the proxy; the proxy talks to the real server. From the client's perspective, the proxy IS the server.

What a proxy can do

Caching
Stores responses locally. If another client requests the same resource, the proxy serves it directly without contacting the origin server — faster, less bandwidth.
Filtering
Controls which clients can access which resources. Can block specific URLs, domains, or content types — used in corporate networks and schools.
Logging
Logs all HTTP traffic passing through. Useful for monitoring, auditing, and debugging network activity.
Part 2 — HTML

What is HTML?

HyperText Markup Language — a computer language used to create web pages. A browser (HTTP client) requests an HTML document from the web server, then parses and renders it. Files use .html or .htm extension.

HTML is a tag language. Tags tell the browser how to display content — format, colour, position, structure. The browser has a built-in HTML parser/interpreter.

Document Structure

Every valid HTML document follows this skeleton:

<html> → Container for the entire document
<head> → Metadata: title, charset, links to CSS etc.
<title> → Page title shown in browser tab
<body> → All visible page content goes here

Tags — The Basics

Tags come in pairs: an opening tag and a closing tag. The content between them is the element content. Tags are not case-sensitive, but the modern standard is lowercase.

Opening: <p>  |  Closing: </p>  |  Empty (self-closing): <img> (no closing tag needed)

Key Tags — Quick Reference

Tag Purpose Notes
<h1>…<h6> Headings 6 levels, h1 = largest
<p> Paragraph Inserts space before/after
<a href="url"> Hyperlink Two parts: address + display text
<img src="url" alt="desc"> Image Empty tag — no closing tag. alt = screen reader description
<div> Division / Section Groups elements; apply format/style to a section

Link Types (3 kinds)

Same Folder
<a href="page.html"> — just the filename, no path needed.
Different Folder
<a href="/folder/page.html"> — relative path from root.
External (Internet)
<a href="https://iitk.ac.in"> — full absolute URL.

Image Path Types

Same Folder
src="samplePic.gif" — browser looks in same directory as HTML file.
Different Folder
src="/images/samplePic.gif" — specify the path explicitly.
DTD (Document Type Definition): Defines what elements/attributes are valid in an HTML document. The <!DOCTYPE html> declaration at the top of a page tells the browser which version of HTML rules to apply.
Part 3 — TELNET

What is TELNET?

TELNET is a protocol (and program) that allows a user to remotely log into another system over a network and work on it as if physically present. It provides a general, bidirectional, 8-bit byte-oriented communication facility.

Protocol
TELNET (RFC 854) — defines how the connection is established, commands are sent, and terminals communicate.
Application / Program
Also called telnet — the client-side program that implements the TELNET protocol. Server-side daemon = telnetd.

Key Technical Facts

Property Value
RFC 854
Default Port 23
Transport TCP — connection-oriented
Data + Control Same TCP connection (unlike FTP which uses port 20 + 21)
Communication Bidirectional, 8-bit byte-oriented
TELNET vs FTP — key difference: FTP uses two separate connections — port 21 for control, port 20 for data. TELNET uses a single TCP connection for everything.

Network Virtual Terminal (NVT)

Different systems have different terminal types, escape characters, and control sequences. To handle this, TELNET introduces an intermediate abstraction layer — the NVT.

NVT = an intermediate representation of a generic terminal. It provides a standard language for communicating terminal control functions between any two systems, regardless of their native terminal type.
Client Side
Local terminal → NVT → TCP
Server Side
TCP → NVT → Server processes

Both TELNET client and server maintain an NVT layer. The client maps its local terminal into NVT; the server maps NVT into whatever its native environment expects.

Negotiated Options

All NVTs support a minimal set of capabilities. Additional capabilities beyond this baseline are optional and negotiated between the two endpoints at connection time.

Options are NOT part of the TELNET protocol itself — they're separate. This means new terminal features can be added without changing the protocol. The two endpoints negotiate mutually acceptable options.

Examples of Negotiated Options

Line mode vs Character mode Echo mode (local vs remote) Character set (ASCII vs EBCDIC)

TELNET Control Functions

TELNET includes a set of control functions for managing the terminal session. These provide a uniform mechanism for both ends to signal things beyond raw data.

Function Code What it does
Interrupt Process (IP) — Suspend / abort the currently running process
Abort Output (AO) — Stop sending further output to the user terminal
Are You There (AYT) — "Are you alive?" — checks if the remote system is still running
Erase Character (EC) — Delete the last character sent
Erase Line (EL) — Delete all input on the current line

IAC — Interpret As Command

Since data and control share the same TCP connection, TELNET needs a way to distinguish regular data from commands. It uses a special escape byte.

IAC (Interpret As Command) = byte value 255. When the TELNET layer sees a 255, it interprets the following byte as a command, not data.
Receive 255 → next byte is 255 → Single byte 255 passed to application as data (escaped)
Receive 255 → next byte is anything else → TELNET layer interprets it as a command and executes accordingly

TELNET as a Generic TCP Client

TELNET is more than just remote login. Because it establishes a raw TCP connection and passes whatever you type, it works as a generic ASCII-based TCP client — useful for testing any ASCII-protocol TCP server.

You can test an SMTP mail server by doing telnet mailserver.com 25 — TELNET connects, and you interact directly with the SMTP protocol. Same idea for any ASCII-based protocol.

Unix Default Servers (accessible via TELNET)

Port Server What it does
7 Echo Echoes back everything you send — useful for connectivity tests
9 Discard Accepts and discards everything — tests that sending works
13 Daytime Responds with current date and time

TELNET — Authentication & Access

When a TELNET connection is established to a remote server, the server prompts for a login and password. Once authenticated, the user gets access according to the permissions set on the server — browse directory structures, execute programs, etc. The session persists until explicitly closed or an error occurs.

Security note (exam awareness): TELNET transmits everything — including passwords — in plain text over TCP. This is why it's been largely replaced by SSH (Secure Shell) in modern systems. TELNET is now mostly used for testing, not real remote access.
