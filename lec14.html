Lecture 14 HIGH PRIORITY

Transport Layer — IV: Reliability & Flow Control

Prof. Sandip Chakraborty · IIT Kharagpur · CNPP

▶ watch lecture
Overview

This lecture covers two core transport layer services: flow control (preventing the sender from overwhelming the receiver) and reliability (ensuring every byte sent is eventually received). These are implemented together in TCP. The lecture also explains why both the transport layer and the data link layer independently implement flow/error control, and introduces the ARQ family of algorithms starting with Stop-and-Wait.

The Problem: Unreliable Network Layer

IP (network layer) is a best-effort delivery service — it makes no guarantees. Intermediate router buffers overflow under heavy load, causing packet drops. The network layer doesn't regulate how fast data is pushed into it, so congestion and loss are routine.

The transport layer's job is to build a reliable, rate-aware channel on top of this inherently unreliable one. The two mechanisms that achieve this are:

Flow Control
Ensures the sender does not transmit faster than the receiver can consume. The sender continuously adjusts its rate based on receiver feedback.
Reliability (Error Control)
Ensures every sent segment eventually arrives. Achieved by detecting loss and retransmitting. The receiver signals what it has received; the sender retransmits what was lost.
Abstraction: The application layer calls rdt_send() (reliable data transfer). The transport layer maps this down to udt_send() (unreliable datagram send) at the network interface, and adds sequence numbers + checksums + timers to paper over the unreliability.
Why Both Data Link Layer AND Transport Layer?

Flow/error control appear at two layers. They serve different scopes and neither alone is sufficient.

Data Link Layer — Hop-by-Hop Flow Control

Each link between adjacent nodes has its own DLL flow control. But this is strictly local. Consider the following path:

S → R1 → 10 Mbps
R1 → R2 → 5 Mbps
R2 → R3 → 3 Mbps
R3 → D → 1 Mbps

The effective end-to-end rate is 1 Mbps (the bottleneck). But DLL flow control only negotiates per-hop — so S happily sends at 10 Mbps, R1 buffers 5 Mbps of excess, R2 buffers more, and each router eventually overflows and drops. Hop-by-hop control cannot tell S that the true end-to-end capacity is 1 Mbps.

Conclusion: DLL-only flow control cannot prevent end-to-end buffer overflow. Transport layer flow control is essential to match the sender's rate to the actual receiver capacity.

Transport Layer — End-to-End Flow Control

Now consider the reverse: end-to-end flow control only, no DLL control. The source is rate-limited to 1 Mbps (matching the receiver). But at an intermediate router, multiple other flows also arrive — say 2 Mbps and 5 Mbps on other ports — and the outgoing link is only 3 Mbps. Total incoming exceeds outgoing. Buffer overflow happens at that intermediate router regardless of what the source does.

Conclusion: End-to-end control alone cannot prevent intermediate congestion caused by traffic from other sources. DLL-level hop-by-hop control is needed to improve performance by reducing congestion locally.

Flow Control Algorithm 1 — Stop and Wait

Error-Free Channel 17:02

The simplest ARQ algorithm. The sender transmits exactly one frame, then halts and waits for the receiver's acknowledgement (ACK) before transmitting the next frame. On an error-free channel there are no losses, so the ACK always arrives and the process repeats.

Send frame → wait for ACK → send next frame → wait for ACK → …

Noisy Channel 18:21

On a real channel, frames can be lost or corrupted. Two mechanisms are added:

Sequence Numbers
Each frame is tagged with a sequence number so the receiver can identify which frame it received and what it expects next. The ACK carries the sequence number of the next expected frame (e.g., ACK 1 = "I got frame 0, send frame 1").
Timeout & Retransmission
After sending, the sender starts a timer. If no ACK arrives before timeout (frame was lost or ACK was lost), the sender retransmits the same frame. This is what makes it reliable.

How many bits do sequence numbers need in Stop-and-Wait?

Only 1 bit (values 0 and 1) is sufficient. At any given time, only one frame can be outstanding in the network — the sender doesn't move on until the current frame is acknowledged. So the receiver only ever needs to distinguish the current expected frame from the last one. Sequence numbers alternate: 0, 1, 0, 1, …

ACK 1 = "received frame 0, expecting frame 1." ACK 0 = "received frame 1, expecting frame 0." This alternating ACK scheme also handles duplicate detection: if a frame is retransmitted and the receiver gets a duplicate, it recognises the sequence number and discards it.

Duplicate Frame Scenario (Key Exam Case)

If the ACK is lost (not the data frame), the sender times out and retransmits frame N. The receiver receives a duplicate frame N. Because the receiver is now expecting frame N+1 (it already ACK'd frame N), it recognises the sequence number mismatch, discards the duplicate, and re-sends the ACK. No data corruption results.

Stop-and-Wait ARQ — Sender State Machine 21:41

The sender FSM has two main states, alternating between waiting to send frame 0 and waiting to send frame 1. (The slide shows this as a Kurose & Ross FSM diagram — worth watching the visual at 21:41.)

State: Wait for call 0 from above
Event→App sends data → attach seq# 0 + checksum, udt_send(), start timer → move to "Wait for ACK 0"
In "Wait for ACK 0"
corrupt pkt→Stay in same state (wait again)
timeout→Retransmit frame 0, restart timer, stay
ACK 1 received (good)→Stop timer → move to "Wait for call 1 from above"
State: Wait for call 1 from above
Event→App sends data → attach seq# 1 + checksum, udt_send(), start timer → "Wait for ACK 1"
ACK 0 received (good)→Stop timer → back to "Wait for call 0 from above"

The state machine guarantees that flow control (one outstanding frame) and reliability (retransmit on timeout) are enforced simultaneously. This combined protocol is called ARQ — Automatic Repeat reQuest.

Problems with Stop-and-Wait 25:19

Stop-and-Wait is correct but extremely inefficient. The sender sits idle waiting for an ACK for every single frame. On a high-bandwidth or high-latency link, this wastes the vast majority of available capacity.

Bidirectional Communication

A bidirectional connection requires two independent Stop-and-Wait instances, doubling the inefficiency. The partial fix is piggybacking — attaching the ACK for the reverse direction onto the outgoing data frame (one combined DATA+ACK packet). This reduces traffic but still doesn't solve the fundamental one-frame-at-a-time bottleneck.

Root problem: Only one frame can be in-flight at any time. The channel is idle while the sender waits for the ACK to travel back. This becomes catastrophic on long-distance or high-speed links (e.g., a 1 Gbps satellite link with 600ms RTT has ~75 MB of "wasted pipe" per ACK round-trip).
Solution — Sliding Window (Pipelined) Protocols 27:49

The solution is to allow multiple frames to be in-flight simultaneously without waiting for individual ACKs. This is the core idea of sliding window / pipelined protocols.

Sliding window protocols are covered in detail in the next lecture. Two major variants exist: Go-Back-N (GBN) and Selective Repeat (SR) — both are important exam topics.
Concept Summary
Why Two Layers?
Transport = essential end-to-end guarantee. Data link = hop-by-hop performance improvement. Neither alone is sufficient; both are needed.
Stop-and-Wait ARQ
Send one frame, wait for ACK. 1-bit sequence numbers. Timeout triggers retransmission. Correct but bandwidth-inefficient — one outstanding frame at all times.
Sliding Window
Send W frames before waiting for any ACK. Pipeline the channel. Much higher utilisation. Covered fully in Lec 15. Two types: Go-Back-N and Selective Repeat.
