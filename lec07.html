Week 2 · Lecture 07 HIGH YIELD

Application Layer III — Client-Server & FTP

Client-server paradigm, sockets & 5-tuple, FTP dual-connection model, TFTP.

01 — Client-Server Paradigm

The dominant model for network application communication. A server is a process that offers a service; a client is a process that requests it. Both can live on the same machine or different machines — the handling is identical either way.

Core rule: Server is always active, sleeping/waiting for a client request. Client initiates on demand.

Iterative vs Concurrent Servers

Iterative Server
when
Duration of each request is known in advance; resource must be allocated one at a time
how
Single copy runs; serves one client fully, then moves to the next
wait
Client may have to wait if server is busy
Concurrent Server
when
Amount of work per request is unknown; most application servers (e.g. HTTP)
how
Forks a child process per client; parent goes back to listening immediately
wait
Multiple clients served simultaneously up to resource limit
MCQ trap: "Server goes back to sleep after serving" describes iterative behaviour. Concurrent servers fork, not sleep.
02 — Sockets & the 5-Tuple

A socket is the OS mechanism for Inter-Process Communication (IPC) — allows one process to speak to another on the same or a different machine. Analogous to a file descriptor: you get a socket ID and use it for all subsequent I/O.

The 5-Tuple

Every network connection is uniquely identified by five fields:

Server IP + Server Port + Client IP + Client Port + Protocol
If client and server are on the same machine, Server IP = Client IP — but port numbers will still differ, so the 5-tuple stays unique.

Half-Socket → Full Association

Half Socket (server side)
Protocol + Server IP + Server Port. Server opens this and listens — waiting for a client.
Half Socket (client side)
Protocol + Client IP + Client Port. Client opens this and sends a connection request to the server.

When the handshake completes the two half-sockets merge into a full association (the complete 5-tuple) and the socket ID is returned for data transfer.

DNS resolves a hostname → IP address before any socket can be opened. Network layer only understands IPs — names are meaningless at L3.
03 — FTP (File Transfer Protocol)

Defined in RFC 959. Transfers files over a network using a client-server model on top of TCP (reliable, connection-oriented). The defining feature of FTP: it uses two separate TCP connections.

Dual-Connection Architecture

Control Connection — Port 21
purpose
Send/receive FTP commands (login, directory ops, logout)
lifetime
Stays open for the entire FTP session
managed by
PI — Protocol Interpreter
direction
Both ways (commands and server replies)
Data Connection — Port 20
purpose
Actual file/data transfer
lifetime
Opened only when a data transfer command is issued; closed after transfer
managed by
DTP — Data Transfer Process
direction
Unilateral — either client→server or server→client, never both simultaneously
MCQ critical: Only one data transfer can occur per data connection. If multiple transfers are needed in one FTP session, a distinct data connection is opened for each.

Active vs Passive Mode

Active Mode
control
Client (large port N) → Server port 21
data
Client port N+1 ← Server port 20
initiator
Server initiates data connection
Passive Mode
control
Client (large port N) → Server port 21
data
Client port N+1 → Server (ephemeral port >1023)
initiator
Client initiates data connection
Ephemeral ports are any port >1023 — i.e. not a reserved/well-known port. Both client and server can agree to use these for data in passive mode.

Internal Components (both client and server)

PI — Protocol Interpreter
Manages the control connection. Translates app-specific commands into RFC 959 FTP commands. Client PI ↔ Server PI talk to each other over port 21.
DTP — Data Transfer Process
Manages the data connection. Does the actual file movement. Activated only when PI determines a data transfer is required.
User Interface (client only)
CLI or GUI that the user interacts with. Translates user actions into PI commands.

File Transfer Modes

ASCII — text, html, etc. Binary — doc, pdf, media, etc.

Must be declared before transfer using the TYPE command.

Key FTP Commands (exam relevant)

Access Commands
USER, PASS, ACCT, REIN (re-init), QUIT (logout), ABOR (abort previous command)
File Management
CWD (change directory), DELE (delete file), MKD (make directory), RMD (remove directory), PWD (print working directory)
Data Formatting
TYPE (ASCII/binary), STRU (file/record/page structure), MODE (stream/block/compressed)
File Transfer
RETR (retrieve/download from server), STOR (store/upload to server), STAT (return status of file set)
04 — TFTP (Trivial FTP)

Trivial File Transfer Protocol — a stripped-down FTP variant for resource-constrained environments. The key contrast with FTP:

FTP
transport
TCP (reliable)
control port
21
data port
20 (or ephemeral)
auth
USER + PASS
features
Full directory ops, multiple transfer modes
TFTP
transport
UDP (unreliable)
initial port
69
subsequent
Client & server negotiate ephemeral port after initial contact
auth
None — inherently insecure
features
Read file from server OR write file to server only
Use cases for TFTP: uploading router/switch firmware or config files at boot, diskless devices downloading boot images, any automated process where user credentials aren't feasible.
MCQ trap: TFTP uses UDP, not TCP. FTP = TCP port 21/20. TFTP = UDP port 69. Don't mix these up.
