Client-Server Model

The client-server paradigm is the dominant model for network application design. A server is a process that offers a service and waits. A client is a process that requests that service. They can be on the same machine or different machines — the mechanism is identical either way.

Lifecycle: Server initializes → goes to sleep listening on a port → client wakes it up with a request → server responds → goes back to sleep → repeat. The server is always active, the client initiates.

Iterative vs Concurrent Servers

When multiple clients arrive simultaneously, the server has two strategies for handling them:

Iterative Server
triggerService time per request is predictable / known in advance
mechanismSingle server process; handles one client completely before moving on
clientsMust queue and wait if server is busy
exampleResource reservation — concurrent access would cause conflicts
Concurrent Server
triggerService time per request is unknown or highly variable
mechanismForks a child process per client; parent immediately goes back to listening
clientsServed simultaneously up to resource limits
exampleHTTP servers — iitkgp.ac.in can't make the whole internet wait
Forking: When a request arrives at a concurrent server, it forks a child process that handles that specific client end-to-end. The parent process immediately returns to listening. As many children as resources allow can run simultaneously.
MCQ trap: "Server goes back to sleep after serving" = iterative behaviour. Concurrent servers fork, not sleep. The parent goes back to listening; the child does the work.
Sockets & the 5-Tuple

Every connection on a network is uniquely identified by a 5-tuple. IP identifies the machine, port identifies the process on that machine. Together they nail down exactly one endpoint on each side.

This is why you can have five browser tabs open to five different sites at the same time — each tab gets a different source port, making each one a distinct 5-tuple and therefore a distinct, independent connection.

What is a Socket?

A socket is the OS mechanism for Inter-Process Communication (IPC). It lets one process talk to another — same machine or across the network. Analogy: a telephone line between two processes.

Half-Association (3-tuple)
One endpoint: Protocol + IP + Port. The server opens a half-socket and waits. The client opens its own half-socket. Neither is a connection yet — they need each other.
Full Association (5-tuple)
Both half-sockets combined. Once the handshake succeeds, the OS issues a socket ID. All subsequent data transfer just uses this ID — no need to re-specify the whole tuple every time.
Socket ID ≈ file descriptor: In C, you open a file and get a file ID to use for all reads and writes. Same concept — open a socket, get a socket ID, use it for all data transfer. The plumbing is hidden behind the ID.
MCQ note: DNS must resolve a hostname → IP before any socket can be opened. The network layer only understands IP addresses — names are meaningless at L3. So the sequence is always: DNS lookup → get IP → open socket.

FTP — File Transfer Protocol

FTP (defined in RFC 959) is the application layer protocol for transferring files over a network. It follows the client-server model and uses TCP underneath for reliable, connection-oriented transfer. FTP also relies on the Telnet protocol for some control operations.

FTP's defining quirk: it maintains two separate TCP connections simultaneously — one for commands and one for data. Most protocols use one. FTP uses two.

The Two Connections

Connection 1 — always open
Control Connection
Server Port 21
Persistent for the entire session. Used for login, FTP commands, directory navigation, and server response messages. Stays open even when no data is being transferred — this is the command channel.
Connection 2 — on demand
Data Connection
Server Port 20 (default)
Opened only when a data transfer is actually needed (file upload, download, directory listing). Closed immediately after each transfer. One distinct data connection per transfer command.
Data connection is unidirectional: in any single data connection, data flows only one way — either client→server or server→client, never both simultaneously. Also, it can be initiated by either side: server-initiated = active, client-initiated = passive.
MCQ critical: Only one data transfer per data connection. If multiple files need transferring in a single FTP session, a distinct data connection is opened for each transfer. The control connection stays open the whole time.

Internal Architecture: PI & DTP

Both client and server applications are internally split into two processes that mirror each other:

PI — Protocol Interpreter
Manages the control connection. Translates application-level commands into RFC 959-compliant FTP commands and communicates them to the other side. Stays active for the whole session.
DTP — Data Transfer Process
Manages the data connection. Handles actual file transfer after the PI has set things up. DTP comes into play only when a command requires data to move. After transfer, control returns to PI.

Client additionally has a User Interface on top. Server has only PI + DTP — no UI needed. Each side has its own file system; data moves between them via the data connection.

Active vs Passive Mode

This refers to which side initiates the data connection (not the control connection — that's always client→server on port 21).

Active Mode
controlClient port N (>1023) → Server port 21
dataClient port N+1 ← Server port 20
initiatorServer initiates data connection → "active"
problemFirewalls on the client side often block inbound connections, breaking this
Passive Mode
controlClient port N (>1023) → Server port 21
dataClient port N+1 → Server large port (>1023)
initiatorClient initiates data connection → "passive"
why it's usedClient always initiates outbound connections, so firewalls don't block it

TFTP — Trivial File Transfer Protocol

TFTP is the stripped-down sibling of FTP for resource-constrained environments — diskless devices, routers, switches that need a firmware or config file pushed at boot time, where full FTP would be overkill.

FTP
transportTCP — reliable, connection-oriented
ports21 (control), 20 (data)
authUsername + password (USER, PASS commands)
featuresFull directory nav, permissions, multiple transfer types, resume support
use caseGeneral file transfer between machines
TFTP
transportUDP — unreliable, no connection setup
ports69 (initial), then client+server negotiate a different port for the rest
authNone — insecure by design
featuresRead a file from server OR write a file to server. That's it.
use caseFirmware updates, diskless device boot, router config pushes
Why UDP for TFTP? These use cases happen on controlled local/internal networks where packet loss is rare. TCP's overhead isn't worth it for a one-time config push over a trusted internal link. Security is handled by the physical environment (no external connections), not the protocol.
MCQ trap: TFTP uses UDP port 69, not TCP. FTP = TCP ports 21 (control) / 20 (data). TFTP = UDP port 69 (initial only — subsequent transfer uses a negotiated ephemeral port). Don't mix these up.
