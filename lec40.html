Lecture 40 HIGH PRIORITY

IP Routing IV — Border Gateway Protocol (BGP)

Exterior gateway routing across autonomous systems: BGP architecture, packet types, path selection, aggregation, and confederations.

Context & Motivation

The internet is a network of networks — a collection of Autonomous Systems (ASes), each under a single administrative authority (e.g., IIT Kharagpur's campus network). Packets must be routed both within and across these ASes. Two protocol categories handle this:

Interior Gateway Protocols (IGP)
Scope Within a single AS
Examples OSPF, RIP
Algorithm Link-state / Distance-vector
Admin Single organisation
Exterior Gateway Protocols (EGP)
Scope Across autonomous systems
Example BGP (Border Gateway Protocol)
Algorithm Path vector
Admin Multiple organisations
BGP Overview

Border Gateway Protocol (BGP) is the standard exterior gateway protocol of the internet. Key facts:

BGP version 4 (BGP-4) is the current standard, defined in RFC 4271. All BGP-enabled devices (BGP routers / BGP speakers) must conform to this standard.
Original Goal
Loop-free inter-AS routing
Evolved to support
Route aggregation & summarisation (CIDR)
Transport
Runs over TCP (reliable delivery)
Standard
IETF RFC 4271 — BGP Version 4
BGP Components & Terminology

Core Terms

Term Definition
BGP Speaker A router configured to run BGP. The active participant in BGP routing.
BGP Neighbor / Peer A pair of BGP speakers that exchange routing information with each other.
IBGP Internal BGP — session between two BGP peers within the same AS. Peers need not be directly connected; IGP routes are used to reach them.
EBGP External BGP — session between two BGP peers in different ASes. Peers typically share a directly connected network.
BGP Session A TCP connection between two BGP neighbors used to exchange routing info. Session liveness monitored via Keepalive messages.
AS Number (ASN) A 16-bit number uniquely identifying an autonomous system. Up to 216 ASes possible.
AS Path An ordered list of AS numbers describing the route to a destination. Used for path selection and loop prevention.
NLRI Network Layer Reachability Information — advertises reachable networks as <length, prefix> tuples (e.g., the CIDR route 202.124.116.0/14).
AS Path example: To reach network N1 in AS1 from AS4, a possible path is AS4 → AS3 → AS2 → AS1. This list of AS numbers is the AS path stored and advertised by BGP.
Traffic Types & AS Classifications

BGP Traffic Types

Local Traffic
Traffic whose source or destination resides within the AS. Either originates or terminates inside the AS.
Transit Traffic
Traffic that neither originates nor terminates in the AS — it merely passes through. BGP aims to minimise transit traffic as it constitutes an unnecessary load.

AS Types

AS Type Connections Traffic Carried Notes
Stub AS Single connection to one other AS Local only No transit traffic possible — no alternate path out
Multihomed AS Two or more ASes Local only (by policy) Configured to block transit traffic; does not forward packets through itself
Transit AS Two or more ASes Local and transit Carries traffic for other ASes; can impose policy restrictions on what is forwarded
Stub is a special case. In practice, most ASes are either multihomed or transit. An AS can also be simultaneously multihomed or transit depending on configuration.
Routing Policy, NLRI & Routes

Routing policies are not defined in the BGP protocol itself — they are configured on individual BGP devices and shape how the device advertises and receives routes. Examples:

Multihomed AS refusing to act as transit Restricting transit to selected partner ASes Steering specific traffic classes (e.g. video) through preferred paths

This is achieved by tailoring route advertisements sent to EBGP peers — advertising only those prefixes that comply with the policy.

Routes and Paths

A route associates a destination (expressed in NLRI format as <length, prefix>) with a collection of path attributes describing how to reach it. Route information is distributed via UPDATE messages.

EBGP–IBGP Communication

BGP does not replace IGP within an AS — it co-operates with IGP (OSPF/RIP) to forward packets correctly. The relationship works as follows:

EBGP Peers
Typically share a directly connected network. Routing info exchange between EBGP peers is straightforward — no intermediate hops needed to reach the peer.
IBGP Peers
Can be anywhere within the AS — do not need to be directly connected. BGP relies on the IGP (OSPF/RIP) to locate and route to IBGP peers. Packet forwarding between IBGP peers uses IGP-learned routes.

Full Mesh IBGP Requirement

All BGP speakers within an AS must maintain a fully meshed set of IBGP sessions — every speaker is connected to every other speaker. Because of this full mesh assumption, when a speaker receives a route update from an IBGP peer, it does not re-advertise that update to other IBGP peers (they are assumed to have received it already). Instead, it propagates the update to EBGP peers in other ASes.

Scalability problem: Full mesh IBGP becomes unmanageable as the number of speakers grows. This is addressed by BGP Confederations (see below).
BGP Packet Types
Packet Type Purpose Direction
OPEN Establishes a BGP session between two peer nodes. Sent by each router after a TCP session is set up. Bidirectional
UPDATE Transfers routing information between BGP peers. Used to advertise feasible routes or withdraw infeasible routes. Also carries reachability information (NLRI) and path attributes. Bidirectional
NOTIFICATION Sent when an error condition is detected. After sending, the BGP transport connection is closed, resources deallocated, affected routing table entries marked invalid, and other peers notified. Unidirectional
KEEPALIVE Periodically exchanged to confirm peers are reachable. BGP does not use TCP's own keepalive mechanism. If no message is received within the hold timer period, an error is assumed. Bidirectional
BGP Functions
Opening & Confirming Connection
After a TCP session is established, each router sends an OPEN message. The BGP session is considered active once confirmed.
Maintaining Connection
BGP messages are exchanged periodically. If none are received within the hold timer window, the router assumes an error, sends a NOTIFICATION, closes the connection, and releases resources.
Sending Reachability Info
UPDATE messages advertise feasible routes or withdraw infeasible ones. When an error closes a peer connection, all routes learned from that peer are marked invalid and peers are notified.
BGP Selection & Path Attributes

BGP is a path vector protocol — the path to a destination is expressed as a sequence of AS numbers. The best path is selected by comparing the number of AS hops across feasible routes, refined by path attributes. There is no single universal metric; each AS has its own evaluation criteria.

When a BGP router advertises a route to a peer, it may add or modify path attributes before forwarding, allowing policy-based path shaping.

Four Path Attribute Categories

Category Must Recognise? Must Send in Every Update? Behaviour if Unrecognised
Well-Known Mandatory Yes — all implementations Yes N/A — must be recognised
Well-Known Discretionary Yes — all implementations No N/A — must be recognised
Optional Transitive No No Accepted and forwarded to peers as-is without analysis
Optional Non-Transitive No No Can be silently ignored and not forwarded to peers

Loop Prevention

Before inserting a received route into its database, a BGP router checks the AS path. If the router's own AS number already appears in the path, the route is discarded — the packet would loop back through the current AS.

Policy Routing

If a path contains an AS that violates the current AS's policy, the UPDATE message is discarded and that path is not considered.

BGP Aggregation

Introduced in BGP Version 4 alongside CIDR support. Aggregation allows BGP peers to consolidate multiple contiguous routing entries into a single advertisement. For example, routes 192.16.0.0/24 through 192.16.124.0/24 can be aggregated and advertised as a single prefix 192.16.0.0/16.

Reduces routing table size Reduces UPDATE message frequency Significantly improves scalability in large internetworks
BGP Confederations

The full mesh IBGP requirement becomes a scalability bottleneck in ASes with a large number of BGP speakers — the number of sessions grows as O(n²). BGP Confederations address this:

A BGP confederation partitions a single large AS into a group of smaller sub-ASes. To peers outside the confederation, the entire group appears as one AS. Within the confederation, full mesh is only required within each sub-AS, not across all speakers globally.
Eliminates full mesh requirement across entire AS Reduces management complexity Maintains correct inter-AS routing behaviour externally
Concept Summary
BGP in one sentence: BGP is the exterior gateway protocol that connects autonomous systems across the internet using a path vector mechanism, TCP-based sessions, policy-driven route advertisements, and path attributes to determine the best inter-AS paths.

Key Takeaways

BGP vs IGP
Complementary, Not Competing
BGP does not replace OSPF or RIP. IGP handles intra-AS forwarding; BGP uses IGP-learned routes to locate IBGP peers and coordinates with them for inter-AS reachability.
EBGP vs IBGP
Two Flavours of BGP
EBGP connects routers across different ASes (typically directly connected). IBGP connects routers within the same AS (can be anywhere; uses IGP to reach peers). IBGP requires full mesh sessions.
Path Vector
Routing by AS Sequences
Unlike link-state (OSPF) or distance-vector (RIP), BGP advertises the full AS path to each destination. This enables loop detection (own AS in path → discard) and policy control (unwanted AS in path → discard).
Scalability
Aggregation + Confederations
Route aggregation (CIDR-based, BGP-4) reduces table size. Confederations reduce full-mesh IBGP overhead by sub-dividing large ASes into manageable sub-ASes.
Session Management
Four Packet Types
OPEN (establish session) → KEEPALIVE (maintain session, check reachability) → UPDATE (exchange routing info, advertise/withdraw routes) → NOTIFICATION (signal error, close connection, release resources).
