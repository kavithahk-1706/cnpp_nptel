Lecture 15 HIGH PRIORITY

Transport Layer V — Sliding Window Protocols

Prof. Sandip Chakraborty  ·  IIT Kharagpur

Topics: Sliding window concept, Go-Back-N ARQ, Selective Repeat ARQ, window size bounds and derivations.

↗ NPTEL Video
1 · Motivation — Why Not Stop-and-Wait?

Stop-and-wait (covered in Lec 14) allows only 1 outstanding packet at a time. Sender transmits, then blocks until ACK arrives — severely under-utilising link capacity especially when propagation delay is large.

Core idea of sliding window: allow the sender to transmit multiple frames in a pipeline without waiting for individual ACKs, as long as the number of unacknowledged frames stays within a bounded window size.
2 · Sliding Window — Key Concepts

Sequence Numbers

Every outbound segment carries a sequence number. For an n-bit sequence number field, the sequence space spans 0 to 2n − 1. Numbers wrap around (circular queue fashion) — after the max, the next frame is numbered 0 again.

Sender Window & Receiver Window

Sender Window
The set of sequence numbers the sender is permitted to transmit without waiting for an ACK. Frames within the window can be sent in parallel.
Receiver Window
The set of sequence numbers the receiver is prepared to accept. The receiver will not accept frames outside this window until earlier ones are acknowledged.

Window Sliding Mechanism

When the sender receives an ACK for the lowest unacknowledged frame, the sending window slides one position to the right — exposing the next sequence number that can now be sent. The receiver window slides correspondingly.

Window size = 1: identical behaviour to stop-and-wait. As window size grows, parallelism increases and link utilisation improves.

3 · Handling Loss — ARQ Variants

In a noisy channel, frames or ACKs can get lost. Like stop-and-wait, sliding window protocols use a timeout mechanism: if the timer for the oldest unacknowledged frame expires, the sender takes action. Two strategies exist:

Go-Back-N ARQ
On timeout
Retransmit all frames currently in the sender window
Receiver
Accepts frames in order only. Discards any out-of-order frame.
Rx buffer
Size = 1 (no buffering of out-of-order frames)
Window size
≤ MAX_SEQ = 2n − 1
ACK type
Cumulative ACK
Simpler?
Yes — no out-of-order buffering needed
Selective Repeat ARQ
On timeout
Retransmit only the lost/unacknowledged frame(s)
Receiver
Accepts frames out of order; buffers them
Rx buffer
Size = window size (must store out-of-order frames)
Window size
≤ (MAX_SEQ + 1) / 2 = 2n−1
ACK type
NAK (Negative ACK) or SACK (Selective ACK)
Efficient?
Yes — avoids unnecessary retransmissions
4 · Go-Back-N ARQ — In Depth

Sender State — Three Parameters

base
Sequence number of the oldest unacknowledged frame. Window starts here.
next_seq_num
Next frame to transmit. Can send without waiting for ACK as long as next_seq_num < base + N.
N (window size)
Maximum number of unacknowledged frames allowed simultaneously.

Sender Logic (pseudocode)

on data from app layer:
if next_seq_num < base + N:
→ send packet(next_seq_num)
→ if base == next_seq_num: start_timer()
→ next_seq_num++
else: refuse data (window full)
on timeout:
→ restart_timer()
→ retransmit all frames from base to next_seq_num − 1
on ACK(n) received (not corrupted):
→ base = n + 1 (slide window)

Receiver Logic

on packet received:
if not corrupted AND seq_num == expected_seq_num:
→ deliver data to application
→ send ACK(expected_seq_num)
→ expected_seq_num++
else: discard frame, re-send ACK for last correctly received frame
Key receiver behaviour: Go-Back-N receiver has NO buffer for out-of-order frames. If frame 2 is lost but frames 3 and 4 arrive, those are discarded. On retransmission, 2, 3, and 4 are all sent again. This is the "go back" part — wasteful but simple.

Window Size Constraint for Go-Back-N

With an n-bit sequence number:

MAX_SEQ = 2n − 1  →  Window size W ≤ MAX_SEQ = 2n − 1

You have 2n distinct sequence numbers but the window size can only be 2n − 1. Why not 2n?

Why window ≠ MAX_SEQ + 1 — the ambiguity problem

Scenario: MAX_SEQ = 3 (4 sequence numbers: 0–3), window size = 4 (= MAX_SEQ + 1).

  1. Sender transmits frames 0, 1, 2, 3.
  2. Receiver receives all correctly and sends ACKs — but all ACKs are lost.
  3. Receiver slides its window and now expects the next batch starting from 0.
  4. Sender times out and retransmits frame 0.
  5. Receiver sees frame 0 and thinks it's the new frame 0 — accepts it as new data. Wrong!
With window = MAX_SEQ + 1, the sender and receiver windows can overlap perfectly after a full-window transmission + ACK loss, making it impossible to distinguish a retransmission from a new frame.

With window = MAX_SEQ (= 3), after the same ACK loss: the sender retransmits only frames 0–2. The receiver is now expecting frame 3 — so it correctly identifies 0–2 as duplicates and discards them.

5 · Selective Repeat ARQ — In Depth

Selective Repeat fixes Go-Back-N's wastefulness. Instead of retransmitting the entire window on any loss, the sender selectively retransmits only the missing frames.

Key Differences from Go-Back-N

Receiver Buffers Out-of-Order Frames
If frame 2 is lost but frame 3 arrives, frame 3 is stored in a buffer rather than discarded. When frame 2 is retransmitted and received, the receiver delivers 2 and 3 together in order.
NAK / SACK Mechanism
Receiver sends a Negative ACK (NAK) or Selective ACK (SACK) to explicitly tell the sender which frame is missing. In TCP, this is implemented as SACK options in the TCP header.

Selective Repeat — Timeline Example

Sender sends frame 0, frame 1 →
Receiver gets 0 and 1, sends cumulative ACK(2) →
Sender receives ACK(2), slides window, sends frame 2 →
Frame 2 LOST in channel
Sender sends frame 3 →
Receiver gets frame 3 (out of order) → buffers it → sends NAK(2) →
Sender retransmits frame 2 ONLY
Receiver gets frame 2, delivers 2 + 3 in order

Window Size Constraint for Selective Repeat

With an n-bit sequence number (2n distinct values, MAX_SEQ = 2n − 1):

Window size W ≤ (MAX_SEQ + 1) / 2 = 2n / 2 = 2n−1

For 3-bit sequence numbers (0–7, 8 values): max window size = 8/2 = 4.

Why is the Selective Repeat window size half of Go-Back-N's?

Because SR receivers accept out-of-order frames, the ambiguity problem is worse. Scenario: MAX_SEQ = 3, window = 3 (one more than allowed).

  1. Sender sends 0, 1, 2. Receiver receives all — all ACKs lost.
  2. Receiver window now expects frames 3, 0, 1 (window of 3 wrapping around).
  3. Sender times out, retransmits frame 0.
  4. Receiver is expecting frame 0 as part of the next batch — it accepts it as a new frame. Wrong!

With window = 2 (= (3+1)/2): sender retransmits 0 and 1; receiver is expecting 2 and 3 — it correctly rejects 0 and 1 as duplicates.

The SR window must be ≤ half the sequence number space so that the sender's and receiver's windows never overlap, preventing old retransmitted frames from being mistaken for new ones.

7 · Concept Summary
Sliding window enables pipelined transmission — sender keeps multiple frames "in flight" without waiting for each individual ACK, governed by a window size.

Go-Back-N: simple, wasteful on loss — entire window retransmitted. Window ≤ 2n−1.

Selective Repeat: efficient on loss — only missing frames retransmitted, but requires receiver buffer and NAK/SACK. Window ≤ 2n−1.

Why the window size limits? Both limits prevent a retransmitted frame (old sequence number) from being confused with a new expected frame — the sequence number spaces of sender and receiver must not fully overlap after a worst-case ACK loss scenario.
