IP Routing — I: Intra-domain Routing

Network Layer · Routing algorithms · Autonomous Systems · Distance Vector intro

Core Concept

Routing vs Forwarding

These two terms are used together constantly and it's important to not conflate them. They refer to two distinct jobs a router does.

Forwarding
What
Moving a single packet from an input port to the correct output port
How
Looks up destination address in the local forwarding/routing table and finds the <out-port, MAC addr> pair
Scope
Local, per-packet, happens in real time at line speed
Routing
What
The process of populating the forwarding table with correct entries
How
Routers exchange messages about reachable networks; routing algorithms compute optimal paths
Scope
Network-wide, runs in background, updates periodically or on topology changes
A router is a Layer 3 device. It opens packets up to the network layer, extracts the destination IP, consults its forwarding table, and pushes the packet out the correct interface. To actually build the frame at Layer 2, it then needs the MAC address of the next hop — which is resolved via ARP (covered in a later lecture).
Design Tradeoff

Stability vs Optimality

Stable routing tables are preferred over rapidly fluctuating ones — even if a slightly suboptimal path results. Two reasons:

Manageability
Transient (rapidly-changing) routes are extremely difficult to debug. At internet scale this is a nightmare. A stable table means consistent, reproducible behaviour.
Higher-layer Optimizations
TCP RTT estimation depends on a reasonably consistent path. If routes alternate between 500 ms and 50 ms every few milliseconds, TCP's congestion window and retransmission logic go haywire.
The tradeoff: Maximum stability = nothing ever changes = potentially stale, suboptimal routes. Maximum optimality = routes update instantly with every change = instability cascades. Real protocols balance both — they update on change but avoid thrashing.
Classification

Routing Algorithms

Global (Centralized)
Topology
All routers have complete knowledge of the entire network topology
Advantage
Can compute truly optimal routes
Problem
Not scalable — impractical for internet-scale networks
Decentralized
Topology
Each router only knows its immediate neighbors and information shared from them
Advantage
Scales to internet size; no single point of knowledge required
Problem
Convergence is slower; can suffer routing loops and counting-to-infinity

Intra-domain vs Inter-domain Routing

Intra-domain
Scope

Priority

Protocols
RIP (Distance Vector), OSPF (Link State)
Inter-domain
Scope
Between different organizations / Autonomous Systems — decentralized, internet scale

Protocols

Key Challenge

Optimality: Network as a Graph

Model the network as a weighted directed graph where nodes = routers and edges = links. Each edge carries a cost based on one or more metrics:

Latency Bandwidth Utilization Queue Length Hop Count

Goal: Find the lowest-cost path between source and destination nodes. Each router individually computes cost; in a distributed system they cooperate to converge on globally optimal (or near-optimal) paths. If a truly optimal path cannot be found, a default route acts as a fallback — any unrecognized destination gets forwarded there.

Key Challenge

Scaling

Every router must be able to forward any destination IP address. A naive approach — one routing table entry per IP address — is completely infeasible. The internet has on the order of 108–109 addressable devices.

Solution: Entry covers a range of addresses. Instead of one row per host, a single entry covers an entire network block (e.g. a /24). One entry in an upstream router handles all 20,000 devices on IIT KGP's campus. This is called address aggregation.

This only works if addresses are allocated in a structured, hierarchical way — i.e., based on network topology. It does not work with randomly assigned addresses (like Ethernet MAC addresses, which are assigned by manufacturer and carry no topological meaning). This is exactly why IP addresses must be allocated based on network structure.

Analogy: A postman can bundle mail for houses 1–6 on the same street. If the houses were numbered 1, 49, 101, 216 at random, bundling is impossible.

Packet Delivery

Direct vs Indirect Delivery

Direct Delivery
Source and destination are on the same network. Packet goes straight from sender to receiver without passing through a router. Final delivery is always direct — even in multi-hop paths, the last hop from the last router to the destination host is a direct delivery.
Indirect Delivery
Source and destination are on different networks. Packet must pass through one or more routers. Each router performs a next-hop lookup and forwards the packet hop by hop until it reaches the destination network, after which final delivery is direct.
Routing Table Design

Route Method vs Next-hop Method

Route Method
Entry
Destination → Full explicit path (e.g. R1 → R2 → Host B)
Problem
Each router must store the complete end-to-end path. Table sizes are enormous. Any topology change invalidates stored paths.
Next-hop Method used in practice
Entry
Destination → Next hop only (e.g. to reach Host B, send to R1)
Why better
Each router only stores one next hop per destination. Individual routers maintain local tables and the correct path emerges from cooperative forwarding decisions.
Routing Table Design

Host-specific vs Network-specific Routing

Host-specific
Entry
One row per individual host (A → R1, B → R1, C → R1, D → R1...)
Problem
Table grows linearly with number of hosts — completely unscalable
Network-specific used in practice
Entry
One row per network block (Network N2 → R1). All hosts A, B, C, D on N2 are covered by this single entry.
Why better
Dramatically reduces table size. Routers forward based on network address, not individual host address.
Fallback Mechanism

Default Route

When a packet's destination does not match any specific entry in the routing table, the router forwards it to the default route — a catch-all entry typically pointing toward an upstream router or the rest of the internet. Without it, unrecognized destinations would simply be dropped.

Destination: N2 → Next Hop: R1
Destination: Any other → Next Hop: R2 (default router)
Router Internals

Basic Routing Module (Classless Addressing)

When a packet arrives, the forwarding module does the following in order:

1. Extract destination IP address from packet header
2. Apply subnet masks (longest prefix first) to find matching network entry
3. Retrieve next-hop IP address and output interface number from table
4. Pass to ARP — resolve next-hop IP → MAC address for Layer 2 framing
5. Forward frame out the identified interface

The routing table stores rows of: Mask (/n) | Network Address | Next-hop Address | Interface. On a lookup miss across all entries, the packet hits the default entry (mask = Any, network = Any).

Forwarding Mechanism

Longest Prefix Match (LPM) — Worked Examples

The routing table is searched from longest mask to shortest. The first match wins. This ensures the most specific route is always preferred.

Scaling Mechanism

Address Aggregation

Multiple downstream organizations can be summarized into a single upstream routing entry if their IP blocks are contiguous and allocatable under a common prefix.

R1 (local router)
4 entries: /26 each for Org 1–4
140.24.7.0/26, .64/26, .128/26, .192/26
R2 (upstream router)
1 aggregated entry: /24
140.24.7.0/24 → m0
Covers all 4 organizations

From R2's perspective, all four organizations collapse into a single /24 entry. If there were 400 organizations under R1 instead of 4, R2's table would still have just one entry. This is the power of hierarchical, topology-based address allocation.

Internet Architecture

Autonomous Systems (AS)

The internet is not a single monolithic network. It is composed of many independently administered networks called Autonomous Systems. Examples: IIT Kharagpur, Stanford, a large ISP, a corporation.

Goal 1
Each AS chooses its own local (intra-domain) routing algorithm. IIT KGP can run OSPF internally; another AS can run RIP. Nobody else's business.
Goal 2
Each AS sets policies for non-local (inter-domain) routing — which external ASes it will accept transit traffic for, which it won't, based on business agreements.
Identifier
Each AS is assigned a unique 16-bit AS Number (ASN) by a regional internet registry.

AS Traffic Types

Type Description
Local Traffic Source and destination both within the same AS
Transit Traffic Traffic passing through an AS to reach another AS
Stub AS Connected to only one other AS — single entry/exit point. Cannot carry transit traffic.
Multihomed AS Connected to multiple ASes for redundancy, but carries no transit traffic (refuses to be a middle-man)
Transit AS Connected to multiple ASes and carries transit traffic — acts as a routing intermediary (e.g. large ISPs)
Algorithm Types

Intra-domain Routing Algorithms

Within an AS, optimality is the primary goal (unlike inter-AS routing where policy dominates). Two fundamental approaches exist:

Distance Vector (DV)
Knowledge
Each node only knows costs to its immediate neighbors
Exchange
Shares its full routing table with neighbors periodically and on change
Overhead
Low — only local state needed, smaller memory footprint
Problems
Harder to debug; susceptible to routing loops and count-to-infinity
Protocol
RIP (Routing Information Protocol)
Link State (LS)
Knowledge
Each node has a complete view of the entire network topology
Exchange
Floods link-state advertisements to all nodes; each independently runs Dijkstra
Overhead
Higher — requires storing and maintaining global state
Problems
More bandwidth and memory intensive, but loop-free and simpler to debug
Protocol
OSPF (Open Shortest Path First)

Algorithm Deep-dive Preview

Distance Vector Routing — Overview

Each node maintains a table of (destination, cost, next-hop) for every known destination. The table is shared with immediate neighbors periodically (e.g. every 30 s) and on change (triggered update).

Initialization: A node starts knowing only directly connected neighbors with their link costs. All other destinations are set to cost = ∞ (unknown). Over successive exchanges, costs propagate and tables converge. 35:31

Example: Node A is directly connected to B (cost 5) and C (cost 2). A cannot reach E directly, so cost(A→E) = ∞ initially. C knows E with cost 4. After C sends its table update to A, A learns: cost(A→E) = cost(A→C) + cost(C→E) = 2 + 4 = 6, via C. Table converges iteratively.

The update process and convergence mechanics (Bellman-Ford equation, count-to-infinity, split horizon) are covered in detail in the next lecture.
Lecture Summary

Key Takeaways

Forwarding ≠ Routing
Forwarding = fast per-packet table lookup. Routing = background process that builds those tables. Both are router responsibilities.
Stability vs Optimality
Real protocols don't chase the absolute optimal path on every update — unstable routes break TCP and make debugging impossible.
Longest Prefix Match
Routing table lookup always tries the most specific (longest) mask first. Default route is the catch-all of last resort.
Address Aggregation
Works only when addresses are allocated hierarchically (IP), not randomly (MAC). It's the key to keeping routing tables manageable at internet scale.
Autonomous Systems
Internet = collection of independently administered ASes. Intra-AS = optimality-first. Inter-AS = policy-first. These require fundamentally different routing protocols.
DV vs LS
Distance Vector (RIP) = local knowledge, light overhead, loop-prone. Link State (OSPF) = global topology, heavier overhead, loop-free.
