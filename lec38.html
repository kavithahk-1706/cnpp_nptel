Routing Protocol Taxonomy

Every router maintains a routing table (also called a forwarding table) — a mapping from destination to outgoing interface. The goal of routing protocols is to keep this table accurate and optimal as network topology changes.

Static Routing
Routes are manually/pre-programmed by administrators. No automatic discovery. No adaptation to failures or topology changes.
Dynamic Routing
Routers automatically discover, calculate, and maintain paths. Adapts to topology changes. Protocols differ in how they discover and recompute routes.

Four Categories of Dynamic Routing Protocols

Intra-domain
Distance Vector

Each router shares costs to neighbors. Localized view. Example: RIP

Intra-domain
Link State

Each router advertises all link states. Global topology view. Example: OSPF

Inter-domain
Path Vector

Advertises destination + full AS path. Used between ASes. Example: BGP

Proprietary
Hybrid

Combines DV + LS attributes. Faster convergence, higher resource cost. Used in controlled/proprietary environments.

Autonomous Systems (AS)

The internet is partitioned into Autonomous Systems (AS) — groups of networks, hosts, and routers under a single administrative/organizational control (e.g., IIT Kharagpur's campus network = one AS). Each AS has a globally unique AS number.

Intra-domain routing = routing within a single AS. Uses Distance Vector or Link State protocols.
Inter-domain routing = routing between different ASes. Uses Path Vector (BGP).

Managing a small AS is far simpler than managing the whole internet. Dynamic routing is necessary because: (1) the internet topology changes constantly — links and routers appear and disappear; (2) there is no single administrative control over the whole internet, so each router must update its own table independently.

Distance Vector Routing

Each router maintains a distance vector table storing the cost (distance/metric) to every known destination, along with the next hop to reach it. The router only directly measures costs to its immediate neighbors; everything else is learned indirectly.

Core principle: "I share my knowledge of the whole network with my immediate neighbors — periodically and on any change." The process resembles a whisper-chain propagating through the AS until all routers converge to a consistent view.

How Distance Vector Updates Work

Each router sends its full routing table to its neighbors either (a) periodically (e.g., every 30 seconds in RIP) or (b) triggered by a topology change. On receiving a neighbor's table, a router applies the Bellman-Ford update rule:

Update rule: New cost to destination D via neighbor N = cost(self → N) + cost(N → D as advertised by N).
If this is less than the current best cost, update the entry and set next-hop to N.

Distance Vector: Instability Problems

Distance Vector routing suffers from instability when a link fails, because stale routing information can circulate and create routing loops with slowly growing costs — the count-to-infinity problem.

Two-Node Instability 20:15

Setup: X — A — B. A is directly connected to X (cost 2); B reaches X via A (cost 6). After the A–X link fails:

Step 1: A sets cost to X = ∞ (link down).
Step 2: Before A's update reaches B, A receives B's stale advertisement (B→X = 6). A thinks: "I can reach X via B: 4 + 6 = 10." A updates X = 10, next-hop = B.
Step 3: B receives A's updated advertisement (A→X = 10). B thinks: "Via A: 4 + 10 = 14." Updates to 14.
Outcome: Costs keep incrementing (10 → 14 → 18 → …) until they hit the max metric (∞ = unreachable). This is count-to-infinity.

Three-Node Instability 22:45

Setup: X — A — B — C (A directly connected to X, cost 2). After A–X fails, A sends ∞ updates to B and C — but the packet to C is lost in transit.

B correctly updates X = ∞ (received A's update).
C still holds stale entry X=5 via A (packet was lost).
Next round: B hears C's advertisement (C→X=5) and updates to 5+3=8.
A then hears B (B→X=8) and updates to 8+4=12. Count-to-infinity resumes.
Packet loss during convergence amplifies the instability to a three-node cycle.
Why this matters: In a large AS with hundreds of routers, stale advertisements arriving out of order can cause cascading instability across many nodes, not just 2 or 3. Solutions (split horizon, poison reverse, hold-down timers) exist but are not covered in this lecture.
Link State Routing

Each router has a complete view of the entire AS topology — not just neighbor costs. This is achieved by flooding Link State Advertisements (LSAs) to every other router in the domain.

A link state is a description of a router's interface: IP address, subnet mask, network type, cost, and which neighboring routers it connects to. The collection of all link states across the network forms a Link State Database (LSDB). Ideally, every router's LSDB is identical.

Link State Approach — 4 Steps 24:25

Step 1 & 2 — Discover & Advertise
Each router identifies all routing devices on directly connected networks and generates an LSA listing all directly connected links and their costs. LSAs are flooded to every other router in the network.
Step 3 & 4 — Build Topology & Compute
Using received LSAs, each router builds an identical topology database (network graph). It then runs Dijkstra's shortest path algorithm on this graph to compute optimal routes and updates its routing table accordingly.

Dijkstra's Shortest Path Algorithm (Link State)

Once the LSDB is complete, each router runs Dijkstra's algorithm from itself as the root to compute shortest paths to all other nodes. 28:51

Two lists maintained:
Tentative list: Nodes discovered but not yet finalised. Stores best-known cost so far.
Permanent list: Nodes with confirmed shortest paths (cost is final, won't change).

Algorithm Steps

1→Set root to the local node. Add it to the tentative list with cost 0.
2→If tentative list is empty → STOP.
3→From the tentative list, select the node with the smallest cumulative cost. Move it to the permanent list.
4→For each unprocessed neighbor of that node: add it to the tentative list if not already present. If it is already in the tentative list but with a larger cost, replace with the new lower cost.
5→Go to step 2.

OSPF — Open Shortest Path First

OSPF is the predominant intra-domain link state routing protocol. It uses LSAs flooded within an AS (or within areas of an AS) and runs Dijkstra on the collected LSDB to populate routing tables. Details of OSPF message types, area hierarchy, and DR/BDR election are covered in subsequent lectures.

Link State Summary: (1) Share knowledge about your neighborhood — (2) with every other router — (3) only when there is a change (or periodically). OSPF implements exactly this using LSA flooding.
Path Vector Routing (Brief — BGP)

Used for inter-domain (between ASes) routing. Similar to distance vector but instead of advertising just the cost, each entry includes the full ordered sequence of ASes on the path to the destination. This allows loop detection and policy-based routing. BGP (Border Gateway Protocol) is the standard inter-domain path vector protocol. Covered in detail in later lectures.

Concept Summary
Distance Vector
Each router knows only neighbor costs. Shares full routing table with neighbors. Bellman-Ford update. Prone to count-to-infinity. Slow convergence. RIP is the example protocol.
Link State
Each router floods its link costs (LSAs) to everyone. All routers build an identical topology DB. Dijkstra computes shortest paths. No count-to-infinity. Fast convergence. OSPF is the example protocol.
Path Vector
Inter-domain. Advertises destination + full AS path sequence. Enables loop detection via path inspection. Policy-based routing possible. BGP is the standard protocol.
Key exam traps:
• DV shares its full routing table with neighbors; LS shares only its direct link costs (LSA) with everyone — don't confuse what's shared and with whom.
• Count-to-infinity is a DV problem, not LS.
• Dijkstra requires a complete topology view — this is only possible in LS, not DV.
• RIP = Distance Vector. OSPF = Link State. BGP = Path Vector. Memorise this cold.
