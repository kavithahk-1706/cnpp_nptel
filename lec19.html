Transmission Control Protocol — I
Primitives

Prof. Sandip Chakraborty · IIT Kharagpur · Transport Layer

▶ Lecture Video (NPTEL)
Motivation

What TCP Was Built To Solve

TCP was specifically designed to provide reliable, end-to-end byte streaming over an unreliable internetwork. The IP-based network layer is fundamentally unreliable — routers can drop packets due to buffer overflow, and IP does nothing about it. TCP sits on top of this and handles reliability itself.

On top of that, the underlying network is heterogeneous — one path might cross a wireless hop, then a wired Ethernet segment, then an optical fiber cable, each with different protocols at the data link layer, different error rates, and different delay characteristics. TCP dynamically adapts to all of it and is robust to many kinds of failures.

Key stat: More than 80% of internet traffic uses TCP for end-to-end data transfer.

Service Model

TCP Connection Properties 05:40

Full Duplex
Both hosts can send and receive data simultaneously over a single TCP connection. Host A → B and B → A at the same time.
Point to Point
TCP is strictly between two endpoints. It does not support multicasting (one to group) or broadcasting (one to all).

Socket — The TCP Endpoint Abstraction

In Unix-based systems, a TCP connection is represented as a socket — what the generic transport layer model called a "logical pipe." A socket is uniquely identified by a 6-tuple:

Source IP + Source Port + Source ISN + Dest IP + Dest Port + Dest ISN

ISN = Initial Sequence Number. This 6-tuple is the full identity of the connection from both ends.

Once a socket is established, hosts interact with it using write() (to send data into the socket) and read() (to pull data out of the receive buffer). The transport layer handles reliable delivery; the network layer handles routing to the correct IP.

Unix Implementation Detail

inetd — Internet Daemon 08:17

In Unix, a single super-daemon process called inetd (internet daemon) runs continuously, listening on multiple well-known ports simultaneously. It avoids having to keep every protocol daemon running at all times.

How it works: inetd listens on a port (e.g., port 80 for HTTP). When the first incoming connection arrives, inetd forks — creates a child process with a new process ID — and starts the relevant daemon (e.g., httpd for HTTP, ftpd for FTP at port 21). That daemon then manages the socket for the duration of the connection.
Core Abstraction

TCP is a Byte Stream, Not a Message Stream 09:44

Every byte in a TCP connection has its own unique 32-bit sequence number. TCP does not preserve message boundaries — it does not care how many write() calls the sender made or how big each chunk was.

Critical point: Message boundaries are not preserved end-to-end. The receiver has no way of knowing how data was originally chunked by the sender.

Concrete Example 09:44

Host B sends three segments to Host A: bytes 100–200 (100 bytes), 201–250 (50 bytes), and 251–400 (150 bytes). Segments 2 and 3 get lost. When Host B retransmits, the flow control algorithm may cause it to resend as two 100-byte segments (201–300 and 301–400) — a completely different segmentation. TCP retransmits bytes, not the original segments.

💡 Visual diagram at 09:44 — the slide shows the three-segment byte layout with segment 2 and 3 dropped, and how retransmission works as a byte range not a segment copy. Worth watching.

Write vs Read Chunk Mismatch

The sender may write in 512-byte chunks; the receiver may read in 1024-byte chunks. TCP accommodates both silently — the application at the receiving end has no knowledge of how data was written on the sending side.

TCP can also accumulate data from several write() calls into one segment or split one write() call into multiple segments depending on flow and congestion control state.

Protocol Mechanics

TCP Header Structure 15:46

A TCP segment = 20-byte fixed header + optional fields + data (payload). Control-only segments (SYN, FIN) carry no data.

Field Size Purpose
Source Port 16 bits Identifies sending application/process
Destination Port 16 bits Identifies receiving application/process
Sequence Number 32 bits Byte-level sequence number of the first data byte in this segment
Acknowledgement Number 32 bits Next expected byte from the other side (cumulative ACK)
Header Length 4 bits Length of the TCP header in 32-bit words
Flag Bits 6–9 bits Control flags — see below
Window Size 16 bits Receiver-advertised available buffer space (for flow control)
Checksum 16 bits Error detection over header + data
Urgent Pointer 16 bits Offset to urgent data — bypasses normal FIFO queue
Options Variable Optional extensions (e.g., MSS, timestamps)
Data Variable Payload from the upper layer (application data)

Control Flag Bits

SYN
Connection initiation. Set in the handshake messages to establish a new TCP connection.
FIN
Connection termination. Signals that this side has finished sending. Used in the connection teardown process.
ACK
Acknowledgement. The Acknowledgement Number field is valid — indicates bytes received up to ACK# - 1.
Urgent Pointer detail: The transport layer queue is a FIFO structure. Normally, bytes are sent in order. If an application marks data as urgent (via socket programming), TCP creates a segment immediately with the urgent bit set, bypassing the normal queue order. The urgent pointer field points to the offset of the last urgent byte in the segment.
Sequence & Acknowledgement

32-bit Byte Sequence Numbers 18:37

Every byte on a TCP connection has its own 32-bit sequence number. TCP uses sliding window-based flow control at the byte level.

Example: If the ACK number received is 31245, it means the receiver has correctly received all bytes up to and including byte 31244, and it is now expecting byte 31245. This is a cumulative acknowledgement — one ACK confirms all bytes before it.

The acknowledgement number always points to the next expected byte, not the last received byte. This is standard cumulative ACK behavior — if ACK = N, bytes 0 through N−1 are confirmed received.

Segment Construction

How TCP Segments Are Formed 20:22

Segment size is constrained by two parameters:

Constraint 1
IP Payload Limit
Maximum data that can go into an IP fragment: 65,515 bytes. TCP segment cannot exceed this.
Constraint 2
MTU — Maximum Transmission Unit
The max frame size on a link. Different per technology (Wi-Fi, Ethernet, optical). Often the tighter constraint in practice.

Path MTU Discovery 25:07

Modern TCP uses Path MTU Discovery, a mechanism implemented inside ICMP (Internet Control Message Protocol) at the network layer. It discovers the MTU of every link along the path from source to destination, then takes the minimum — because a packet cannot exceed the smallest MTU on any link in the path.

Example: If path links support 512 B → 1 KB → 1 KB → 256 B, the path MTU is 256 bytes. TCP must not send segments larger than that.

Maximum Segment Size (MSS)

MSS is established during the TCP connection setup phase using the path MTU discovery result (and other parameters like available buffer size). It is the largest amount of data (excluding header) that TCP will put in a single segment.

Segment Size Decision Algorithm 25:07

After each received ACK (which also carries the current receiver advertised window size), the sender checks:

If receiver window < MSS → construct a single segment (can't fill an MSS-sized segment anyway).

If receiver window ≥ MSS → create as many full MSS-sized segments as the window allows.

If sender buffer has very little data → don't wait for a full MSS worth of data to accumulate. Send what's available. Waiting would unnecessarily delay small messages.

Design Challenges

What TCP's Design Has to Handle 28:56

Dynamic Segment Construction
Retransmissions don't guarantee the same segments — a lost 50-byte + 150-byte pair may be retransmitted as two 100-byte segments. TCP handles bytes, not segments.
Out-of-Order Delivery
IP decides routing per-packet, and load balancing can send different packets on different paths. TCP segments can arrive out of order and the receiver must handle them correctly.

Out-of-Order Handling — The Smart Approach

Naively applying go-back-N here would waste bandwidth: "received something out of order → discard everything after the gap → wait for full retransmission." TCP is smarter:

The receiver buffers out-of-order segments. When the missing bytes arrive and fill the gap, the entire contiguous block is now valid. The receiver sends a cumulative ACK for the whole range. The sender stops retransmitting and advances its window. Network resources saved.

Example: Receiver has bytes 100–120 and 151–500 (gap at 121–150). When the retransmitted segment arrives covering bytes 121–160, the receiver chops out 121–150, inserts it, now has 100–500 contiguously. Cumulative ACK for byte 501 sent. Sender stops immediately.

Flow Control

Window Size Field 32:10

TCP uses a variable-size sliding window protocol for flow control. The 16-bit window size field in the TCP header is how the receiver tells the sender how much buffer space it currently has available.

Window = 0 → sender must stop (receiver buffer full) Window > 0 → sender may transmit up to that many bytes

The final TCP acknowledgement is always a combination of two things: the acknowledgement number (what bytes have been received) and the advertised window size (how much more can be sent). The sender uses both to tune its transmission rate.

The actual sender window size is:

Sender Window = min(Congestion Window, Receiver Advertised Window)
Congestion window grows additively (AIMD), receiver window reflects available buffer. The tighter one wins.
