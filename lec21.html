TCP Flow Control Mechanism

Sliding Window with Go-Back-N

TCP uses a sliding window flow control algorithm based on the Go-Back-N ARQ principle. On a timeout, the sender retransmits everything currently inside the sender window. Sequence numbers start from the ISN negotiated during the three-way handshake; the example below uses 0 for simplicity.

Worked Example — Receiver buffer capacity: 4 kB
1 Application writes 2 kB → sender transmits SEQ=0, 2048 bytes. Receiver buffer: 2 kB occupied, 2 kB free. Receiver sends ACK=2048, WIN=2048.
2 Application writes another 2 kB → sender transmits SEQ=2048, 2048 bytes. Receiver buffer: 4 kB full. Receiver sends ACK=4096, WIN=0. Sender is blocked.
3 Receiver application reads 2 kB. Buffer: 2 kB free. Receiver sends ACK=4096, WIN=2048 (window update). Sender unblocks.
4 Sender transmits 1 kB → SEQ=4096, 1024 bytes. Receiver sends ACK=5120, WIN=1024. And so on.
★ If an ACK is lost, the sender retransmits the entire window contents per Go-Back-N. Unacknowledged data stays in the sender buffer until the ACK arrives.
Small Segment Problem

The Overhead Problem — Telnet Example

In interactive applications like telnet (remote command execution), every keystroke generates a 1-byte payload. TCP's minimum segment header is 20 bytes, so each keypress produces a 21-byte segment carrying 1 byte of actual data — a 95% header overhead. Each such segment also triggers an ACK and window update from the receiver, compounding the waste.

Root problem: Tiny data units from the application result in disproportionate control traffic, consuming bandwidth without transferring meaningful data.

Two independent mechanisms address this from opposite ends of the connection:

Solution 1 — Receiver Side

Delayed Acknowledgements

Instead of immediately acknowledging every received segment, the receiver delays the ACK and window update for up to 500 ms, hoping to accumulate more incoming data before sending a single, larger acknowledgement. This prevents the sender from being triggered to immediately send another tiny segment.

Delayed ACK reduces the number of ACK segments and window update messages. However, the sender is still free to send small segments at will — delayed ACK only controls the receiver's response behaviour.
Solution 2 — Sender Side

Nagle's Algorithm

Nagle's algorithm controls the sender directly. The rule is simple: when data arrives at the sender in small pieces, send only the first piece immediately and buffer everything else until the ACK for that first piece arrives. Once the ACK arrives, all buffered data is flushed in a single larger segment, and the process repeats.

Core invariant: At any given instant, only one short packet may be outstanding in the network. This prevents a flood of tiny segments from entering the network simultaneously.

By the time the ACK for the first byte returns, the sender has typically accumulated several more bytes in its buffer, which are then sent together — dramatically improving effective payload ratio.

When NOT to use Nagle's Algorithm

Nagle's algorithm intentionally introduces delay — it withholds data until an ACK arrives. For delay-sensitive applications (real-time games, interactive GUIs, trading systems), this latency is unacceptable. Such applications disable Nagle's algorithm using the TCP_NODELAY socket option.

Critical Interaction: Nagle + Delayed ACK = Starvation

Do not combine Nagle's algorithm and delayed acknowledgements. Nagle's algorithm waits for an ACK before sending buffered data. Delayed ACK withholds that ACK for up to 500 ms. The result: the sender waits for an ACK that the receiver is deliberately not sending, while the receiver waits for more data the sender is deliberately not sending. Both sides stall — this is a deadlock-like starvation causing severe application response degradation.
Silly Window Syndrome

The Problem

Silly Window Syndrome (SWS) occurs when the receiver-side application reads data very slowly (e.g., 1 byte at a time) while the sender has large amounts of data to transmit. The sequence of events creates a self-reinforcing loop of tiny transmissions:

1Receiver buffer is full → sends ACK, WIN=0. Sender blocked.
2Application reads 1 byte → 1 byte free in buffer → receiver sends WIN=1 (window update).
3Sender receives WIN=1 → sends 1 byte of data (1 byte payload + 20 byte header = 21 byte segment).
4Buffer full again → WIN=0. Back to step 1. Infinite loop of 21-byte segments.

The same bandwidth-wasting problem as the telnet case, but triggered from the receiver side rather than the sending application.

Clark's Solution

Clark's solution addresses SWS at the receiver: do not send a window update until sufficient buffer space is available. The receiver suppresses window updates when only a trivially small amount of space has opened up. "Sufficient" is implementation-defined — typically a meaningful percentage of the total buffer size, or at least enough for a full-sized segment.

Combined View

Nagle's Algorithm and Clark's Solution are Complementary

Nagle's Algorithm (Sender-side)
ProblemSending application delivers data 1 byte at a time to TCP.
FixBuffer all small pieces at sender until ACK received, then send combined segment.
EffectReduces number of small segments entering the network from sender.
Clark's Solution (Receiver-side)
ProblemReceiving application fetches data from TCP buffer 1 byte at a time.

EffectPrevents sender from being triggered to send 1-byte segments by tiny window updates.

Unlike the Nagle + Delayed ACK combination, Nagle + Clark does not cause starvation — they work on different sides of the connection and do not create a mutual waiting dependency. Both may introduce some latency overhead, but no deadlock scenario is possible.

PSH Flag — Override for Real-Time Applications

When an application needs data delivered immediately, bypassing both Nagle's algorithm and the Clark solution, it sets the PSH (Push) flag in the TCP header. This signals the sender to create and transmit a segment immediately without waiting for the buffer to accumulate, and signals the receiver to pass data to the application layer immediately without waiting for a full buffer.

Out-of-Order Segment Handling

Duplicate Acknowledgement (DUPACK)

TCP uses cumulative acknowledgements — an ACK number N means "I have received everything up to byte N−1." When an out-of-order segment arrives, TCP buffers it but sends a duplicate ACK repeating the last successfully received in-order byte, signalling that there is a gap.

Worked Example — Received bytes: 0, 1, 2, [3 missing], 4, 5, 6, 7
1Bytes 0, 1, 2 received → ACK=3 (expecting byte 3 next).
2Byte 4 arrives (out of order, gap at 3) → TCP buffers byte 4, sends DUPACK: ACK=3 again.
3Bytes 5, 6, 7 arrive → each triggers another DUPACK: ACK=3. All buffered.
4Multiple DUPACKs trigger the congestion control algorithm (details in next lecture). After timeout, sender retransmits byte 3.
5Byte 3 received → buffer now holds bytes 3–7 contiguously → receiver sends ACK=8 (cumulative, acknowledging all).
DUPACK is a critical signal in TCP: receiving 3 duplicate ACKs is used by the congestion control algorithm as an indicator of packet loss without waiting for a full timeout (fast retransmit — covered in the next lecture).
TCP Timer Management

Retransmission Timeout (RTO)

When a segment is sent, TCP starts a retransmission timer. If an ACK arrives before the timer expires, the timer stops. If the timer expires first, the segment is retransmitted and the congestion control algorithm is triggered. The fundamental question is: what should the RTO value be?

The natural candidate is a multiple of the Round Trip Time (RTT): RTO = n × RTT. But this requires a reliable RTT estimate — and that estimate is difficult to obtain at the transport layer.

Why RTT Estimation is Hard at the Transport Layer

Data Link Layer RTT
Two directly connected nodes. Single link, minimal variability. RTT distribution is narrow and stable — a simple average gives a reliable estimate.
Unit: microseconds, tight bell curve.
Transport Layer RTT
End-to-end across the entire internet, through multiple intermediate routers with dynamic queues. RTT distribution is wide and multimodal — the average is an unreliable estimate with high deviation.
Unit: milliseconds, flat spread over multiple peaks.

Setting RTO based on a naïve RTT average at the transport layer produces spurious RTOs — the timer fires prematurely during normal high-load conditions, triggering unnecessary retransmissions. The solution is a dynamic algorithm that continuously adapts.

RTT Estimation — Jacobson's Algorithm (1988)

SRTT: Smoothed Round Trip Time

For each TCP connection, a variable SRTT (Smoothed Round Trip Time) is maintained as the best current estimate of RTT to the destination. Each time an ACK is received, the measured RTT R is used to update SRTT via an Exponentially Weighted Moving Average (EWMA):

SRTT = α · SRTT + (1 − α) · R

α = smoothing factor controlling how quickly old values are forgotten. Typically α = 7/8.
A high α means old values persist longer (more inertia). A low α means recent measurements dominate (more reactive).

The same timer used for retransmission also measures RTT: when the segment is sent the timer starts; when the ACK arrives the elapsed time is R. If the timer expires before the ACK arrives, R is not measured (see Karn's algorithm below).

Problem with EWMA Alone: High-Load Variance

The initial TCP implementation set RTO = 2 × SRTT. This proved too inflexible: under high network load, RTT fluctuations increase dramatically. A fixed multiplier on SRTT fails to track the variance, producing spurious timeouts when actual RTT momentarily spikes above 2×SRTT.

Full RTO Formula: Incorporating RTTVAR

The solution is to track the variance of RTT measurements alongside the mean, and fold it into the RTO calculation:

Step 1 — Update variance:
RTTVAR = β · RTTVAR + (1 − β) · |SRTT − R|
β = 3/4 (typically)

Step 2 — Compute RTO:
RTO = SRTT + 4 × RTTVAR

When network load is high and RTT fluctuates widely, RTTVAR increases, which automatically inflates RTO — preventing spurious timeouts.
Why the coefficient 4? It is somewhat arbitrary — Jacobson chose it partly because 4 = 2², allowing the multiplication to be implemented as a single binary left-shift, keeping the computation lightweight on 1988-era hardware. The Jacobson paper is notable for using only integer addition, subtraction, and bit-shift operations throughout.

RTT Estimation on Retransmission — Karn's Algorithm

The Retransmission Ambiguity Problem

If a segment is lost and retransmitted, the ACK that eventually arrives is ambiguous — it could be acknowledging the original transmission or the retransmission. Using the elapsed time from the original send as an RTT sample would corrupt the SRTT estimate with a grossly inflated value.

Karn's Algorithm

Rule 1: Do not update SRTT or RTTVAR for any retransmitted segment. RTT samples are only taken from segments that were transmitted exactly once.

Rule 2: On each successive retransmission timeout, double the RTO: RTO → 2·RTO → 4·RTO → … until the segment is successfully acknowledged. This is exponential backoff. Once an ACK is received for a successfully delivered (non-retransmitted) segment, RTO is reset to the Jacobson-computed value.

Exponential backoff serves a dual purpose: it avoids congesting the network further when segments are being dropped, and it gives the network progressively more time to recover. The doubling sequence (RTO, 2·RTO, 4·RTO, …) is capped in practice to prevent indefinite growth.

Other TCP Timers

Summary of All TCP Timers

TimerPurposeBehaviour on Expiry
Retransmission (RTO) Detect lost segments during normal data transfer. Retransmit segment; trigger congestion control; double RTO (Karn).
Persistent Prevent deadlock when receiver advertises WIN=0. The sender has data to send but is blocked — without this timer, the connection would stall permanently if the window-update ACK is lost. Send a probe segment to the receiver to solicit an updated window advertisement.
Keepalive Detect and close idle connections. After a long period of inactivity, the connection may have been abandoned by the remote peer without a proper FIN exchange. Close the connection (or send keepalive probes first, depending on implementation).
TIME_WAIT Active-close side waits after sending final ACK, to absorb delayed duplicates and prevent them being misinterpreted by a new connection on the same port pair. Fully close the connection. Duration = 2 × maximum packet lifetime.
Persistent timer and WIN=0 deadlock: When the receiver sends WIN=0, the sender stops. If the subsequent window-update ACK (WIN > 0) is lost in transit, both sides wait indefinitely. The persistent timer breaks this deadlock by having the sender proactively probe the receiver rather than waiting forever.
Concept Summary

Flow Control Mechanism

TCP uses sliding window with Go-Back-N. The receiver advertises its available buffer space in the WIN field of every ACK. The sender may not have more unacknowledged data outstanding than the advertised window. When WIN=0 is received, the sender blocks until a non-zero window update arrives.

Small Segment Optimisations

Delayed ACK (receiver): hold ACK for up to 500 ms hoping to batch it with a window update or piggyback on a data segment going the other way. Nagle's algorithm (sender): send first small piece immediately, buffer all subsequent small pieces until that ACK is received, then flush everything in one segment. Clark's solution (receiver, for silly window syndrome): suppress window update until sufficient buffer space is available. Nagle + Clark are complementary and safe. Nagle + Delayed ACK causes starvation — do not use together. The PSH flag bypasses both mechanisms for latency-critical data.

Out-of-Order and DUPACK

Out-of-order segments are buffered. A duplicate ACK (DUPACK) is sent for each one, repeating the last in-order ACK number. Three DUPACKs triggers fast retransmit (congestion control, next lecture). Once the missing segment arrives, a single cumulative ACK covers all contiguous buffered data.

RTO and RTT Estimation

RTO is set dynamically. Jacobson's algorithm maintains SRTT (EWMA of measured RTT, α=7/8) and RTTVAR (EWMA of variance, β=3/4). RTO = SRTT + 4·RTTVAR. Karn's algorithm ensures RTT samples are only taken from segments sent exactly once (no retransmissions), and doubles RTO on each successive retransmission until the segment gets through.
