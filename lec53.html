Lecture 53 Medium Priority

Connecting LANs & VLAN

Data Link Layer — Prof. Soumya Kanti Ghosh, IIT Kharagpur

Topics: Layer-wise LAN connectivity · Repeaters · Bridges · Bridge loop problem · Spanning Tree Protocol · Backbone networks · VLAN basics, operation, tagging, and types

1 — Layer-wise Connectivity Recap

Each intermediate device in a network can process packets up to the layer at which it operates. A higher-layer device inherits all capabilities of devices below it — at a cost of greater processing overhead and higher price.

Layer Device Function Domains
L1 — Physical Hub / Repeater Signal regeneration only Same collision + broadcast domain
L2 — Data Link Bridge / L2 Switch MAC-based frame forwarding Separate collision; same broadcast domain
L3 — Network Router / L3 Switch IP routing between networks Separate collision + broadcast domain
L4–L7 Gateway Application-to-application connectivity Full protocol translation possible
Cost tradeoff: Higher layer = more capabilities + more processing time + significantly higher cost. You design for the minimum layer your use case needs.

The network should also be scalable — backbone infrastructure is expensive to change, so the design must accommodate growth without full replacement.

2 — Repeaters / Hubs (L1)

Operate purely at the physical layer. Their only job is to regenerate and retransmit signals, extending cable reach.

Characteristics
Acts as signal regenerator — no data inspection
No filtering (does not understand IP or MAC)
All ports in the same collision domain
All ports in the same broadcast domain
Hub = multi-port repeater
Implications
More devices connected → more collisions → more retransmissions → bandwidth crunch
Can be arranged in a hierarchy of hubs to extend network reach
Cheap, simple — use only where L1 reach extension is all that's needed
3 — Bridges (L2)

A bridge operates at the data link layer, connecting LAN segments using MAC address intelligence. A two-port bridge connects two LANs; a multi-port bridge is what we call a Layer 2 switch.

Separate collision domains Same broadcast domain MAC-based filtering Self-learning bridge table

Bridge Table (Self-Learning)

A bridge begins with an empty table. As frames are transmitted, the bridge reads the source MAC address of each incoming frame and records which port it arrived on. This builds a MAC → Port mapping over time.

Learning process: A sends to D → bridge learns A is on port 1. E sends to A → bridge learns E is on port 3. Once the table is populated, the bridge forwards unicast frames only to the correct port, not all ports — reducing unnecessary traffic and cleanly separating collision domains.

Connected segments remain in the same broadcast domain — a broadcast (FF:FF:FF:FF:FF:FF) is still forwarded out all ports.

4 — Loop Problem in Bridged Networks

For redundancy, network administrators often install multiple connections between LANs so that if one link fails, another takes over. However, with Layer 2 bridges, this creates a serious problem: bridge loops.

Why loops are catastrophic: Ethernet frames have no TTL field (unlike IP packets). A frame caught in a loop circulates forever. The bridge keeps updating its table with contradictory port-MAC mappings, consuming all available bandwidth and making communication between any source and destination impossible.

Scenario: A sends to D. Bridge 1 and Bridge 2 both see the frame and forward it. Each bridge then learns A from the opposite port (via the other bridge), leading to contradictory table entries. The frame circulates in a loop indefinitely between the two redundant paths.

5 — Spanning Tree Protocol (STP)

STP solves the loop problem by logically reducing a redundant mesh of bridges and LANs into a loop-free tree topology. Redundant links still exist physically — they are just placed in a blocking state and activated only on failure.

STP Steps

Step 1 → Every bridge has a unique ID. Select the bridge with the smallest ID as the Root Bridge.
Step 2 → On every non-root bridge, mark one port as the Root Port — the port with the least-cost path to the root bridge. Marked with ★.
Step 3 → For each LAN, elect a Designated Bridge — the bridge with the least-cost path between that LAN and the root bridge. The port connecting the LAN to its designated bridge is the Designated Port. Marked with ★★.
Step 4 → Ports with ★ or ★★ become Forwarding Ports. All others become Blocking Ports.
Result → There is exactly one path between any two bridges. Loops are eliminated. Redundant links remain physically present but logically blocked.
Failover: If the active (forwarding) path fails, the blocked redundant path is promoted to forwarding state, restoring connectivity. This is how STP enables redundant connections without loops.

Least-cost path criteria is administrator-defined — it may be minimum hop count, available bandwidth, link congestion, or other metrics.

6 — Backbone Networks

Two common architectures for connecting multiple LANs at the backbone level:

Star Backbone
A central high-capacity multiport switch (typically 8–48+ ports) acts as the backbone hub. All LANs connect to it with star topology. Simple, high performance, easy to manage.
Bridge / Point-to-Point Backbone
LANs connected via point-to-point links between bridges. More flexible for geographically spread networks. Multiple LANs can be daisy-chained through bridges.
7 — Why VLANs?

A switched Layer 2 network, even with separate collision domains, still sits in a single broadcast domain. Every device receives every broadcast — inefficient, insecure, and unmanageable at scale.

Real-world problem: An organisation has accounting, engineering, and sales departments spread across multiple floors and buildings, all connected to the same physical switches. They need to communicate as if they're on the same network — but without broadcasting to each other or sharing security boundaries.

VLAN definition: A Virtual LAN is a broadcast domain created logically by one or more switches. It groups ports (and the devices on them) into isolated broadcast domains, regardless of physical location or switch.

Traditional segmentation required physical routers — one per subnet boundary. VLANs replicate that segmentation logically within the switch fabric itself, at Layer 2, without needing separate physical hardware for each group.

8 — VLAN Basics
Broadcast domain segmentation at L2 Location-independent logical grouping Scalability & security Inter-VLAN traffic needs L3 routing

Key properties of a VLAN:

All ports in the same VLAN share the same broadcast domain. Broadcasts sent by one VLAN member reach only other members of that VLAN — not devices on other VLANs. Switches will not bridge traffic between VLANs; doing so would violate VLAN broadcast domain integrity. Traffic between VLANs must be routed via a Layer 3 device (router or L3 switch).

Without VLANs
All devices in one broadcast domain
ARP request floods to every host on the switch
Unnecessary bandwidth consumption
Security boundaries require physical routers per group
Moving a workstation = physical recabling
With VLANs
Each VLAN = separate broadcast domain
ARP request reaches only VLAN members
Bandwidth conserved; host CPU load reduced
Security policies applied per-VLAN
Moving a workstation = update port assignment in software
9 — VLAN Operation

Static (Port-Based) VLANs

The network administrator manually assigns each switch port to a VLAN. When a device connects to that port, it automatically becomes a member of the assigned VLAN. This is the dominant method in practice — simple, predictable, easy to manage.

Management VLAN: The default VLAN for every switch port is VLAN 1. VLAN 1 is the management VLAN and cannot be deleted. All other ports may be reassigned to alternate VLANs.

Dynamic VLANs

VLAN membership is determined by the MAC address of the connecting device, not the port. Requires dedicated network management software and a MAC-to-VLAN database. When a device connects, the switch queries this database to determine VLAN membership. More flexible but significantly more complex to administer.

Assigning a Host to a VLAN — Two-Step Process

Step 1 → Connect the host to the correct physical port of the switch (the port assigned to the target VLAN).
Step 2 → Assign the host an IP address in the correct subnet for that VLAN. VLAN membership effectively implies subnet membership — VLAN ≡ Subnet.

Both steps are necessary. Physical port alone is not sufficient — the host's IP must also belong to the VLAN's subnet for correct routing.

10 — VLAN Types
Type Basis Status Notes
Port-Based Switch port Dominant Simple, widely used. DHCP handles IP assignment. Ports assigned individually, in rows, or across 2+ switches.
MAC-Based Device MAC address Rare Each MAC manually entered and configured. Flexible but difficult to administer and troubleshoot at scale.
Protocol-Based IP address / protocol Rare Similar to MAC-based but uses logical L3 address. No longer common due to DHCP.
11 — VLAN Tagging & Trunking

When a single physical link between two switches must carry traffic for multiple VLANs simultaneously, plain forwarding is not enough — the receiving switch cannot tell which VLAN a given frame belongs to. This is solved by VLAN tagging.

How Tagging Works

1. Tag → When the source switch receives a frame from an end device, it inserts a VLAN ID tag into the frame header, identifying its VLAN membership.
2. Transit → The tagged frame travels over the trunk link. Intermediate switches forward it to the appropriate ports based on the VLAN ID and MAC address.
3. Strip → Upon reaching the destination switch, the VLAN tag is removed and the original frame is delivered to the attached end device.
Trunk link (also called VLAN trunking): a single physical link configured to carry traffic for multiple VLANs simultaneously. Without trunking, each VLAN would require its own dedicated physical link between switches — wasteful and unscalable.

VLAN tagging controls broadcast propagation — broadcasts are only sent to ports belonging to the same VLAN, preventing them from leaking into other broadcast domains over the trunk.

12 — VLAN Deployment Models
End-to-End / Campus-Wide VLANs
VLANs span the entire campus network regardless of physical location. Based on function or workgroup. "VLAN everywhere" model — Accounting VLAN 10 exists across all switches in the organisation. Users retain the same VLAN membership even when they move to a different physical location.
Geographic / Local VLANs
VLANs are scoped to a physical location (floor, building, switch cluster). Dedicated to an access-layer switch cluster. Accounting users on different floors/buildings may be in different VLANs (e.g., VLAN 10 vs VLAN 30). Simpler per-location management; less flexible for user mobility.
Critical point: Devices on different VLANs — even on the same physical switch — are in separate broadcast and collision domains and cannot communicate directly. Communication between VLANs always requires a Layer 3 router or L3 switch. A plain L2 switch cannot bridge inter-VLAN traffic without violating VLAN isolation.
Summary

Concept Snapshots

Hub / Repeater
L1 only. Signal regeneration. No filtering. Single collision + broadcast domain. Adding more = more collisions = bandwidth degraded.
Bridge / L2 Switch
L2. Self-learning MAC table. Separate collision domains per port. Still one broadcast domain. Loop-prone when redundant links exist — requires STP.
STP
Elect root bridge (smallest ID). Assign root ports + designated ports. Block all redundant paths. One path active at a time — blocked paths activate on failure.
VLAN
Logical broadcast domain created by switch config. Segments L2 network without physical routers. Each VLAN = its own subnet. Devices must have matching IP subnet to belong to the VLAN.
VLAN Tagging / Trunking
Trunk link carries multiple VLANs on one cable. VLAN ID inserted at source, stripped at destination. Enables VLAN extension across multiple switches efficiently.
Inter-VLAN Routing
VLANs are isolated — traffic cannot cross between them at L2. A Layer 3 device (router or L3 switch) is always required to route packets between different VLANs.
