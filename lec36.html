IP Routing — Introduction & Routing Table

Network layer routing fundamentals: IPv4 addressing recap, hierarchical prefix-based forwarding, longest prefix match, forwarding table construction, and ARP overview.

▶ Watch Lecture
IPv4 Address — Quick Recap

IPv4 addresses are 32-bit logical identifiers written in dotted-quad notation (four 8-bit octets). They uniquely identify an interface on a host or router — not the device itself. Every system on the internet must have a unique IP address; duplicates break routing entirely.

Example: 14.35.158.15 → binary: 00001110.00100011.10011110.00001111. The IP layer operates at Layer 3; routers understand up to this layer to make forwarding decisions.

IP addresses are assigned either statically by a sysadmin or dynamically via DHCP. MAC addresses (physical/hardware addresses burned into the NIC) are separate — they live at Layer 2 and are used for local delivery.

Internetworking

The internet is a network of networks. Individual LANs (e.g., CSE dept, Admin block at IIT KGP) are connected via routers into a Wide Area Network. Routers are Layer 3 devices — they read the IP header, consult a forwarding table, and push packets toward the destination network.

A router has multiple interfaces (each connecting to a different network/router). Each interface has its own IP address. Packets arrive on one interface and are forwarded out another based on the destination address.
The Scalability Problem

If routers stored one entry per host globally, every addition or removal of a machine would require updating routing tables across every router in the world. With billions of devices, this is completely infeasible. The solution: route by network prefix, not by individual host address.

Per-Host Routing (Bad)
Table sizeOne entry per device globally — billions of rows
On changeEvery router worldwide must update
FeasibilityCompletely impractical at internet scale
Per-Network (Prefix) Routing (Good)
Table sizeOne entry per network prefix — manageable
On changeNew host in same network → no table update needed
FeasibilityScales to global internet
Hierarchical Addressing — IP Prefixes & Subnet Masks

An IP address is split into a network portion and a host portion using a subnet mask — a 32-bit value consisting of a contiguous run of 1s followed by 0s. Applying the mask via bitwise AND extracts the network address.

Example — /24 mask:
IP: 12.34.158.5
Mask: 255.255.255.0 (i.e., /24 → first 24 bits = network)
Network address: 12.34.158.0/24 — covers 28 = 256 host addresses.

This is analogous to a postal address: Country → State → District → City → Street → House. Each level narrows scope. Only the local post office needs to know individual house addresses; the national postal system only routes to the city.

Prefix Length Subnet Mask Network Bits Host Bits Hosts per Network
/8255.0.0.0824~16.7 million
/16255.255.0.01616~65,536
/24255.255.255.0248256
/15255.254.0.01517~131,072
Key scalability win: Adding a new host to network 5.6.7.0/24 (e.g., 5.6.7.213) requires zero changes to any router's forwarding table. The table entry 5.6.7.0/24 already covers it.
Classful Addressing

The original IPv4 scheme assigned fixed-size blocks based on the leading bits of the address. The mask is implicit — determined solely from the address itself.

Class Leading Bits Prefix Example Use
A0*/820.0.0.0/8Very large orgs (~16M hosts)
B10*/16128.40.0.0/16Large orgs (~65K hosts)
C110*/24192.60.225.0/24Small orgs (256 hosts)
D1110*——Multicast groups
E11110*——Reserved for future use

Classful addressing is too coarse — a Class B block wastes tens of thousands of addresses for an org needing only a few hundred. This led to CIDR.

CIDR — Classless Inter-Domain Routing

CIDR decouples the prefix length from the address value entirely. A network is represented as IP address + explicit mask of any length.

Format: Network_Address / prefix_length
Example: 12.4.0.0/15 — first 15 bits are the network prefix, remaining 17 bits for hosts.

CIDR: Hierarchical Address Allocation

CIDR enables address aggregation: a provider assigned 201.10.0.0/21 can sub-allocate to customers (201.10.0.0/22, 201.10.4.0/24, etc.). Routers outside that provider's domain only need one routing entry — 201.10.0.0/21 — and the provider internally handles the rest.

Aggregation challenge — multi-homing: If a customer (201.10.6.0/23) is connected to two different ISPs, the simple aggregation breaks — the internet must know how to reach that customer through both providers separately. This is a known hard problem.

CIDR Complicates Forwarding

With variable-length masks, a single destination IP may match multiple forwarding table entries simultaneously. For example, 201.10.6.17 matches both 201.10.0.0/21 and 201.10.6.0/23. The router must resolve this ambiguity — which leads to the Longest Prefix Match rule.

IP Address Allocation Hierarchy

IP address blocks are allocated top-down through a strict hierarchy:

ICANN
Allocates large blocks to RIRs globally
RIRs (e.g., ARIN for Americas)
Allocates blocks to ISPs & large institutions
ISPs
Allocates sub-blocks to their customers
Institutions (e.g., IIT KGP)

Address registries are public records. Ownership can be queried via whois (e.g., whois -h whois.arin.net <address>), though records are notoriously out of date.

Are 32-Bit Addresses Enough?

232 = ~4.29 billion addresses. With reserved blocks, multi-device households, IoT, and server farms, this is genuinely insufficient. Short-term and long-term solutions coexist:

Long-Term
IPv6

128-bit addresses → 2128 ≈ 3.4 × 1038 addresses. Effectively unlimited.

Short-Term (IPv4)
Private Addressing + NAT

Non-routable private ranges (e.g., 10.x.x.x) used internally; NAT translates to a public IP at the gateway.

Short-Term (IPv4)
DHCP

Dynamic allocation — IPs reused as devices connect/disconnect rather than permanently assigned.

Hop-by-Hop Packet Forwarding

IP routing is hop-by-hop and destination-based. Each router independently makes a local forwarding decision based solely on the packet's destination IP — no router has end-to-end knowledge of the full path.

Router forwarding algorithm (per packet):
1. Receive packet on an input interface.
2. Extract destination IP address from the IP header.
3. Perform a lookup in the forwarding/routing table → find the longest matching prefix.
4. Forward the packet out the corresponding outgoing interface.
5. The next router repeats this process until the destination is reached.

If no matching prefix is found, the router uses a default route (typically 0.0.0.0/0) — a catch-all entry that matches everything with a zero-length prefix.

Routing table vs Forwarding table: These terms are used interchangeably in this course. Strictly: the routing table contains richer protocol metadata; the forwarding table is the optimized lookup structure derived from it.
Longest Prefix Match (LPM) Forwarding

When CIDR allows multiple table entries to match a destination, the router selects the entry with the longest (most specific) matching prefix.

Example: Destination = 201.10.6.17. Forwarding table has entries:
201.10.0.0/21 — matches (21-bit prefix)
201.10.6.0/23 — also matches (23-bit prefix)
LPM selects: 201.10.6.0/23 — longer prefix = more specific = correct choice.
Destination IP Forwarding Table Entry (match) Outgoing Interface
201.10.6.17201.10.6.0/23 ✓ (longest)Serial0/0.1
201.10.4.5201.10.4.0/24 ✓ (longest)eth0/1
8.8.8.80.0.0.0/0 (default route)WAN uplink
LPM Algorithm & Efficiency Challenge

A naive linear scan of the forwarding table is O(n) in the number of entries. Modern routing tables have 200,000–300,000+ entries. With packets arriving every few nanoseconds at line rate, linear scan is completely unacceptable.

Patricia Tree (Binary Trie)

A traditional solution: store prefixes in a binary trie where each level corresponds to one bit of the address. Lookup traverses at most 32 levels (one per bit) rather than scanning all entries. However, the tree can be skewed and deep with large tables, limiting efficiency.

Hint: The Patricia tree/trie structure is best understood visually — refer to the lecture diagram at 33:00. The binary branching (0 → left, 1 → right) at each bit position is shown explicitly.

Faster Alternatives

k-ary Tree
Multi-way Trie

Instead of binary, each node has k children (e.g., 4-way trie processes 2 bits per level). Fewer levels, faster lookup — at the cost of more memory per node.

Special Hardware
CAM (Content Addressable Memory)

Lookup by key rather than by address. Ternary CAM (TCAM) supports wildcard matching — essential for LPM. Used in high-speed routers; lookup in O(1) hardware time.

Where Do Forwarding Tables Come From?

Forwarding tables map IP prefixes to outgoing interfaces. They can be populated two ways:

Static Configuration
HowAdmin manually adds entries (e.g., "map 12.34.158.0/24 to Serial0/0.1")
ProsSimple, predictable, zero overhead
ConsNot adaptive — fails to react to topology changes, new devices, or link failures
Routing Protocols (Dynamic)
HowRouters exchange reachability information automatically (covered in subsequent lectures)
ProsAdaptive — handles failures, load balancing, topology changes
ConsProtocol overhead; convergence time after failure
Terminology distinction: A routing protocol (e.g., OSPF, BGP) builds and maintains the routing table. A routed protocol (e.g., IP) is the protocol whose packets are being forwarded. These are different things.
Packet Forwarding by End Hosts

End hosts (PCs, laptops) do not run routing protocols. Their forwarding logic is simple:

Destination Action
Self (e.g., 127.0.0.1 or own IP)Deliver locally (loopback)
Same LAN (e.g., 1.2.3.0/24)Send directly out the interface (resolve MAC via ARP)
External (0.0.0.0/0)Send to the configured default gateway

The gateway address is learned either via static configuration (manual IP setup) or dynamically via DHCP.

Final-Hop Delivery — ARP

IP routing gets a packet to the correct network, but the actual frame delivery on a local link requires a MAC address. Each NIC has a globally unique MAC address burned into its ROM — a flat, non-hierarchical 48-bit identifier with no routing semantics.

To find the MAC address corresponding to a known IP address, the network layer uses ARP (Address Resolution Protocol):

ARP role: Maps IP address (Layer 3) → MAC address (Layer 2). The last router on the path broadcasts an ARP request on the local link; the target host responds with its MAC. The router then encapsulates the IP packet in a frame addressed to that MAC and delivers it.
RARP (Reverse ARP) goes the other direction: MAC → IP. Used historically by diskless workstations to learn their own IP address. Largely superseded by DHCP.
Scalability Through Hierarchy — Key Principles
Principle 1
No global knowledge needed

Each router only needs to know how to reach the next hop, not the full path. Hierarchy ensures each level only needs knowledge of its own sub-tree.

Principle 2
Updates are localized

Changes within an institution (new host, subnet restructure) only require updates at that institution's level — not propagated globally.

Principle 3
CIDR over classful

Classful addressing was too coarse (forced large wasteful blocks). CIDR's variable-length prefixes allow right-sized allocations but introduce LPM complexity.

Principle 4
Aggregation reduces table size

Multiple sub-networks summarized into one prefix at higher levels. External internet only sees the aggregate; internal routing handles details.

Concept Summary
IPv4: 32-bit, dotted-quad, identifies an interface. Assigned by sysadmin or DHCP.
Subnet mask: 32-bit value (contiguous 1s then 0s). Bitwise AND with IP extracts network address. Written as /prefix_length (CIDR notation).
Classful addressing: Fixed mask inferred from leading bits (A=/8, B=/16, C=/24). Too coarse → replaced by CIDR.
CIDR: Variable-length prefix. Enables efficient allocation and aggregation but introduces multiple-match ambiguity in forwarding tables.
Hop-by-hop forwarding: Each router independently consults its forwarding table using the destination IP. No router knows the full path end-to-end.
Longest Prefix Match (LPM): When multiple table entries match, the most specific (longest prefix) wins. This is the fundamental forwarding rule under CIDR.
LPM efficiency: Linear scan is too slow (200K+ entries, nanosecond packet rates). Solutions: Patricia trie (O(32)), k-ary trie, TCAM hardware (O(1)).
Forwarding table sources: Static config (simple but not adaptive) or routing protocols (dynamic, adaptive — covered in subsequent lectures).
ARP: Resolves IP address → MAC address for final local-link delivery. Required because IP routing operates at Layer 3 but frames are delivered at Layer 2.
Address hierarchy (ICANN → RIR → ISP → Institution → Host): Prefix assigned to institution; institution assigns host addresses internally. Only the institution's border needs updating when internal topology changes.
