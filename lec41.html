IP Routers

Internal architecture of an IP router — control plane vs data plane, TCAM, FIB, RIB, switch fabric, and the evolution of router hardware across five generations.

The Life of a Router

A router's entire existence can be summarised as a do-repeat loop: find a path to a destination, forward packets to it, find the next path, forward more packets — repeat until powered off.

This gives us the two fundamental operations every router performs:

Control Plane
Constructs the routing table by running routing protocols (RIP, OSPF, BGP). Executed periodically or on topology change — relatively low frequency.
Data Plane (Forwarding Plane)
Performs a routing table lookup per arriving packet and forwards it to the correct output interface. Executed per packet — extremely high frequency.
Why this split matters for hardware design: At 100 Gbps with average packet size 1 MB, a router must process ~12,500 packets per second per interface. The forwarding operation must happen in hardware — software would never keep up. Control plane operations (routing protocol exchanges) happen orders of magnitude less often, so software is fine there.
Evolution of Router Architecture
Generation Era Architecture
1stUntil 1980sStandard general-purpose computer with multiple network interfaces
2ndEarly 1990sDedicated device with delegated interfaces — purpose-built hardware
3rdLate 1990sDistributed router architecture — processing distributed across components
4thEarly 2000sDistributed over multiple racks — multiple routers interconnected to scale interface count (e.g. 4 × 32-port routers = 128 interfaces)
5thTodaySoftware Defined Networking (SDN) routers — control plane decoupled and centralised; widely used in large-scale data centres
Basic Architectural Components

At the broadest level, a router consists of:

Processor (Route Processor)
General-purpose CPU + memory. Runs the router OS and routing protocols. Responsible for the control plane — builds and maintains the routing table.
Interconnection Network
Internal bus/fabric connecting the processor to all interface cards. Routes packets internally from the input interface to the correct output interface.
Interface Cards (NICs)
Network I/O interfaces — RJ45 ports for wired, transmitter/receiver for wireless. Each card has its own microcontroller that handles the data plane (forwarding) for packets arriving on that interface.

Router Internals: The Full Picture

Interface Processor → Packet Processor (forwarding engine, implemented in TCAM hardware) → Data Plane

Route Processor (routing engine, general-purpose CPU running routing algorithms) → Control Plane

The packet processor uses TCAM to do hardware-speed lookups against the routing table. The route processor runs software (the router OS) to keep that table accurate.

Physical Infrastructure — Slotted Chassis

Large production routers are built as a slotted chassis: a physical enclosure with multiple card slots. Each interface card slots in individually, as does the route processor card. This modular design means individual components can be repaired or upgraded without replacing the entire unit — hot-swappable in many enterprise models.

Functional Component Breakdown

Control Plane — Routing Functions

The control plane is a special-purpose software environment (the Router OS) running on the route processor. Three functions it must perform:

Route Calculation Routing Table Maintenance Execution of Routing Protocol

On commercial routers, all three are handled by a single general-purpose CPU — the route processor. Example: Cisco IOS (Internetwork Operating System) — a router OS that implements multiple routing protocols (RIP, OSPF, BGP) as part of its software stack.

Data Plane — IP Forwarding

The data plane implements per-packet forwarding. Its logic mirrors a Layer 2 switch's switch fabric — except decisions are based on IP destination addresses, not MAC addresses.

Switch Fabric: a hardware interconnect that maps any input port to any output port. In a router, all interfaces are simultaneously potential input and output ports. The fabric logic (a set of logic gates) copies packet data from the input port to the determined output port.

Each interface also maintains an interface buffer to implement store-and-forward: packets queue up, are examined, and are transmitted one by one over the data link layer.

IP Forwarding is distributed: each individual interface controller (a microcontroller on the NIC) handles forwarding for packets arriving on that specific interface — not a central CPU. This is why TCAM hardware lives on the NIC.
TCAM — Ternary Content Addressable Memory

Ternary Content Addressable Memory (TCAM) is the specialised hardware that makes per-packet forwarding fast enough to sustain line-rate processing.

A normal memory lookup: give an address, get the value. TCAM flips this: give the content (the destination IP), get the matching table entry — in a single clock cycle, regardless of table size. "Ternary" means each bit can be 0, 1, or wildcard (don't care), which is exactly what you need for longest-prefix matching in IP routing tables.

TCAM lives on the interface card and is used to implement the Forwarding Information Base (FIB) — the hardware-resident copy of the routing table.

FIB vs RIB
RIB — Routing Information Base
Lives inSoftware (control plane, route processor memory)
ContainsComplete routing information from all protocols — BGP table, OSPF link-state database, static routes
Updated byRouting protocols, continuously and dynamically
ScopeEntire routing topology
FIB — Forwarding Information Base
Lives inTCAM hardware on each interface card (data plane)
ContainsOnly the routes relevant to that interface — a subset of the RIB, copied on demand
Updated byRIB, only when a FIB miss occurs
ScopeLocal forwarding decisions for one interface

FIB Lookup Flow — Per Packet

1. PACKET_IN→Arrives at input interface
2. FIB Lookup→TCAM checks local FIB for destination IP match
FIB HIT→Forward packet directly to output interface → PACKET_OUT
FIB MISS→Escalate to Route Lookup → query RIB via routing protocol → update FIB → then forward
RIB feeds FIB: Multiple routing protocol tables (BGP, OSPF link-state DB, manually configured static routes) all feed into the RIB. The RIB selects the best routes and pushes relevant entries into the TCAM FIB at each interface as needed.
Per-Packet Processing — Full Architecture

Putting it all together, the sequence for every packet:

Extract destination IP from IP header
TCAM lookup against FIB HIT or MISS
Routing decision — determine output interface
Enqueue in input buffer
Switch fabric copies to output queue → transmit

The routing table (FIB copy) is distributed — a relevant subset lives at each individual interface controller. Each interface makes its own forwarding decision independently, which is what makes line-rate processing achievable.

Concept Summary

Control Plane vs Data Plane — the core design insight

The control plane (routing protocol execution, RIB construction) runs infrequently in software on a general-purpose processor. The data plane (per-packet lookup and forwarding) runs at line rate in hardware (TCAM + switch fabric) on each interface card. These two planes are implemented on completely different hardware because their performance requirements differ by orders of magnitude.

TCAM enables hardware-speed longest-prefix matching

Standard RAM can't do this fast enough. TCAM stores routing table entries and matches an incoming destination IP against all of them simultaneously in one cycle, using wildcard bits to handle prefix masks.

FIB is not the same as the routing table (RIB)

The RIB is the complete, software-maintained routing table in the control plane. The FIB is a per-interface hardware cache of only the routes that interface currently needs. FIB misses trigger a RIB lookup, which updates the FIB. This caching design keeps TCAM usage efficient while keeping forwarding decisions fast.

Switch fabric = the internal crossbar

Once the forwarding decision is made (which output interface?), the switch fabric physically moves the packet data from the input port's buffer to the correct output port's queue. In a router, every interface is both a potential input and output port.
