Lecture 51 HIGH PRIORITY

ARP, RARP & Proxy ARP

Layer 2 address resolution — mapping IP addresses to physical (MAC) addresses, the reverse, and the proxy variant. Packet format, encapsulation, cache mechanics, and four canonical use cases.

▶ Watch Lecture
Motivation

Why IP Alone Is Not Sufficient

Communication between two devices on a network requires physical (hardware) connectivity and knowledge of the destination's physical address. An IP address identifies a device logically at Layer 3, but the data link layer (Layer 2) must deliver frames using MAC addresses. A source that knows only the destination IP address cannot construct a valid Layer 2 frame without first resolving that IP address to a corresponding MAC address.

This gap is bridged by the Address Resolution Protocol (ARP) and its complement, Reverse ARP (RARP).

ARP
Direction: IP → MAC (logical → physical)
Use case: Sending a frame to a known IP address when the MAC is unknown.
RARP
Direction: MAC → IP (physical → logical)
Use case: Diskless/dumb terminals that know their own MAC but need to be assigned an IP at boot time.
Protocol Positioning

Where ARP/RARP Sit in the TCP/IP Stack

ARP and RARP are often described as companion protocols to IP. They operate at the boundary between the Network Layer (Layer 3) and the Data Link Layer (Layer 2): they carry IP addresses (a Layer 3 concept) on one side and physical addresses (a Layer 2 concept) on the other. Some literature places them as an interface layer between IP and the MAC sublayer.

Core Mechanism

How ARP Works — Request and Reply

When Host A wishes to send a frame to Host B and knows only B's IP address, ARP proceeds as follows:

Step 1 — ARP Request (Broadcast): Host A broadcasts an ARP request onto the local network segment: "Who has IP X.X.X.X? Tell me your MAC address." The destination MAC in the enclosing Ethernet frame is the broadcast address FF:FF:FF:FF:FF:FF.
Step 2 — ARP Reply (Unicast): Only the host whose IP matches the query responds. It sends a unicast ARP reply back to Host A containing its MAC address. All other hosts silently discard the broadcast.
Step 3 — Frame Transmission: Host A now has both the destination IP and the resolved MAC address. It constructs the Layer 2 frame and transmits it normally.

Key distinction: ARP request = broadcast; ARP reply = unicast.

Encapsulation

ARP Inside an Ethernet Frame

The ARP packet is encapsulated as the payload of a standard Ethernet frame. The Ethernet Type field is set to 0x0806 to indicate that the payload contains an ARP message. (Recall: 0x0800 = IP, 0x0806 = ARP.)

The ARP request uses destination MAC FF:FF:FF:FF:FF:FF (broadcast). The ARP reply uses the actual unicast MAC of the requester as the destination.

Four Cases — HIGH PRIORITY

Four Canonical ARP Scenarios

ARP is used in four distinct forwarding contexts, covering all combinations of source and destination device types:

Case Sender → Receiver Target IP in ARP Request Notes
1 Host → Host (same network) Destination IP in the datagram Direct delivery; ARP resolves destination MAC on the LAN.
2 Host → Router (different network) IP address of the default gateway router Host knows destination is off-LAN; ARP resolves the router's MAC so the frame reaches the gateway.
3 Router → Router (inter-network) IP of the next-hop router from the routing table Intermediate hop resolution; ARP runs on each outgoing interface.
4 Router → Host (final delivery) Destination IP in the datagram Last hop; ARP resolves the final destination host's MAC on the local segment.
Router interfaces: A router with N interfaces has N IP addresses and N MAC addresses (one per NIC). ARP resolution on a router is always performed on the specific interface through which the next hop is reachable.
Proxy ARP

Proxy ARP (a.k.a. Promiscuous ARP / ARP Hack)

A Proxy ARP router responds to ARP requests on behalf of hosts that are not on the same physical segment as the requester. When an ARP request arrives for an IP address belonging to a host behind the router, the router replies with its own MAC address. The sender then addresses frames to the router's MAC; the router forwards them onward to the actual destination host.

Key mechanism: Hosts behind the proxy ARP router are invisible to the requester. As far as the requester is concerned, all those hosts appear to be on the same physical network, reachable via the router's MAC.

Technical Definition

Proxy ARP maps a single IP network prefix to one or more physical addresses, effectively allowing the same network address space to be used across more than one physical segment. A router R running Proxy ARP between Networks A and B can use the same network ID for both networks, responding to ARP requests from either side on behalf of the other.

How It Works (PPP Example)

Suppose Network A (10.10.1.x/24) is an Ethernet LAN, and Network B (also 10.10.1.x/24) is a PPP-connected segment. Router R runs Proxy ARP. When a host on Network A sends an ARP request for 10.10.1.5 (which is on Network B via PPP), Router R replies with its own MAC address. The host on Network A sends the frame to R's MAC; R then delivers it to 10.10.1.5 over the PPP link. In the ARP tables of hosts on Network A, R's MAC is recorded as the hardware address for all destinations in Network B.

Advantages of Proxy ARP

The primary advantage is simplicity: hosts on both sides need no special configuration to communicate across the boundary. Adding or removing hosts within a segment does not require updating ARP tables on remote segments, as long as all changes remain within the managed network prefix. Management is centralised at the router. Proxy ARP also naturally restricts broadcast propagation, since the router absorbs broadcast ARP traffic rather than forwarding it onto remote segments.

Implementation

ARP Software Modules

An ARP implementation typically consists of five cooperating software components:

Cache Table
Stores recent IP-to-MAC mappings. Any incoming ARP request is checked against the cache first; a cache hit avoids a network broadcast entirely.
Queues
Packets awaiting resolution of their destination MAC are held in per-destination queues. Once the ARP reply arrives, queued packets are dequeued and forwarded.
Output Module
Receives IP datagrams from the IP layer. Looks up the destination IP in the cache; if resolved, passes the frame to the data link layer; if pending, enqueues the packet; if absent, creates a new cache entry and broadcasts an ARP request.
Input Module
Receives incoming ARP packets (requests and replies). Updates the cache with newly learned mappings and triggers delivery of any waiting packets for that destination.
Cache-Control Module
Performs periodic housekeeping. Decrements timeout counters for all cache entries. Transitions timed-out resolved entries to FREE (flushed). Increments the attempt counter for pending entries; entries exceeding the maximum attempt count are also freed.
Cache Table — HIGH PRIORITY

ARP Cache Table Structure and States

Each entry in the ARP cache table carries the following fields:

Field Meaning
StateR = Resolved (MAC known), P = Pending (ARP request sent, awaiting reply), F = Free (entry flushed/available)
QueueIndex of the queue holding packets awaiting this resolution
AttemptsNumber of ARP requests sent for this entry; incremented by cache-control module each cycle
Time-OutCountdown timer; when it reaches 0, a Resolved entry transitions to Free
Protocol AddressDestination IP address (the key)
Hardware AddressResolved MAC address (empty while Pending)

Output Module Decision Logic

Entry found, state = RESOLVED: Extract the hardware address, send the frame to the data link layer immediately. Cache unchanged.
Entry found, state = PENDING: Enqueue the packet. Wait until the ARP reply arrives and the state transitions to RESOLVED.
Entry not found: Create a new entry with state = PENDING and Attempts = 1. Create a queue for this destination. Enqueue the packet. Broadcast an ARP request.

Cache-Control Module Housekeeping (Periodic)

On each periodic tick, the cache-control module decrements timeout values for all RESOLVED entries. Any RESOLVED entry whose timeout reaches zero transitions to FREE (its slot is reclaimed). For each PENDING entry, the Attempts counter is incremented; entries that exceed the maximum retry count transition to FREE. This ensures stale or unresolvable entries do not occupy cache space indefinitely.

RARP

Reverse ARP (RARP)

RARP performs the inverse mapping: given a known physical (MAC) address, obtain the corresponding IP address. This is required by diskless or dumb terminals that possess a NIC (and thus a MAC address) but have no stored IP configuration. At boot time, such a device broadcasts a RARP request; a designated RARP server on the network responds with the appropriate IP address. RARP was later superseded by BOOTP and then DHCP, which provide richer configuration information.

Summary

Concept Summary

ARP in One Line
Broadcast request (IP → MAC), unicast reply. Result cached to avoid repeated broadcasts. ARP request = operation 1, reply = operation 2.
Proxy ARP in One Line
Router intercepts ARP requests for off-segment hosts and replies with its own MAC, acting as an intermediary. Simplifies host configuration; centralises management.
ARP Cache in One Line
Lookup table of IP → MAC mappings with state (R/P/F), timeout, and retry tracking. Avoids network broadcasts for recently seen destinations.
