Application Layer — Part I

▶ watch lecture
Lecture Overview

This is the first lecture dedicated to a specific protocol layer. It covers the client-server interaction model, well-known port number assignments, the responsibilities of the Application layer, and an overview of the principal application layer protocols: DNS, FTP/TFTP, HTTP, SMTP, SNMP, and Telnet. The lecture concludes with a detailed examination of how sockets function in practice, including the five-tuple that uniquely identifies every network connection.

The Client-Server Model ↗ ~03:00
Every application layer interaction follows a client-server model. One process (the client) initiates a request for a service; another process (the server) responds by providing it. The two processes do not need to reside on different machines — they may run on the same host — but in network contexts they are typically on separate systems. Every application layer protocol has a defined client variant and a server variant that implement their respective roles.

For example: a web browser is an HTTP client; the machine hosting iitkgp.ac.in is an HTTP server. An FTP client connects to an FTP server; a Telnet client connects to a Telnet server. This asymmetry — one party initiating, one party serving — is the defining characteristic of the model.

How a Server Process Works ↗ ~26:01

A server runs a daemon process — a background process that is always active and listening on a specific, well-known port. When an incoming client request arrives, the daemon reads the client's IP address and port number, then either handles the request directly or forks a child process to service it. Forking allows the parent daemon to immediately return to listening for new requests while the child handles the current one — enabling concurrent servicing of multiple clients.

Example: an HTTP server daemon (httpd) is permanently listening on port 80. When a browser sends a request, httpd forks a child process to serve the response. The parent process continues listening and can accept additional client requests simultaneously — this is how a single server can serve thousands of concurrent users.

Sockets — The Full Picture ↗ ~24:22

A connection between a client and server is uniquely identified by five pieces of information — the 5-tuple:

client IP + client port + server IP + server port + protocol

This 5-tuple uniquely identifies every active connection in the network. A socket is the endpoint — the (IP address + port number) pair on one side of the connection. When a client connects to a server, a socket is created on each side, and the connection is represented as the pair of sockets. Because the client's port is dynamically assigned and differs for every connection, multiple clients can simultaneously connect to the same server port without conflict.

Multiple browser tabs connecting to the same server (e.g., port 80) at the same time are each a distinct connection, because each has a different client port in its 5-tuple. The server can distinguish and track all of them simultaneously.

The Application layer is split into user space (application code — a browser, an FTP client, a custom program) and kernel space (the TCP/UDP and IP implementations, managed by the OS). The socket API is the standardized interface that bridges these two spaces: application code calls socket functions, and the OS kernel handles the actual protocol execution.

When configuring TCP/IP settings on a host (IP address, subnet mask, default gateway, DNS server address), the configuration targets the kernel-space layers. The NIC and its firmware handle everything below that boundary automatically, without user intervention.

Application Layer Responsibilities ↗ ~14:35

The Application layer is responsible for the full set of concerns above reliable data delivery: identifying and verifying the availability of a communication partner (is the intended server reachable and ready?), synchronizing cooperating applications (orchestrating the correct request-response sequence), establishing error recovery procedures at the application level, and ensuring data integrity from the application's perspective. In short, the Application layer handles everything that remains after the lower layers have delivered the data.

Application Layer Protocols — Overview ↗ ~16:08
DNS
Domain Name System · UDP port 53
Translates human-readable domain names to IP addresses. A DNS client on the host queries a DNS server; the server responds with the corresponding IP. All routing from that point forward operates on the IP address — intermediate routers have no knowledge of domain names. DHCP typically auto-configures the DNS server address when a host joins a network.
FTP vs. TFTP
File Transfer · TCP 21 vs. UDP 69
FTP (File Transfer Protocol) is connection-oriented, uses TCP for reliable transfer, and requires authentication credentials. TFTP (Trivial FTP) is connectionless, uses UDP, and is intentionally minimal — it is used in contexts such as pushing firmware images to routers, where simplicity and small code size matter more than guaranteed delivery.
HTTP
Hypertext Transfer Protocol · TCP 80
The dominant application-layer protocol on the internet. A browser sends an HTTP request; the web server sends an HTTP response containing the requested resource. HTTP is so ubiquitous that most firewalls and routers permit it by default, which is why web services and APIs are commonly built on top of it even for non-web use cases.
SMTP
Simple Mail Transfer Protocol · TCP 25
Handles the transmission of email between mail servers and from email clients to outgoing mail servers. SMTP is specifically for sending and relaying mail — POP3 and IMAP are separate protocols used by clients to retrieve mail from their mailbox on the server.
SNMP
Simple Network Management Protocol · UDP 161
An application layer protocol used for network device monitoring and management. SNMP agents run on managed devices (routers, switches, servers) and report status information to a central Network Management System (NMS). It is not directly used by end users, but it is the mechanism that keeps large networks operational and observable.
Telnet
Remote Login · TCP 23
Provides a command-line interface to a remote machine over a network — a Telnet client establishes a TCP connection to a Telnet server, which grants access to a shell or terminal. Telnet transmits all data, including credentials, in plaintext and has been largely superseded by SSH, which provides the same functionality over an encrypted channel.
DNS Domain Structure ↗ ~17:18

Domain names follow a hierarchical structure, read from right to left — the rightmost component is the most general (the Top Level Domain), and each component to the left is a subdivision of the one to its right:

TLD = Top Level Domain. Over 200 TLDs exist: .in (India), .us, .uk, .edu (educational institutions), .com (commercial entities), .org (non-profit organizations), .gov (government), .net (network service providers). The DNS resolution hierarchy mirrors this structure — root name servers → TLD servers → authoritative name servers for each registered domain.

TCP vs. UDP — Which Protocols Use Which ↗ ~13:33
Use TCP when: data integrity and ordered delivery are non-negotiable — missing or corrupted data would render the result incorrect. HTTP, FTP, SMTP, Telnet, and most file transfer protocols fall into this category.

Use UDP when: low latency or high throughput is the priority, or when the application itself handles retransmission and recovery. DNS uses UDP because queries are small and simply retransmitting on timeout is sufficient. SNMP, TFTP, and real-time media streaming (where a retransmitted packet would arrive too late to be useful) also use UDP.

In rare cases, applications can bypass the Transport layer entirely and communicate directly with IP — but this requires the application to implement its own reliability mechanisms and is uncommon in practice.
