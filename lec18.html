Lecture 18 medium priority

Transport Layer Primitives

A synthesis lecture tying together all transport layer services — multiplexing, connection management, flow control, reliability, and congestion control — into a coherent end-to-end protocol model. Sets up the TCP deep-dive that follows.

1 — The Problem

Multiplexing Applications Across Machines

A single physical machine runs multiple applications simultaneously (multitasking). Over the network, any application on any machine must be able to communicate with any specific application on any other machine. Two distinct identification problems arise:

Which machine?
Solved at the network layer using the IP address (32-bit). Every host on the internet has a unique IP address that routers use to forward packets.
Which application on that machine?
Solved at the transport layer using the port number (16-bit). Each running application is bound to a port; the OS uses it to demultiplex incoming segments to the right process.

The transport layer header therefore always carries a source port and a destination port. This is how a segment arriving at a host is routed to the correct application process — not just the correct host.

2 — Logical Pipe

The End-to-End Logical Pipe (Socket)

The transport layer defines a logical pipe between two communicating transport entities. Physically, packets travel through the network layer (unreliable); logically, the transport layer presents a reliable, ordered channel to the application above.

This logical pipe must be uniquely identified. IP address + port alone is not sufficient — if a machine crashes and restarts, a new connection on the same {src IP, src port, dst IP, dst port} quadruple must be distinguishable from any delayed duplicates of the old connection. Therefore the transport layer uses a 6-tuple:

Src IP
+
Src Port
+
Src ISN
+
Dst IP
+
Dst Port
+

=
Socket

The inclusion of Initial Sequence Numbers (ISN) on both sides ensures that even if the same {IP, port} pair is reused after a crash, any delayed duplicates from the prior connection (which carry the old ISN) cannot be mistakenly accepted. In Unix terminology, this logical pipe is called a socket.

Key rule: When selecting an ISN, it must not fall within the forbidden region — the range of sequence numbers still potentially in flight from a previous connection sharing the same 4-tuple. This prevents delayed duplicate confusion after a restart.
3 — Primitives

Hypothetical Transport Layer API

To write network applications, the OS exposes transport layer services through a set of primitives (system calls). A minimal client-server example (client sends "hello server", server replies "I am fine") requires five primitives:

Primitive Side Description
LISTEN Server Places the server in a waiting state — ready to accept an incoming connection. A machine not in LISTEN state cannot accept connections.
CONNECT Client Initiates end-to-end connection establishment (e.g., 3-way handshake in TCP). Blocks until the connection is established or times out.
SEND Both Passes data to the transport layer for transmission. Only valid in the ESTABLISHED state — calling SEND before connection setup is an error.
RECEIVE Both Retrieves data from the transport layer's receiver buffer. The application blocks here until data is available. Also only valid in ESTABLISHED state.
DISCONNECT Both Initiates connection teardown once data transfer is complete. Executed synchronously with timeout (asynchronous closure is not implementable on an unreliable channel).
Stateful protocol: The transport layer must remember the current state of each pipe. A SEND call before CONNECT has been completed is invalid. The protocol enforces this via state — only in ESTABLISHED state are SEND/RECEIVE permitted.

5 — Terminology

Segment vs Packet vs Frame

These terms have been used interchangeably in earlier lectures (acceptable when the concept being discussed — e.g., flow control — applies at multiple layers). From here on, use layer-specific terminology:

Application data / message
Transport → Segment (TCP) / Datagram (UDP)
Network → Packet / Datagram (IP)
Data Link → Frame
Physical → bits / signals

Each layer takes the unit from the layer above as its payload, prepends its own header, and passes the enlarged unit down. The term "frame" has been used in the flow control discussion at the data link layer, but flow control mechanisms exist at both the transport and data link layers — hence the acceptable informal overlap.

6 — Full Protocol Flow

Combining All Transport Layer Services

A complete connection-oriented transport protocol synthesises four components in sequence:

1 — Connection Establishment
Select ISN avoiding the forbidden region. Exchange ISNs with peer via handshake (e.g., 3-way). Socket = 6-tuple including both ISNs.
2 — Flow Control & Reliability
ARQ protocols (Go-Back-N / Selective Repeat) use sequence numbers to detect loss and retransmit. Receiver advertises buffer capacity to prevent overflow.
3 — Congestion Control
AIMD (Additive Increase, Multiplicative Decrease) dynamically adjusts sender rate. On congestion (packet loss), rate is halved; otherwise, linearly increased.

Sender Rate Formula

sender_rate = min(network_rate, receiver_advertised_rate)

The network rate is estimated and managed by AIMD. The receiver rate is explicitly advertised in each acknowledgement. The sender always respects the lower of the two constraints — this couples flow control and congestion control.

AIMD Behaviour Over Time

The sender rate vs. time graph (Refer Slide Time: 24:45 — drawn live on screen, worth watching) shows the following pattern:

Rate starts low → increases linearly (additive increase) → saturates at receiver advertised rate → packet loss detected → rate drops sharply (multiplicative decrease, e.g., halved) → cycle repeats. If receiver advertises a new rate, saturation point shifts. Result: rate oscillates in a sawtooth pattern, but remains bounded and fair.
Diagram at 24:45. The sender rate vs. time graph is drawn live on the blackboard. The intersection of the increasing sender rate curve with the receiver advertised rate line, and the sudden drops on loss detection, are clearly visible. Recommended to view directly.

Connection Closure

Once data transfer is complete, the DISCONNECT primitive is executed. As established in prior lectures, synchronous closure with timeout is used — asynchronous closure (both sides independently closing without acknowledgement) cannot be implemented reliably over an unreliable channel in a distributed system.

Summary

Lecture Summary

Addressing
IP address → identifies host (network layer). Port number → identifies application (transport layer). Together as a 6-tuple (with ISNs) → uniquely identifies a socket/logical pipe.
Stateful Protocol
The transport layer must track connection state. SEND/RECEIVE are only valid in ESTABLISHED state. State transitions are driven by executing primitives or receiving segments from the peer.
Primitives
LISTEN → CONNECT → SEND/RECEIVE → DISCONNECT. The OS exposes these as system calls (sockets API). Applications interact with the transport layer exclusively through these primitives.
Terminology
Segment (transport) → Packet/Datagram (network) → Frame (data link). Each layer encapsulates the layer above as payload and prepends its own header.
