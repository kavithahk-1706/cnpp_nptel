Network Security III — TCP/IP Security

Coverage of layer-wise security mechanisms across the TCP/IP stack: IPSec at the network layer, SSL/TLS at the transport layer, PGP at the application layer, and firewalls as perimeter defences. Prof. note: security is not a core exam topic for this course — treat this as conceptual exposure, not something to grind.

Motivation

Security as a Vertical Concern

The TCP/IP and OSI stacks were designed for communication, not security. Security was bolted on later, and that creates two fundamental tensions:

Backward compatibility constraint: A security-enabled packet must still be forwardable by intermediate devices (routers, switches) that are not security-aware. If adding a security header breaks the packet format such that a legacy router cannot parse it, the router drops the packet entirely. Every security mechanism must respect this.

Security is also an end-to-end phenomenon — it has to be maintained across all layers, peer-to-peer, from source to destination. Piecemeal security at just one layer leaves gaps everywhere else.

Physical and data link layers are hop-to-hop and typically under the administrative control of a single authority (e.g. within a campus), so security there is more of a physical management concern. From the network layer upward, traffic traverses paths outside your control — unknown intermediate routers, ISPs, etc. — which is where cryptographic security becomes essential.

Common Structure

Generic Security Protocol Packet Structure

Regardless of which layer applies security, the pattern is the same: the original payload is wrapped in a security envelope.

outer wrapper
Header of security protocol
content (encrypted)
Payload (IP / TCP / SMTP data)
integrity check
MAC — Message Authentication Code
outer wrapper
Trailer of security protocol
MAC ≠ MAC address. In this context MAC = Message Authentication Code — a cryptographic tag used to verify message integrity and authenticity. Nothing to do with Layer 2 hardware addresses.

The encapsulated payload + MAC portion is encrypted. The security header/trailer must remain parseable to legacy devices even if they cannot understand or decrypt the contents — otherwise they drop the packet.

Network Layer Security

IPSec — IP Security

IPSec is a collection of protocols standardised by the IETF to provide security at the network (IP) layer. It slots in where IP normally sits, so upper layers (TCP, UDP, applications) ride on top of it transparently.

Two Modes

Transport Mode
protects
Transport layer payload only (TCP/UDP segment)
IP header
Original IP header is NOT protected — left intact for routing
structure
IP-H → IPSec-H → IPSec payload → IPSec-T
use case
Host-to-host communication where both endpoints understand IPSec
Tunnel Mode
protects
Entire original IP packet including the original IP header
IP header
A new IP header is prepended; original is encapsulated and encrypted
structure
New IP-H → IPSec-H → [orig IP-H + payload] → IPSec-T
use case
VPN gateways; creates a virtual tunnel between two endpoints

IPSec Sub-Protocols: AH vs ESP

IPSec uses two types of security headers. You do not need the packet format details — knowing what each provides is sufficient.

Service AH (Authentication Header) ESP (Encapsulating Security Payload)
Access control✓ Yes✓ Yes
Message integrity / authentication✓ Yes✓ Yes
Entity authentication (data source)✓ Yes✓ Yes
Confidentiality (encryption)✗ No✓ Yes
Replay attack protection✓ Yes✓ Yes
Key takeaway: AH authenticates but does not encrypt. ESP does both. In practice, ESP is more commonly used precisely because confidentiality is usually required.
Transport Layer Security

SSL / TLS

SSL (Secure Sockets Layer) is the original protocol. TLS (Transport Layer Security) is the IETF-standardised successor — TLS is to SSL what TCP/IP is to ARPANET protocols. When people say "SSL" today they almost always mean TLS.

SSL/TLS operates between the application layer and TCP — it provides security at the transport layer so that the application layer riding on top does not need to implement its own crypto. HTTPS = HTTP over TLS.

Four SSL Sub-Protocols

Handshake Protocol
Negotiation between client and server — agree on cipher suites, exchange certificates, authenticate identities, establish session keys.
ChangeCipherSpec Protocol
Signals that subsequent messages will use the newly negotiated cipher parameters. Essentially a "switching lanes" notification.
Alert Protocol
Communicates error conditions and warnings (e.g. bad certificate, decryption failure, connection close). Can be fatal or non-fatal.
Record Protocol
The workhorse — takes data from the three protocols above plus application data, fragments it, compresses, applies MAC, encrypts, and hands it down to TCP. This is what actually interfaces with the transport layer.
Architecture note
SSL sits conceptually between application and transport. The three upper sub-protocols feed into the Record Protocol, which then sends down to TCP. Peer Record Protocols on both sides must agree on how to extract secured data.
Application Layer Security

PGP — Pretty Good Privacy

PGP is an example of security implemented at the application layer, specifically for email. The application layer has the advantage of being resource-rich — it can handle complex crypto without constraints that lower layers face.

PGP provides authentication and confidentiality for email messages. The flow (simplified):

Alice → Bob:
1. Alice's email message is hashed → creates a digest.
2. Alice signs the digest with her private key (authentication).
3. A one-time symmetric session key encrypts the message + digest (confidentiality).
4. The session key itself is encrypted with Bob's public key.
5. Bob uses his private key to recover the session key, decrypts the message, verifies Alice's signature.

Four algorithm components are involved: a public-key algorithm for session key encryption (PA1), a public-key algorithm for signing the digest (PA2), a symmetric algorithm for encrypting message + digest (SA), and a hash algorithm for creating the digest (HA).

The same philosophy applies to any application-layer security — the specific crypto scheme may differ (SMTP uses STARTTLS, HTTPS has its own cert infrastructure, SFTP has SSH), but the pattern of authenticate + encrypt + integrity-check is universal.
Perimeter Defence

Firewalls

A firewall is a logical boundary between a trusted internal network and an untrusted external network (typically the internet). It monitors and controls both incoming (internet → internal) and outgoing (internal → internet) traffic based on a defined policy.

What Firewalls Are Effective At

Protect local systems Block network-based threats Control outbound access to internet Restrict inbound access to internal servers

Types of Firewalls

Packet Filter
Packet Filtering Firewall
Operates at network + transport layers (L3/L4). Filters traffic by inspecting IP headers and TCP/UDP headers — source/destination IP address, source/destination port, protocol type. Rules are set up as an Access Control List (ACL): each incoming packet is matched against the list and either forwarded or discarded. Two default policies: discard all unless explicitly allowed (whitelist) or allow all unless explicitly blocked (blacklist).
Simple Transparent to users High speed Difficult to configure rules for large orgs No authentication of source
App Gateway
Application-Level Gateway (Proxy Firewall)
Operates at the application layer. Acts as a relay — the outside host connects to the proxy, the proxy connects to the inside host. The outside host never has a direct connection to the internal network. Service-specific: separate proxies for HTTP, FTP, SMTP, Telnet etc. Filters based on application-level rules (e.g. which URLs are allowed, which SMTP commands are permitted).
Higher security than packet filters Easy to log and audit all traffic Additional processing overhead per connection Gateway is a single point of failure
Circuit Gateway
Circuit-Level Gateway
Operates at the transport layer. Does not permit a direct end-to-end TCP connection between outside and inside hosts. Instead, the gateway sets up two separate TCP connections — one from outside host to gateway, one from gateway to inside host — and relays TCP segments between them without inspecting content. The security function is simply deciding which connections are allowed, based on session-level rules (e.g. recognised computer initiating the session). The SOCKS package is a classic example.
Breaks direct external-to-internal TCP path Does not inspect payload content Less common in practice

Firewall Configurations

Beyond a single firewall device, real deployments combine components for layered defence. Three common configurations:

Screened Host (Single-Homed) Packet filtering router + bastion host. Router only passes packets to/from the bastion host. Bastion host handles authentication and proxy functions. An attacker must compromise two systems to breach the internal network.
Screened Host (Dual-Homed) Bastion host has two physical network interfaces — one facing internet, one facing internal network. IP forwarding disabled, so packets physically cannot pass from one interface to the other without going through the bastion host software. Stronger isolation than single-homed.
Screened Subnet (DMZ) Two routers (outside + inside) with a subnet in between — the DMZ. Public-facing servers (DNS, mail, web) live in the DMZ. To reach the internal network an attacker must breach the outer firewall, traverse the DMZ, AND breach the inner firewall. Most secure common configuration. Highest hardware and management cost.

Typical Enterprise Deployment

A full perimeter defence combines: outer firewall router → DMZ zone (DNS/mail/web servers + honeypot + IDS) → inner firewall router → intranet. IDS (Intrusion Detection Systems) sit both on the internet-facing side and within the DMZ to catch traffic that slips past the outer firewall. The honeypot in the DMZ deliberately attracts attacks to study attacker signatures.

DMZ = Demilitarised Zone. A deliberately semi-exposed subnet. Servers here are accessible from the internet but separated from the trusted internal network by a second firewall. If a DMZ server gets compromised, the attacker still cannot directly reach internal hosts.
Concept Summary

Quick Reference: Security by Layer

Layer Protocol / Mechanism What it secures Key point
Network (L3) IPSec IP packet payload (transport mode) or entire IP packet (tunnel mode) AH = auth only; ESP = auth + encryption
Transport (L4) SSL/TLS TCP connection payload 4 sub-protocols: Handshake, ChangeCipherSpec, Alert, Record
Application (L5/L7) PGP Application-level messages (e.g. email) App layer has most resources; security is service-specific
Perimeter Firewall Network boundary — inbound + outbound traffic Packet filter (L3/4), App gateway (L7), Circuit gateway (L4)

Things to Remember

IPSec transport mode protects the payload but leaves the IP header exposed. Tunnel mode wraps the entire original packet in a new IP header — the original header is encrypted inside. Tunnel mode is what VPNs use.
SSL/TLS is not a single protocol — it's four sub-protocols working together. The Record Protocol is the one that actually integrates with TCP. The Handshake Protocol is the one doing the heavy crypto lifting at connection setup.
Packet filter firewall = fast but dumb (no auth, hard to configure for large nets). Application gateway = smart but slow (full application awareness, logs everything, but overhead per connection). Circuit gateway = breaks the TCP path without reading content (niche use, assumes trusted users inside).
Screened subnet / DMZ is the gold standard configuration in practice. Outer router + DMZ + inner router = three layers an attacker must penetrate. Honeypot in the DMZ is a bonus: let attackers think they're succeeding so you can learn their methods.
