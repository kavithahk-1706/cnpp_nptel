Protocol Stack Placement

Connecting the Network Stack to the OS

The TCP/IP protocol stack is concretely embedded inside the operating system. Understanding where each layer lives is the foundation of socket programming.

Application (APP1, APP2, …)
User Space
Transport (TCP / UDP)
Kernel Space
Network (IP)
Kernel Space
Data Link — upper part (MAC + LLC)
Kernel Space
Physical + lower Data Link
Hardware (NIC)
Key point: Application code lives in user space. To use the network stack, it must cross into kernel space via system calls — the UNIX mechanism for requesting kernel operations from user-space programs. The full kernel network stack implementation is at /usr/src/linux/net/.
Core Concept

What is a Socket?

A socket is a logical pipe — an endpoint of an end-to-end connection between two processes on (potentially different) machines.

Definition: A socket is a logical connection from one process to another, creating an end-to-end pipe identified by the 5-tuple: (src IP, src port, dst IP, dst port, protocol).
IP Address → Uniquely identifies the machine on the network (Network Layer)
Port Number → Identifies the specific process/application on that machine (Transport Layer)

Example: A browser on 202.141.81.2:8081 talking to an HTTP server at 203.12.23.43:8080 — the socket is the logical pipe joining these two endpoints. Multiple such pipes can co-exist, each identified by a different port combination.

Socket Types

Three Kinds of Sockets

SOCK_STREAM — Stream Socket
Protocol TCP
Nature Reliable, connection-oriented
Use case File transfer, HTTP — delivery guarantees needed
Data I/O read() / write()
SOCK_DGRAM — Datagram Socket
Protocol UDP
Nature Unreliable, connectionless
Use case VoIP, streaming, DNS — low latency over guarantees
Data I/O recvfrom() / sendto()
Raw Socket (SOCK_RAW): Bypasses the transport layer entirely and interacts directly with the IP layer. Not covered in detail in this lecture.
Socket API

Server-Side Call Sequence

System CallPurposeNotes
socket() Create socket; returns a socket file descriptor Args: domain, type, protocol
bind() Bind socket to a port on this machine Associates fd with a sockaddr_in struct
listen() Put socket in passive listening state Backlog param = max queued pending connections
accept() Accept incoming connection; returns a new socket fd for that client Blocks until client connects; triggers TCP 3-way handshake
send() / recv() Transfer data Or read() / write() for stream sockets
close() Tear down connection and free socket

Client-Side Call Sequence

System CallPurpose
socket() Create client-side socket endpoint
connect() Initiate connection to server IP:port — fires TCP 3-way handshake
send() / recv() Transfer data
close() Close the connection
Why no bind/listen on the client? The server must publicly announce a fixed port so clients can reach it. The client only ever initiates outward — it does not need to be findable. The OS assigns the client a random ephemeral port automatically at runtime.

Active Open vs Passive Open

Server
Passive Open

Announces address, remains permanently in listening state waiting for incoming connections. Driven by listen() + accept().

Client
Active Open

Opens a connection only when data transfer is needed. Driven by connect(). Connection is always client-initiated.

Why accept() Returns a New Socket FD

The original listening socket must stay open to receive future clients. When a client connects, accept() creates a brand-new socket fd dedicated to that specific client's logical pipe. All data transfer for that client uses the new fd. This is how the server handles multiple simultaneous clients — one fd per client.

API Details

socket() Parameters

int s = socket(domain, type, protocol);

ParameterTypical ValueMeaning
domain AF_INET IPv4 addressing
type SOCK_STREAM / SOCK_DGRAM TCP or UDP socket respectively
protocol 0 Let OS choose appropriate protocol for the given socket type

struct sockaddr_in Fields

sin_family → Address family — set to AF_INET for IPv4
sin_addr.s_addr → INADDR_ANY — OS auto-selects the machine's active network interface address. Can also be set to a specific IP if needed.
sin_port → Port number — must be converted to network byte order: htons(port)
Host vs Network Byte Order

Endianness and Why htons() is Mandatory

Little Endian
Order Least significant byte stored first (low address)
Example 0A0B0C0D → memory: 0D, 0C, 0B, 0A
CPUs x86, x86-64 — most modern PCs
Big Endian — Network Standard
Order Most significant byte stored first (low address)
Example 0A0B0C0D → memory: 0A, 0B, 0C, 0D
CPUs SPARC, older MIPS; chosen as network byte order
The problem: A little-endian machine sends bytes in reversed order. A big-endian receiver reads them in its own (forward) order — the value is silently misinterpreted. No error is thrown.
The fix: Network byte order is standardised as big-endian. Convert before transmitting with htons() (16-bit, e.g. port numbers) or htonl() (32-bit, e.g. IP addresses). Convert after receiving with ntohs() / ntohl(). Always do this — skipping it is silent corruption.
Data Transfer

Send/Receive Functions by Socket Type

TypeReceiveSendNote
SOCK_STREAM (TCP) read(newsockfd, buf, size) write(newsockfd, msg, len) Use the new fd from accept()
SOCK_DGRAM (UDP) recvfrom(sock, buf, size, 0, &from, &len) sendto(sock, msg, len, 0, &addr, addrlen) Addresses passed explicitly per call — no connection
UDP shortcut: No connect() or accept(). Server: bind → recvfrom loop. Client: socket → sendto. That's it.

UDP Server (udpserver.c) — Step by Step

socket()→AF_INET, SOCK_DGRAM, 0
sockaddr_in→AF_INET, INADDR_ANY, htons(port from argv)
bind()→Bind socket to specified port
setsockopt()→SO_REUSEADDR — allows port reuse across connections
recvfrom() loop→Wait for data; client address automatically populated in clientAddr struct
sendto()→Echo data back to client using clientAddr from recvfrom()

UDP Client (udpclient.c) — Step by Step

socket()→AF_INET, SOCK_DGRAM, 0
gethostbyname()→Resolve server hostname ("localhost") to IP
sockaddr_in→Set server IP + port from command-line args
sendto()→Send "Hello there!!" directly — no connect() needed
recvfrom()→Receive echoed reply from server
Ephemeral ports observed in demo: The client never calls bind(), so the OS assigns a random available port at runtime. Multiple runs of the same client show different source ports (43036, 60026, 34170, 47378…). This is standard and expected for any unbound client socket.
Summary

Key Takeaways

What
Socket = Logical Pipe

Software abstraction for process-to-process communication identified by the 5-tuple (src IP, src port, dst IP, dst port, protocol).

How
System Calls (UNIX)

Server: socket → bind → listen → accept. Client: socket → connect. TCP 3-way handshake fires inside connect/accept. UDP skips both.

Types
Stream vs Datagram

SOCK_STREAM = TCP = reliable, connection-oriented. SOCK_DGRAM = UDP = unreliable, connectionless. Determines which calls and I/O functions to use.

Gotcha
Byte Order Conversion

Always htons() port numbers into sockaddr_in. Always ntohs() when reading from network. Skipping this = silent corruption between different-endian machines.

MCQ Checkpoints

Q: Which calls are absent in a UDP server but present in a TCP server?
A: listen() and accept() — UDP has no connection establishment.
Q: Why does accept() return a new socket file descriptor?
A: To maintain separate logical pipes for each simultaneously connected client, while the original listening socket stays open for future connections.
Q: What is network byte order and why is it needed?
A: Network byte order is big-endian — a universal standard ensuring machines with different endianness (e.g. little-endian x86 vs big-endian SPARC) don't misinterpret multi-byte values like port numbers and IP addresses.
Q: What does the backlog parameter in listen() control?
A: The maximum number of connection requests queued while the server is processing. E.g. listen(sockfd, 5) allows up to 5 pending connections before new ones are refused.
Q: Why does a UDP client use a different source port on every run?
A: The client never calls bind(), so the OS picks a random available ephemeral port at runtime. Each execution gets a new one.
