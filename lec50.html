Lecture 50 medium priority

Data Link Layer — Flow & Error Control

Covers the three primary ARQ mechanisms deployed at Layer 2: Stop-and-Wait, Go-Back-N, and Selective Repeat. Conceptually mirrors transport-layer flow/error control, but operates hop-to-hop rather than end-to-end.

Context & Motivation

The data link layer must handle two fundamental problems in frame delivery over a single hop (e.g., host to a nearby switch):

Flow Control
Regulates how much data the sender can transmit before waiting for an ACK. The receiver has a finite processing speed and a finite buffer — it must be able to signal the sender to slow down or stop before it is overwhelmed.
Error Control
Detects and recovers from lost, damaged, or out-of-order frames. Built on ARQ (Automatic Repeat reQuest): whenever an error is detected, the affected frames are retransmitted.
Hop-to-hop vs end-to-end: Transport layer (TCP) provides flow/error control end-to-end (process-to-process). The data link layer provides the same guarantees hop-to-hop (node-to-node). Both problems exist at both layers, but the scope differs.
Mechanism 1

Stop-and-Wait ARQ

The simplest protocol. The sender transmits exactly one frame, then halts and waits for an acknowledgement before sending the next.

Normal Operation

Both frames and ACKs are numbered alternately 0 and 1. The sender holds control variable S (last sent frame number); the receiver holds R (next expected frame number). ACK number = number of the next expected frame, not the one just received.

S=0→Frame 0 sent
R=0→Receiver accepts, sends ACK 1
S=1→Frame 1 sent
R=1→Receiver accepts, sends ACK 0
...→Alternates: 0 1 0 1 …

Lost / Damaged Frame

If the receiver gets a damaged frame, it discards it and holds R unchanged. The sender's timer expires and it retransmits the same frame.

Lost ACK

If the ACK is lost or damaged, the sender's timer expires and it retransmits the frame. The receiver — having already accepted that frame — is now expecting the next one. It therefore discards the duplicate using the sequence number and sends no new ACK. The sender eventually receives the retransmitted copy's ACK and advances. Error detection in the ACK uses the frame's FCS / CRC-32 field.

Delayed ACK

If an ACK arrives after the timer has expired, the sender has already retransmitted. The receiver discards the duplicate frame (R mismatch). The sender receives two ACKs; it discards the second one.

Piggybacking

When both stations have data to send simultaneously, the ACK for received data is embedded inside the outgoing data frame rather than sent as a separate control frame. This saves bandwidth by eliminating dedicated ACK transmissions.

Key limitation: At any instant, only one unacknowledged frame is in flight. The transmission medium sits idle while waiting for the ACK — a significant waste of available bandwidth.
Mechanism 2

Go-Back-N ARQ

Addresses Stop-and-Wait's inefficiency by allowing the sender to transmit up to W frames before needing any ACK. A window of outstanding (unacknowledged) frames is maintained by the sender.

Sequence Numbers

Frames are numbered sequentially. If the header allocates m bits for sequence numbers, the range is 0 to 2m − 1. The sequence then wraps: 0, 1, 2, …, 7, 0, 1, 2, … (for m = 3).

Sender Sliding Window

The sender maintains three variables: SF (first outstanding frame), SL (last frame sent), and S (next frame to send). Maximum window size = 2m − 1. When a correct ACK arrives, the window slides forward to admit new unsent frames.

Receiver Window

Receiver window size is always 1. The receiver maintains a single variable R. It only accepts a frame whose sequence number exactly equals R. Any frame arriving out of order is silently discarded — no buffering.

Acknowledgement Behaviour

Cumulative ACKs: ACKs are cumulative. ACK N means "all frames up to and including N−1 received correctly; send me N next." A single ACK can implicitly acknowledge multiple frames. If ACK 1, ACK 2, ACK 3 are all lost but ACK 4 arrives before any timer expires, no retransmission is needed.

Error Scenario — Lost Frame

Suppose frames 0–5 are in flight and frame 2 is lost. The receiver receives frame 3, but R = 2, so it discards frame 3 (and all subsequent frames). The timer for frame 2 expires at the sender. The sender goes back to 2 and retransmits frames 2, 3, 4, 5 — hence the name Go-Back-N.

Window Size Constraint — Why 2m − 1?

If window size equals 2m (the full sequence range), a complete cycle of frames can be sent and all ACKs can be lost. The retransmitted frame 0 would be indistinguishable from a brand-new frame 0, causing the receiver to accept it erroneously. Capping the window at 2m − 1 ensures the first frame of the new cycle can never appear inside the current window.

Mechanism 3

Selective Repeat ARQ

A refinement of Go-Back-N that avoids retransmitting frames that were already received correctly.

NAK (Negative Acknowledgement)

When the receiver detects a damaged or missing frame, it sends a NAK N identifying the specific frame number N that needs to be retransmitted. The sender retransmits only that frame — not everything after it. This makes Selective Repeat more bandwidth-efficient on noisy links.

Error Scenario — Lost Frame

Frames 0, 1, 3 are received correctly; frame 2 is lost. Frames 0, 1, 3 are buffered at the receiver (they are within the receiver window). The receiver sends NAK 2. The sender retransmits only frame 2. Once received, the receiver delivers the complete sequence 0, 1, 2, 3 in order.

Window Size Constraint — Why 2m / 2?

Both sender and receiver windows must be at most 2m / 2 (half the sequence space). Because the receiver now buffers out-of-order frames, a larger window creates ambiguity: a retransmitted frame 0 could be mistaken for a new frame 0 from the next cycle and accepted erroneously. Halving the window ensures no overlap between current and next cycle.

Example (m = 2): Sequence range 0–3 (4 values). Max window size = 4 / 2 = 2. If window size is 3 (too large) and all ACKs are lost, the duplicate frame 0 falls within the receiver window and is accepted as a new frame — an error.

Concept Summary
Flow Control — manages the rate of transmission so a fast sender does not overwhelm a slow receiver's buffer. The receiver signals the sender to pause or reduce speed before limits are breached.
Error Control — detects lost or damaged frames using FCS/CRC and triggers retransmission via ARQ. Three strategies exist: (1) Stop-and-Wait: simple but wastes bandwidth; (2) Go-Back-N: sends a window of frames, retransmits everything from the error point — simple receiver but can waste bandwidth on noisy links; (3) Selective Repeat: sends a window, retransmits only the bad frame — most efficient but requires the receiver to buffer and reorder frames.
Window size matters: Setting the window too large causes sequence number ambiguity — the receiver cannot distinguish a retransmitted old frame from a new frame with the same number, leading to silent data corruption. The constraints 2m − 1 (Go-Back-N) and 2m/2 (Selective Repeat) are not arbitrary; they are exact thresholds below which ambiguity is impossible.
Piggybacking — an optimisation where a node embeds its ACK inside a data frame travelling in the opposite direction. Eliminates standalone ACK transmissions and saves bandwidth in full-duplex scenarios.
