Lecture 13 HIGH PRIORITY

Transport Layer – II: Connection (Contd.)

Continuation of connection establishment — forbidden regions, ISN selection, self-clocking, three-way handshake, and connection release. Core TCP mechanics live here.

Recap from Lec 12

The core problem: in a packet-switched network, packets can be arbitrarily delayed. A connection request from a previous, already-dead connection can show up as a perfectly valid-looking new request — the delayed duplicate problem.

The solution anchor: use byte sequence numbers as a virtual clock. A receiver can tell duplicates apart only if the ISN of a new connection doesn't collide with sequence numbers still alive from the old one.

Key constraint: within any time window T (max packet lifetime in the network), no two bytes with the same sequence number should be simultaneously outstanding for the same source-destination-port pair.
The Forbidden Region

What It Is

Plot sequence numbers on the Y-axis, time on the X-axis. A connection's sequence numbers trace an upward line as it sends data. Every byte at a given sequence number has a lifetime T. The band around that line within time T is the forbidden region of that connection.

If a new connection's ISN falls inside the forbidden region of a previous connection (same src-dst-port), the receiver cannot distinguish old bytes from new — ambiguity, corruption, bad.

Visual hint: The forbidden region diagram — sequence numbers vs. time, two overlapping shaded bands for Conn 1 and Conn 2 — appears at 04:48. Sketching it yourself is worth two re-reads of the text.

Two Ways to Avoid Overlap

Shift in Time
Wait long enough after Conn 1 crashes before starting Conn 2, so all Conn 1 bytes are guaranteed dead (exceeded lifetime T). Forbidden regions are temporally separated.
Downside
You have to sit idle and wait. Not practical.
Shift in Sequence Number Space
Choose an ISN for Conn 2 high enough above Conn 1's sequence numbers that forbidden regions don't overlap, even if Conn 2 starts immediately.
Preferred
No waiting. This is the approach TCP uses.
Rate Constraints on Sequence Number Growth

The Fast Connection Problem 10:12

If Conn 2 starts with a safe ISN but then sends data too fast, its sequence numbers race ahead and can wrap around into Conn 1's forbidden region from the other side — overlap created even though the ISN was chosen correctly.

The Slow Connection Problem 12:33

If Conn 2 sends too slowly, its sequence numbers creep up so gradually that by the time it reaches Conn 1's old range, those bytes from Conn 1 are still alive in the network. Overlap again.

Conclusion: both too-fast and too-slow growth cause forbidden region overlaps. Sequence numbers must grow at a bounded rate.

Self-Clocking / ACK Clocking 13:30

TCP enforces a bounded rate using the hardware clock of the local machine only — no cross-machine synchronisation required. The rule: at most one segment per clock tick. Clock ticks are adjusted based on received ACKs — every ACK triggers the next transmission. This is called self-clocking.

This also prevents the 32-bit sequence number space (2³² values) from wrapping around faster than lifetime T. Application data is buffered at the transport layer; the transport layer picks bytes from the buffer and transmits at this bounded rate — not at the application's arbitrary generation rate.

The full interaction between ISN selection and rate control will be clearer after the flow control lectures (ARQ). Hold that thought.
Three-Way Handshake

Why It's Needed 17:28

The receiver does not store old sequence numbers from past connections. Each sender manages its own ISN. The three-way handshake lets both sides independently validate each other's ISNs through acknowledgement — positive synchronisation without shared state.

Normal Case

Host 1→ SYN (seq = x)→ Host 2
Host 2→ SYN-ACK (ack = x, seq = y)→ Host 1
Host 1→ ACK (ack = y) + DATA→ Host 2

Host 1 sends ISN = x. Host 2 ACKs x and proposes its own ISN = y (connections are bidirectional — both directions need ISNs). Host 1 validates the ACK, then ACKs y and starts sending data. Both directions are now synchronised.

Case 1: Delayed Duplicate SYN 19:12

An old SYN(x) from a crashed+restarted Host 1 arrives at Host 2. Host 2 sends SYN-ACK(x, y). Host 1 receives this, recognises it doesn't correspond to any current connection request, and sends a REJECT. Host 2 sees the reject → abandons the half-open connection. Handled cleanly.

Case 2: Both SYN and ACK Are Delayed Duplicates 20:27

Old SYN(x) → Host 2 sends SYN-ACK(x, y) → Host 1 sends REJECT (as above). Separately, an old ACK arrives at Host 2 claiming ack = z. But Host 2 proposed seq = y — z ≠ y → mismatch → Host 2 discards the stale ACK. Both sides correctly identify and discard the zombie exchange.

Connection Release

Releasing a connection avoids the ISN problem entirely, but introduces a different challenge: how do you shut down a bidirectional connection without losing in-flight data?

Asymmetric Release
One party unilaterally sends a DR (Data Release) and closes. Done.
Problem
The other party may still have data to send. If it transmits after the first party closed, that data is lost permanently.
Verdict
Data loss is possible. Simple, but unsafe.
Symmetric Release
Bidirectional connection = two independent unidirectional connections. Each direction released separately. Full connection closed only when both halves are done.
Protocol
Host 1: "I am done." Host 2: "I am done too." → Both halves released → connection closed.
Works when
Each side knows exactly when it finishes sending. Suited to fixed-size transfers.

The Two Army Problem — Why Perfect Symmetric Release Is Impossible 28:23

A white army sits in a valley. Two halves of a blue army sit on opposite hills — together they outnumber the white army, but only if they attack simultaneously. Messengers must cross the valley to coordinate, but the white army can intercept any messenger.

No finite exchange of messages can give both blue halves certainty that the other received the attack signal. Every "final confirmation" is itself unreliable, creating infinite regress. Consensus over an unreliable channel is provably impossible.

Mapping to networks: two hosts over an unreliable channel can never reach guaranteed mutual certainty about closing state. A provably correct symmetric release protocol cannot exist.

Practical Protocol: Symmetric + Timeouts 30:22

We attempt symmetric release and fall back gracefully:

Host 1→DR (Data Release)→Host 2 — starts timer
Host 2→DR + ACK→Host 1 — starts timer
Host 1→ACK → releases connection
Host 2receives ACK → releases; or releases on timeout

Failure Scenarios

Final ACK lost: Host 2 never receives it. Timer expires → Host 2 closes independently. No problem.

DR from Host 2 lost: Host 1 times out on its own DR → retransmits. Host 2 replies again. Exchange completes.

Both DR and ACK lost: Both hosts time out. Each retries N times. After N × T, both forcibly close. Some data loss is possible but connection terminates.

Accepted trade-off: some residual data loss is possible and accepted. The goal is to minimise it via timeouts — not eliminate it, which is impossible. Performance over total correctness.
Concept Summary

Connection Establishment

A new connection's forbidden region must not overlap with that of an old connection on the same src-dst-port. Overlap is avoided by either waiting (impractical) or choosing a sufficiently high ISN (TCP's approach). Sequence numbers must grow at a bounded rate — enforced by self/ACK-clocking using the local hardware clock. The three-way handshake gives both sides independent ISN validation and the ability to detect delayed duplicates by checking whether ACK numbers match what they actually sent.

Connection Release

Asymmetric release is simple but data-lossy. Symmetric release is cleaner — each unidirectional half is closed separately — but a perfect protocol for it is provably impossible on unreliable networks (Two Army Problem). In practice: attempt symmetric release with timeouts, fall back to independent forced closure after N retries. Residual data loss is accepted as unavoidable.

What Comes Next

Once the ISN is established via the three-way handshake, flow control and ARQ (Automatic Repeat reQuest) protocols maintain sequence numbers for all subsequent data segments. The hard part was the initial handshake; flow control picks up from there.
