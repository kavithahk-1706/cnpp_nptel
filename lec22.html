Lecture 22 ★ High Priority

Transmission Control Protocol — IV
Congestion Control

Prof. Sandip Chakraborthy · IIT Kharagpur · CNPP

Overview

This lecture covers TCP's congestion control mechanism — how the protocol detects network congestion and dynamically adjusts its sending rate to achieve both efficiency (maximum link utilisation) and fairness (equitable sharing among competing flows). It builds directly on the AIMD principles and flow control concepts from the previous lectures.

Core insight: TCP merges two independent rate constraints — one from the network (congestion window, CWnd) and one from the receiver (receiver window, RWnd) — and always transmits at the more conservative of the two.
Key Variables

Windows and Sending Rate

Congestion Window
CWnd

Maximum bytes the sender may have unacknowledged in the network at any instant. Set and updated by the sender based on observed congestion signals.

Receiver Window
RWnd

Advertised by the receiver — the free buffer space available at the destination. Used by flow control (Lec 21).

Sender Window
SWnd = min(CWnd, RWnd)

The actual operating window. Using the minimum ensures neither the network nor the receiver is overloaded simultaneously.

Sending Rate (approx.)
Rate ≈ CWnd / RTT

Dividing the congestion window by RTT approximates throughput in bytes/second.

Why AIMD?

AIMD is chosen because it provably converges to a state that simultaneously maximises link utilisation and satisfies max-min fairness. AIAD converges too slowly; MIMD is unstable and violates fairness. TCP uses packet loss as its binary congestion signal — no explicit network feedback is needed.

Historical Context

The 1986 Congestion Collapse

The growing popularity of the early internet led in 1986 to the first documented congestion collapse — a prolonged period during which network goodput dropped by more than a factor of 100. This motivated Van Jacobson to design the first TCP congestion control algorithm, published in 1988.

Jacobson's constraint: The algorithm had to work within the existing TCP header and state machine, requiring no changes to deployed machines — it had to be instantly deployable on the live internet.

Why Packet Loss = Congestion Signal (in Wired Networks)

In wired networks, channels are effectively lossless — guided media are shielded from interference, and the link layer handles residual bit errors. Any packet loss therefore originates from buffer overflow at an intermediate router, which is by definition congestion. Packet loss is thus a reliable, in-band congestion signal detectable via the existing retransmission timeout mechanism.

Wireless caveat: In wireless networks, channel-induced losses are frequent and cannot be distinguished from congestion-induced drops. TCP's assumption breaks down — a known limitation in wireless environments.
ACK Clocking

ACKs as Bottleneck Probes 12:09

In a multi-hop path, packets sent as a burst drain onto the bottleneck (slowest) link at that link's rate. Consequently, ACKs return to the sender at approximately the rate of the bottleneck link — regardless of how fast earlier links are.

Jacobson used this insight: ACKs serve as natural clocks that pace CWnd adjustments to the bottleneck capacity. Every incoming ACK triggers a congestion window update. No external probing timer is needed.

Motivation for Slow Start

Problem: Pure Additive Increase is Too Slow 14:41

With CWnd starting at 1 and increasing by 1 MSS per RTT, reaching the Bandwidth-Delay Product takes prohibitively long on fast links:

Link→10 Mbps, RTT = 100 ms
BDP (target CWnd)→1 Mbit (= 100 × 1250-byte packets)
Time to reach BDP at +1/RTT→100 RTTs = 10 seconds

Most HTTP connections close before 10 seconds. Pure additive increase means the connection never operates near its capacity. Solution: slow start.

Phase 1 — Slow Start

Exponential Growth at Connection Start 18:08

Slow start is a misnomer. The name reflects starting from a low rate (CWnd = 1 MSS), not slow growth. CWnd is doubled every RTT — growth is exponential.

Mechanism: For each ACK received before the retransmission timer fires, the sender adds 1 MSS to CWnd. A window of N packets generates N ACKs, so CWnd doubles each RTT.

Slow Start Threshold

ssthresh — Gating Exponential Growth 19:58

Initial value
ssthresh = BDP (or very large)

Set large at start — sender has no prior knowledge of network capacity, so it probes aggressively.

On packet loss (RTO)
ssthresh = CWnd / 2

Halved to record the observed congestion point. CWnd resets to 1 MSS and slow start restarts.

Rule: CWnd < ssthresh → slow start (exponential). CWnd ≥ ssthresh → congestion avoidance (linear additive increase). ssthresh acts as a memory of where the network last became congested.

Phase 2 — Congestion Avoidance

Additive Increase above ssthresh 21:10

Above ssthresh, TCP enters congestion avoidance — the additive increase phase of AIMD. A per-ACK partial increment approximates adding 1 MSS per full RTT:

Formula (applied per ACK received):
CWnd = CWnd + (MSS × MSS) / CWnd

Over one RTT, the sum of CWnd/MSS such increments equals approximately 1 MSS — the intended additive increase rate.

This linear growth continues until a congestion event (RTO or triple DUPACK) is detected, which triggers multiplicative decrease.

Congestion Detection

Two Signals 24:56

Why 3 DUPACKs?

The threshold of three is an engineering heuristic by Jacobson — chosen to filter out natural out-of-order delivery (which typically generates 1–2 DUPACKs) while reliably identifying actual packet loss.

Inferring the Lost Packet

TCP uses cumulative acknowledgements. Receiving DUPACK(3) repeatedly means everything up to sequence 3 was delivered, but a later segment is missing. Packets 5, 6, 7 arriving out of order each trigger DUPACK(3). The sender infers segment 4 (the next expected in sequence) is the lost packet and can retransmit it immediately.

TCP Tahoe

Fast Retransmit 27:35

TCP Tahoe introduced fast retransmit: on 3 DUPACKs, retransmit the inferred lost segment without waiting for RTO. However, Tahoe's congestion response is the same regardless of whether the signal was RTO or 3 DUPACKs:

ssthresh ← CWnd / 2 CWnd ← 1 MSS Restart slow start from 1

The limitation: Tahoe treats 3 DUPACKs as severely as an RTO, even though DUPACKs indicate ongoing flow. This causes unnecessary slow start cycles and reduces throughput for mild congestion events.

TCP Reno

Fast Recovery 28:51

TCP Reno's key insight: 3 DUPACKs mean some packets are still in transit. A full slow start restart is wasteful. Reno introduces fast recovery to stay near the congestion point:

FAST RECOVERY — On receiving 3 DUPACKs
1→ssthresh ← CWnd / 2
2→Retransmit missing segment (fast retransmit)
3→CWnd ← ssthresh + 3 (credit: receiver already holds 3 out-of-order segments)
4→Each further DUPACK: CWnd += 1, send new segment if window permits
5→On new (cumulative) ACK: CWnd ← ssthresh, exit fast recovery
6→Resume congestion avoidance (linear increase from ssthresh)

The +3 at Step 3 reflects that three segments have already been received and buffered at the receiver — those slots in the network are occupied, so CWnd can immediately account for them. Each additional DUPACK represents one more buffered packet.

Tahoe vs Reno — critical distinction: Both fast-retransmit on 3 DUPACKs. Tahoe then resets CWnd to 1 MSS and redoes slow start (slow recovery). Reno enters fast recovery, resumes at ssthresh, and skips slow start — significantly higher throughput for transient congestion.

If the congestion is detected by RTO instead, Reno behaves identically to Tahoe — CWnd → 1 MSS, restart slow start — because RTO signals severe congestion requiring a full rate reset.

TCP Variants

Chronological Overview

TCP Tahoe
First congestion control. Introduces CWnd, ssthresh, slow start, fast retransmit. Resets CWnd to 1 MSS on any loss signal.
TCP Reno
Adds fast recovery for 3-DUPACK events. Skips slow start for mild congestion. Still resets to 1 MSS on RTO.
TCP SACK
Selective ACK — receiver explicitly reports received-out-of-order ranges. Sender retransmits only missing segments (Selective Repeat ARQ semantics, not Go-Back-N).

TCP New Reno, CUBIC, BBR, and many others followed. The core three-phase structure is preserved across all classical variants.

Key Takeaways
AIMD
Additive Increase, Multiplicative Decrease
The rate-control discipline. +1 MSS/RTT on no loss; ÷2 on loss. Provably converges to max-min fair, high-utilisation operating point. MIMD and AIAD do not have this property.
SWnd
Sender Window = min(CWnd, RWnd)
Unifies congestion control (network constraint) and flow control (receiver constraint). The binding constraint determines actual throughput.
ACK Clock
ACKs return at bottleneck rate
ACKs naturally pace the sender to the slowest link in the path. CWnd adjustments triggered per-ACK are implicitly bottleneck-aware with no explicit feedback.
ssthresh
Slow Start Threshold
Divides exponential slow start from linear congestion avoidance. Updated to CWnd/2 on every loss event — it is the network's congestion point memory.
Loss Signal
RTO vs 3 DUPACKs
RTO = severe congestion → reset to 1 MSS. 3 DUPACKs = mild congestion (flow ongoing) → fast recovery to ssthresh. Reno distinguishes these; Tahoe does not.
