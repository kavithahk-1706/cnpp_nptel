Lecture 23 HIGH PRIORITY

User Datagram Protocol

Transport layer's "fast and loose" option — plus QUIC, the protocol that's quietly eating TCP's lunch.

Motivation

Why UDP? TCP's Overhead Problem

TCP is reliable and correct, but reliability comes at a cost. For many real-world applications, that cost is unacceptable. Three specific overheads make TCP impractical in certain scenarios:

Connection Overhead
Every TCP session requires a 3-way handshake to open and another exchange to close. For a short DNS request-response, you're burning 6 messages just on setup and teardown — before sending a single byte of actual data.
Retransmission Blocking
TCP's reliability mechanism blocks new data when a packet is lost. The stream stalls waiting for a retransmission, which increases latency for subsequent packets even if they arrived fine.
Slow Start
TCP's congestion control forces the sender to begin at a tiny congestion window and ramp up slowly. On high-speed links with short-lived flows (e.g. HTTP), the connection closes before the window ever reaches operating capacity.
Bottom line: For short-lived, latency-sensitive, or loss-tolerant applications, TCP's overhead costs more than it saves. UDP strips all of that out.
Protocol Overview

UDP — User Datagram Protocol

UDP is essentially a thin wrapper over the IP layer. It passes IP's services directly to the application, adding only the minimum transport-layer functionality: port-based multiplexing and a basic integrity check.

If the application needs any of the above, it must implement them itself at the application layer. UDP simply provides the pipe.

Header Structure

UDP Header

Compared to TCP's 20-60 byte header, the UDP header is a fixed 8 bytes with exactly four fields:

Field Size Purpose
Source Port 16 bits Port number of the sending application (identifies the sender's socket)
Destination Port 16 bits Port number of the receiving application (identifies the target socket)
Length 16 bits Total length of UDP header + data in bytes. Required since there is no stream boundary — receiver must know where the datagram ends.
Checksum 16 bits Error detection. Even though UDP is unreliable, the receiver still needs to verify the data was not corrupted in transit.
Note: The simplicity of the UDP header is one of the reasons UDP is faster — there is simply less to process at both ends.
Checksum Mechanics

Internet Checksum

The checksum is a hash function applied to the message: f(M) = C. The resulting fixed-size value C is stored in the header. At the receiver, the same function is applied to the received data and compared against C. A mismatch means corruption occurred.

What the Checksum Protects Against

The internet checksum detects hardware/signal faults — bit flips introduced during analog-to-digital conversion, modulation/demodulation, or physical transmission errors. It is not a cryptographic mechanism and does not protect against deliberate tampering or security attacks.

How It's Computed — The Simple Method

The internet checksum avoids expensive cryptographic hash functions (too slow for per-packet computation). Instead:

Divide the entire message into fixed-size blocks → perform one's complement addition across all blocks → the result is the checksum. Simple, fast, sufficient for error detection.

The Pseudo Header

Both TCP and UDP compute their checksum over three components: transport header + data + a pseudo header. The pseudo header is never transmitted — it exists only for checksum computation and is discarded immediately after.

Pseudo Header FieldSourcePurpose
Source IPIP headerValidates critical routing fields end-to-end. IP routers re-compute their own IP-level checksum at every hop, so the IP checksum alone cannot guarantee these fields are intact from source to destination.
Destination IPIP header

Reserved bits (8)IP header
Why bother? Routers strip, process, and re-attach the IP header at every hop. A faulty router could silently corrupt the source/destination IP. Since TCP/UDP headers are end-to-end (never modified by routers), including the pseudo header in the checksum gives transport layer a second, independent check on those critical IP fields.

Modern Protocol

QUIC — Quick UDP Internet Connections

Developed by Google (first published at SIGCOMM 2017). QUIC is a next-generation transport protocol built on top of UDP that reimplements TCP's useful features (reliability, flow control, congestion control, ordering) at the application layer, while fixing TCP's fundamental structural problems.

QUIC Feature 1 — 0-RTT Connection

TCP requires a new 3-way handshake for every connection. QUIC dramatically reduces this overhead using a credential-caching mechanism:

ScenarioHandshake TypeWhat Happens
First ever connection 1-RTT Client sends inchoate CHLO → server rejects with security credential → client sends complete CHLO + encrypted request → server sends SHLO + encrypted response. 1 round trip required.
Subsequent connections (same server) 0-RTT Client already has server's credential from last time. Sends complete CHLO + encrypted request immediately. No waiting for credential exchange.
Server credential changed 1-RTT Server rejects complete CHLO and sends updated credential. Client reinitates with new credential. Falls back to 1-RTT for this one session only.
Key terminology: CHLO = Client Hello (credential exchange message). SHLO = Server Hello. RTT = Round Trip Time.

QUIC Feature 2 — Multi-Streaming + Head-of-Line Blocking Free

This is QUIC's most significant architectural improvement over TCP. Understanding it requires knowing what head-of-line (HoL) blocking is:

Head-of-Line Blocking (TCP's problem)

HTTP/2 (and SPDY before it) multiplexes multiple request-response streams over a single TCP connection to avoid per-stream handshakes. But TCP guarantees in-order delivery. If one packet in one stream is lost, TCP halts delivery of all streams waiting for that single packet — even if the other streams' packets arrived safely. One lost packet blocks everything.

QUIC's Solution

QUIC runs multiple streams over a single UDP connection. UDP has no in-order requirement — it simply passes packets upward as they arrive. QUIC itself handles per-stream sequencing. A lost packet in stream 1 only blocks stream 1; streams 2 and 3 continue unaffected. Each stream gets independent flow and congestion control.

Evolution of Multi-streaming

HTTP/1.1: Separate TCP connection per request → many handshakes, many slow starts.
HTTP/2 / SPDY: Multiplex over one TCP → single handshake, but HoL blocking introduced.
QUIC (HTTP/3): Multiplex over one UDP → single handshake, no HoL blocking, independent stream control.

QUIC Feature 3 — Packet Sequence Numbers (Not Byte Sequence)

TCP uses byte sequence numbers, which creates an ambiguity problem during retransmission: a retransmitted packet has the same sequence number as the original. This leads to the "retransmission ambiguity problem" and forces TCP to use duplicate ACKs carefully.

QUIC assigns a new sequence number to every transmitted packet, including retransmissions. This means:

No duplicate sequence numbers No retransmission ambiguity No duplicate ACK mechanism needed Simpler loss detection

QUIC — Built-in Security

Unlike TCP where encryption is layered on via TLS/SSL, QUIC has end-to-end encryption as a core design property. Every packet is encrypted. No separate security negotiation phase is needed because security credentials are exchanged during the QUIC handshake itself.

Deployment status: QUIC is already deployed in Chromium-based browsers. Google services (YouTube, Google Drive, etc.) run on QUIC. It is the basis of HTTP/3 and is likely the future of internet transport, potentially displacing TCP for most web traffic.

Exam Concepts

Key Definitions to Know

Internet Checksum
Error detection mechanism using one's complement addition over fixed-size message blocks. Not cryptographic — detects signal/hardware faults, not attacks. Used in both TCP and UDP.
Pseudo Header
A temporary structure (Source IP + Dest IP + Protocol + Reserved bits, all from the IP header) included in checksum computation but never transmitted. Provides end-to-end validation of critical routing fields.
Head-of-Line Blocking
When a single lost/delayed packet causes an entire TCP stream (and all multiplexed sub-streams) to stall, even when packets for other streams have already arrived.
0-RTT Handshake
QUIC's mechanism where a returning client reuses cached server credentials to begin sending encrypted data immediately, eliminating handshake round-trip latency for repeat connections.
