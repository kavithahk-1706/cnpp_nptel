Overview — The Six-Step Security Framework

Network security is not a one-time activity; it must be executed on a routine basis because both the system and the attack landscape evolve continuously. The course organises the process into six sequential steps.

Step 2 — Implementation Tools

Firewall

A firewall applies filtering rules to all packets passing through it. Three major types exist.

Packet Filter
Filters packets based on destination IP, port, or protocol. Stateless — examines each packet independently.
Stateful Firewall
Tracks ongoing TCP sessions. Ensures that out-of-session packets (packets that do not belong to an established connection) are discarded.
Application Proxy
Acts as a proxy for a specific application. Inspects all layers for malicious data — the deepest form of inspection.
Default policy for incoming packets: Drop by default. Rules are then written to permit specific traffic. Rules for outgoing packets are optional and depend on organisational policy.

Intrusion Detection System (IDS)

An IDS scans incoming messages and raises alerts when suspected scan or attack activity is detected. It operates outside the firewall (in the DMZ region or on a shared hub/broadcast domain) so that it can observe raw attack traffic before it is filtered. A widely used free IDS is Snort, available on multiple platforms. Most IDSes send alerts to log files; serious events can trigger paging, email, or telephone alerts.

Honeypot / Honeynet

A honeypot simulates a decoy host (or network) with exposed services, intentionally attracting attackers. The goal is to learn the signatures of attacks, which are then used to update the IDS knowledge base. A collection of interconnected honeypots forms a honeynet, which can be shared across organisations. Honeypots are especially valuable in large installations where understanding the attacker's methodology is strategically important.

Network Architecture (DMZ)

A typical secure internal network uses a Demilitarised Zone (DMZ) to hold publicly accessible servers (e.g., web servers) while protecting the internal network behind an additional firewall. The architecture from outer to inner is:

Internet → Router → Hub (shared domain, IDS monitors here) → Firewall → Switch → DMZ (external servers) and inner Firewall/NAT → Internal Network

The IDS is placed on the hub side of the outer firewall so it can observe all incoming attack traffic. The inner firewall protects the private internal network from both the DMZ and the internet.

Steps 3 & 4 — Reconnaissance and Vulnerability Scanning

Step 3: Reconnaissance

Before an attack (or a defensive assessment), the network must be surveyed to identify: all host IP addresses, key servers and the critical data they hold, services running on those hosts, and known vulnerabilities in those services. Reconnaissance takes two forms.

Passive Reconnaissance
Observes existing network traffic without injecting any packets. Undetectable by an IDS. Examples include traffic sniffing and examining publicly available DNS records.
Active Reconnaissance
Actively probes the target network (e.g., port scans, ping sweeps). Detectable by an IDS. Yields more precise information but risks alerting the defender.

Step 4: Vulnerability Scanning

Armed with a list of hosts and services, automated scanners probe each service for known vulnerabilities. Scanners fall into two categories.

Detection Scanners (e.g., Nessus)
Detect vulnerabilities and produce a risk report. New vulnerabilities are supported by installing plug-ins written in the Nessus Attack Scripting Language (NASL). Updatable and extensible.
Exploitation Frameworks (e.g., Metasploit)
Allow the tester to select a payload and actively exploit a detected vulnerability (e.g., obtain a root shell). The payload is the code that executes once the exploit succeeds.
Zero-day exploits are vulnerabilities for which no patch exists yet. They are particularly dangerous because even a fully up-to-date system is vulnerable until a patch is released.
Step 5 — Penetration Testing

Penetration testing (also called a tiger team attack or red team attack) is the method of evaluating the real-world strength of all security controls by simulating a malicious attacker. Its goal is to violate the site security policy — not to destroy the system, but to reveal whether it can be breached.

Penetration testing is not a replacement for careful design and structured testing. It is a supplementary methodology applied once the system is in place, and it examines both procedural/operational controls and technological controls.

Penetration Test Methodology (Cyclic Loop)

The process is iterative, not linear. Each pass through the loop can reveal new targets within the already-compromised system.

Information Gathering & Reconnaissance → Scanning & Enumeration → Vulnerability Identification → Exploitation (via exploit database) → Gain Access → Escalation & Advancement → Start to Exploit Other Targets → (loop back to Information Gathering)

Penetration Testing Tools

Proprietary Tools
Expensive; may contain backdoors; inextensible; no access to source code; not easily adaptable to novel scenarios.
Public Domain Tools
Free and generally effective, but other parties also know them. Not fully integrated or automated. Organisations often need to build custom, proprietary frameworks on top of them.

System Vulnerability

A vulnerability is a hole or weakness arising from a design flaw, an implementation bug, or an integration flaw in an application. Common types include SQL injection and buffer overflow. The formal definition of penetration testing is: the method of evaluating the vulnerability of a computer system or network by simulating the attack of a malicious hacker, carried out in a non-destructive (ethical) manner.

Steps 6 & Ongoing — Post-Attack Investigation and Vulnerability Assessment

Step 6: Post-Attack Investigation (Forensics)

After a confirmed attack, a forensic investigation must be conducted. This process is heavily guided by law and varies between organisations (academic vs. financial organisations have different obligations). Key principles include:

Retain chain of evidence: The primary evidence is the data on the compromised host. Log files hold the attacker's footsteps and fingerprints. Every minute of activity on the host must be accounted for. For legal reasons, a low-level copy of the disk must be examined — the original must not be modified. This is especially critical for commercial organisations who may face litigation.

Post-attack investigation is normally conducted by a third party to maintain objectivity and legal credibility.

Ongoing: Vulnerability Assessment

Modern enterprises are entirely IT-enabled. Therefore, proactive self-security through vulnerability assessment is mandatory, not optional. The three main components are: Content-aware Intrusion Prevention Systems (IPS), file system scanning, and penetration testing. These must be repeated on a routine basis because both the system configuration and the attack landscape change over time.

Proxy Server, NAT, and Firewall

Proxy servers, NAT, and firewalls are common network components that were not primarily designed for security, but each provides security benefits as a side effect of its primary function.

Proxy Server

A proxy server is an intermediary that acts on behalf of clients. It acts as a server when communicating with a client, and acts as a client when communicating with the real server. The most commonly used HTTP proxy is Squid, available on all platforms.

Location

The proxy sits between all user agents (browsers) and the origin server. It holds an Access Control List (ACL) defining which requests are permitted, and a cache of previously fetched responses.

Primary Purposes

Performance (Caching)
Stores results of previous requests. If the same resource is requested again, the proxy serves it from cache — conserving bandwidth and reducing latency significantly for a group of users.
Request Filtering (Access Control)
Allows or denies access based on destination URL, content strings within pages, or file type (e.g., blocking video files). Application-layer proxies can inspect packet payloads for content-based filtering.

Anonymous Proxy

An anonymous proxy hides the user's real IP address. All requests to external servers originate with the proxy's IP address instead of the client's. This is useful for group subscriptions (e.g., institutional access to IEEE journals or digital libraries), where authentication is based on the proxy's registered IP rather than individual client IPs.

Functions of an HTTP Proxy (Summary)

Request Forwarding
Primary function. Intercepts all outgoing requests; forwards to real server only if cache miss. Acts as a rudimentary firewall.
Access Control
Allows or denies access based on content and location. Application-layer proxies can open packets up to Layer 7 to inspect payload.
Cache Management
Efficient utilisation of bandwidth. Faster response for repeated requests served directly from cache.
Network Address Translator (NAT)

NAT (standardised in RFC 1631) allows a single device (a router or a dedicated NAT box) to act as the agent between a public network (the internet) and a private network (the LAN). It maps one set of IP addresses (private, non-routable) to another set (public, routable), enabling many internal hosts to share a single public IP.

Address Translation Table (ATT)

The NAT device maintains an Address Translation Table (ATT) that maps each internal host's private IP and source port to a corresponding public IP and NAT port number. Simultaneous connections from multiple internal hosts are distinguished by different port numbers.

Capability Limit

The maximum number of concurrent translations is primarily limited by the memory available to store the ATT. Each ATT entry occupies approximately 160 bits. A NAT device with 8 MB of memory can support approximately 8 × 1024 × 1024 × 8 / 160 ≈ 4,19,000 concurrent translations — sufficient for any typical organisation.

Main Uses of NAT

IP address pooling ISP migration support IP masquerading Server load balancing

When an organisation changes its ISP and thus its public IP block, internal private IP addresses remain unchanged — only the NAT mapping is updated. This greatly simplifies network administration.

Concerns About NAT

Performance Overhead
Modifying the IP header requires recalculation of the IP checksum. Modifying the port number requires recalculation of the TCP checksum. This introduces a processing overhead at every packet.
Fragmentation Issues
If a datagram is fragmented before reaching the NAT device, each fragment must not be assigned a different IP or port — all fragments of the same datagram must be treated as a unit. Handling this correctly requires additional logic.
End-to-End Connectivity
NAT destroys universal end-to-end reachability. A public internet host often cannot initiate a connection to a host inside a private network. This is a fundamental architectural violation of the internet's original design. Communication between two hosts each behind NAT requires two translation hops.
IP addresses in application payloads: Applications that embed IP addresses in their payload data (e.g., FTP active mode, SIP) do not work correctly across a NAT boundary, because the NAT only rewrites the IP header — not the payload. Some advanced NAT devices perform Application Layer Gateway (ALG) inspection to handle this, translating addresses found inside the payload as well.

Security Benefit of NAT

NAT automatically creates an implicit firewall between the internal and external networks. Since the ATT only contains entries for connections initiated from within the private network, an outside host cannot directly initiate a connection to any internal host. Inbound connections to internal hosts require configuring static NAT mappings (port forwarding).

Concept Summary
Security Policy (Step 1): A full security roadmap covering usage policies, user training, privacy, DMZ placement, firewall rules, and IDS deployment.
Firewall types: Packet filter (IP/port/protocol) → Stateful (session tracking) → Application proxy (payload inspection). Progressively more expensive and powerful.
IDS vs. Honeypot: IDS detects attacks reactively. Honeypot lures attackers to a decoy to learn their techniques proactively.
Reconnaissance: Passive (undetectable, observes traffic) vs. Active (detectable, probes the network).
Penetration testing: Ethical, non-destructive simulation of a real attack. Tests both technology and procedure. Not a substitute for good design. Also called tiger team / red team attack.
Proxy purposes: Performance (caching) + request filtering (access control) + anonymity.
NAT: Maps private IPs to a public IP using an ATT. Provides implicit inbound firewall. Breaks end-to-end connectivity. Layer 3. Proxy is Layer 4+.
