Application Layer II — DNS

▶ watch lecture
Lecture Overview

This lecture covers the Domain Name System (DNS) — the internet's global naming infrastructure. DNS solves a fundamental usability problem: humans remember names, but routers and network devices only understand IP addresses. DNS is the translator between the two. The lecture covers the hierarchical structure of the domain namespace, how DNS servers are organised and replicate data, the resolution process (recursive and iterative), and the structure of DNS messages and resource records.

What is DNS — The Problem it Solves ↗ ~01:11
DNS (Domain Name System) is a globally distributed database that maps human-readable domain names (like www.iitkgp.ac.in) to machine-readable IP addresses (like 203.16.x.x). Without DNS, every user would need to memorize raw IP addresses to navigate the internet.

The core problem: routers operate at the network layer and only understand IP addresses. Domain names exist at the application layer — purely for human convenience. DNS acts as the bridge. When you type a URL into a browser, DNS is queried first to resolve the name to an IP; only then can packets actually be routed to the destination.

Why routers can't just use names: Routers look up the network layer (IP). Names only make sense at the application level — they have no meaning below that. So the translation must happen before any packets leave your machine. DNS handles this invisibly every time you open a website.
Domain Name Space — Hierarchical Tree Structure ↗ ~04:16

Domain names are arranged in an inverted tree structure. The root is at the top (represented by a single trailing dot), branching downward through top-level domains, second-level domains, and further subdomains. Each node in the tree is called a label. A full domain name is formed by reading labels from the specific leaf up to the root, separated by dots.

Domain Name Breakdown — Reading a Domain

Each segment separated by a dot is a label. Read right-to-left for the hierarchy. For cse.iitkgp.ac.in:

subdomaincse

The fully qualified name is globally unique. Even though IIT Bombay also has a "cse", cse.iitb.ac.in ≠ cse.iitkgp.ac.in — the hierarchy ensures no clashes anywhere in the tree.

Fully Qualified Domain Names (FQDN) ↗ ~08:25
A Fully Qualified Domain Name (FQDN), also called an absolute domain name, is a domain name that ends with a trailing dot representing the root — making it completely unambiguous. A name without the trailing dot is partially qualified, meaning the DNS resolver may append a configured local suffix to complete it before querying.

Types of Top-Level Domains ↗ ~09:16
Generic TLDs (gTLD)
Three or more characters. Represent categories of organisations. Registered and maintained by ICANN (Internet Corporation for Assigned Names and Numbers — icann.org). Examples include: com (commercial), edu (education), gov (US government), org (non-profit), mil (US military), net (network providers), int (international), aero, museum, biz, info, jobs.
Country Code TLDs (ccTLD)
Exactly two characters. Based on ISO 3166 international country codes. Every country gets one. Examples: in (India), us (USA), uk (British/Irish), au (Australia), de (Germany), jp (Japan), fr (French). Countries often define their own second-level domains underneath — e.g. India uses ac.in (academic) and co.in (commercial) to mirror generic TLD categories.
Zones vs Domains ↗ ~12:27
These two terms look similar but are distinct. A domain is the complete set of all names and machines under an organisational domain name — including everything delegated away to sub-servers. A zone is that domain minus any sub-domains that have been delegated out to other DNS servers. Think of it as: zone = what this particular DNS server is directly responsible for.

A DNS server gives an authoritative answer for records within its own zone. If it answers about something it received from another server (via zone transfer), that is a non-authoritative answer.

Primary & Secondary Servers — Zone Transfer ↗ ~13:57

Every domain is typically served by two or more DNS servers for redundancy — if one fails, others can still answer queries. The setup follows a primary/secondary model:

primary server
Master copy

Holds the authoritative master copy of zone data. Loads from its own local disk. Only one primary is allowed per zone.

secondary server(s)
Replica copies

Gets its data from the primary via zone transfer. Multiple secondary servers are possible. Handles queries when primary is unreachable.

Zone Transfer: The synchronisation mechanism between primary and secondary. The primary loads zone data from disk. The secondary loads all zone data from the primary. When the primary downloads/re-syncs information from a secondary, this process is specifically called a zone transfer.
DNS in the Internet — Three Domain Categories ↗ ~15:19

The entire DNS namespace under the root is divided into three broad categories of domains:

Generic Domains
gTLDs — 3+ character names organised by category of organisation. com, edu, gov, org, mil, net, int etc. Geography-agnostic.
Country Domains
ccTLDs — 2-character ISO country codes. Every country gets its own TLD. in, us, uk, au, de, jp etc. Geographically organised.
Inverse Domain
Reverse DNS — maps IP addresses back to domain names. Lives under the in-addr.arpa special namespace. IP octets written in reverse order as labels.

Inverse Domain — Reverse DNS Resolution

Normal DNS is forward resolution: name → IP. Inverse DNS is reverse resolution: IP → name. A common use case is authentication — an FTP server receives a connection from IP 153.2.7.9 and wants to verify whether that IP belongs to an authorised client. It performs a reverse DNS lookup to find the domain name associated with that IP.

The inverse domain uses the in-addr.arpa special namespace. IP octets are written in reverse order as tree labels. IP 132.34.45.121 becomes the query: 121.45.34.132.in-addr.arpa. — and the DNS server returns the domain name that owns that IP.

Name Resolution — BIND & nslookup ↗ ~17:25
The most widely used DNS server software is BIND (Berkeley Internet Name Domain). It runs on UNIX/Linux as a background daemon process called named. When any application needs to resolve a domain name, it invokes the DNS name resolver, which translates an FQDN into the corresponding IP address.
command-line tool
nslookup

Manually query a DNS server from the terminal. Type a domain name and it returns the resolved IP. Useful for testing DNS resolution and debugging.

BIND daemon
named

The BIND process running in the background on a DNS server. Handles incoming queries, consults zone data and cache, and returns responses. If it doesn't know locally, it asks its primary server.

Resolution Methods — Recursive vs Iterative ↗ ~20:04

Recursive Resolution

The client asks a local DNS server and that server takes full responsibility for finding the answer — querying root, TLD, and authoritative servers on the client's behalf. Only one response comes back to the client: the final answer.

Local DNS does all the legwork. Client sends one query, gets one final answer. The local resolver follows the chain: root → TLD → authoritative, then hands the IP back.

Iterative Resolution

In iterative resolution, if a DNS server doesn't know the answer, it returns a referral — the address of the next DNS server the client should ask — rather than querying that server itself. The client is responsible for following the chain of referrals one by one. Multiple responses come back to the client, each pointing to the next server to query.

Full Resolver vs Stub Resolver ↗ ~21:04
Full Resolver
A standalone DNS resolver process separate from the user application. The user program sends its query to the full resolver, which contacts name servers, follows referrals, maintains its own cache of recent resolutions, and returns a final answer. If a local name server doesn't know, the full resolver contacts a foreign name server itself. Decoupled from the application but an extra step in the chain.
Stub Resolver commonly used
A small routine directly linked into the user program itself. Forwards queries directly to a name server — no separate resolver process needed. Implemented via two standard library functions: gethostbyname() (name → IP) and gethostbyaddr() (IP → name). Faster, simpler, and the standard approach on modern platforms. The resolver logic lives inside the application.
DNS Resource Records (RR) ↗ ~22:50
The DNS distributed database is built from Resource Records (RRs). Each RR maps a domain name to a network object. The most common mapping is domain name → IP address, but RRs can map to mail servers, name servers, aliases, and other objects. A zone's complete RR database is what a DNS server is authoritative for.

Resource Record Fields

field 1
Name

The domain name this record belongs to

field 2
Type

A = address (name→IP), NS = name server, MX = mail server, PTR = pointer/inverse, SOA = start of authority

field 3
Class

Network class. IN = internet in almost every real-world case

field 4
TTL

Time To Live — how long this record may be cached before being considered stale and re-fetched

field 5
RDlength

Length in bytes of the RData field that follows

field 6
RData

The actual record payload — e.g. the 4-byte IP address for an A record

A zone's resource records always start with a SOA (Start of Authority) record identifying the zone, followed by NS (Name Server) records identifying which servers are authoritative. The NS records specifically determine whether a server's answers are authoritative or not for that zone.
