IP Addressing (IPv4) III — Network Address Translation (NAT)

▶ watch lecture
Lecture Overview

This lecture addresses a fundamental problem with IPv4: the address space is exhausted. The proposed and widely-deployed solution is Network Address Translation (NAT) — a mechanism that allows a large number of devices in a private network to share a small pool of public IP addresses. The lecture covers the motivation, the basic operation of NAT, IP masquerading (port-based NAT / PNAT), use cases including ISP migration and load balancing, and the key limitation of NAT around inbound connections.

The Problem — IPv4 Address Exhaustion ↗ ~00:45
Core problem: IPv4 is 32 bits, giving a theoretical maximum of 232 addresses. In practice, the usable address space is significantly smaller, and the number of devices requiring IP addresses has grown exponentially since IPv4 was designed.

Why the usable address space is smaller than 232

The 32-bit space is divided into classes A, B, C, D, and E. Class D is reserved for multicast (rarely used in practice, so largely wasted). Class E is reserved and cannot be used for normal data transfer at all. Within classes A, B, and C, every individual network consumes at least two addresses that cannot be assigned to hosts — the network address and the broadcast address. This further erodes the pool. Additionally, modern devices often have multiple network interfaces, each requiring its own IP address.

Private IP Address Ranges ↗ ~08:33

Three blocks of IPv4 address space are designated as private addresses. These are reusable — the same private address block can exist inside any number of separate organisations simultaneously. They are never routed on the public internet.

Class Private Range Notes
Class A 10.0.0.0 — 10.255.255.255 Single large block. Your home/office WiFi is almost certainly in this range.
Class B 172.16.0.0 — 172.32.255.255 16 contiguous Class B networks.
Class C 192.168.0.0 — 192.168.255.255 256 Class C networks. Most common in home routers.
The reusability principle: Two Sandip Chakrabortys at IIT KGP and IIT Bombay can coexist because the context (which institute) disambiguates them. Similarly, 10.0.1.2 at IIT KGP and 10.0.1.2 at Stanford are completely different machines — the organisation they belong to is the disambiguating context. Private IPs are only meaningful within their own network boundary.
Basic NAT Operation ↗ ~11:45
NAT device sits at the boundary between a private network and the public internet. It maintains a NAT table — a mapping between private IP addresses and the public IPs assigned to them. On the way out, it replaces the private source IP with a public one. On the way back, it reverses the substitution.

Outgoing packet flow (inside → outside)

Step 1 → Internal machine (10.0.1.2) sends packet with src=10.0.1.2, dst=213.168.112.3
Step 2 → Packet arrives at NAT device. NAT selects a public IP from its pool (e.g. 128.143.71.21)
Step 3 → NAT records mapping: 10.0.1.2 ↔ 128.143.71.21 in NAT table
Step 4 → NAT replaces src IP in packet header: src becomes 128.143.71.21. Forwards to public internet.

Incoming reply flow (outside → inside)

Step 5 → Destination machine (213.168.112.3) replies. Sets dst=128.143.71.21 (the public NAT IP it saw as source)
Step 6 → Reply arrives at NAT device. NAT looks up 128.143.71.21 in NAT table → finds mapping to 10.0.1.2
Step 7 → NAT replaces dst IP: 128.143.71.21 → 10.0.1.2. Forwards into private network.
Step 8 → Internal network delivers packet to 10.0.1.2. Machine receives the reply.
The destination machine on the public internet never sees the private IP — it only ever interacts with the NAT's public IP. The internal network topology is completely hidden. This is a secondary but important security benefit: the internal structure of an organisation is invisible to the outside world.
IP Masquerading — Port-Based NAT (PNAT) ↗ ~19:12

Basic NAT maps one private IP to one public IP. This still requires a pool of public IPs proportional to the number of simultaneous users. IP masquerading (also called Port-based NAT or PNAT) takes this further — it allows a single public IP to serve multiple private hosts simultaneously by using the port number as an additional discriminator.

The key insight: network communication is actually process-to-process. A process is identified not just by IP address but by the (IP, port) pair. So two different internal machines can share one public IP as long as they use different ports — the port alone is enough to distinguish their traffic on the way back.

PNAT example — two machines, one public IP

Machine A → Private: 10.0.1.2 : port 2001 → mapped to → Public: 128.143.71.21 : port 2100
Machine B → Private: 10.0.1.3 : port 3020 → mapped to → Public: 128.143.71.21 : port 4444

Both machines share the same public IP 128.143.71.21. When a reply arrives at the NAT device addressed to port 2100, the NAT table resolves it to machine A at 10.0.1.2:2001. A reply to port 4444 goes to machine B at 10.0.1.3:3020. The port number is the differentiator.

Port numbers are 16-bit, giving ~65,000 values. Even after subtracting reserved/well-known ports, you have tens of thousands of usable port numbers. With PNAT, a single public IP can theoretically serve that many simultaneous connections. This is why home routers work — your entire household shares one public IP through PNAT.

NAT table structure in PNAT

Private IP : Port Public IP : Port Direction
10.0.1.2 : 2001 128.143.71.21 : 2100 outgoing / reverse lookup
10.0.1.3 : 3020 128.143.71.21 : 4444 outgoing / reverse lookup
NAT Use Cases ↗ ~16:48

1. Address Conservation (Primary Use Case)

Large numbers of private hosts share a small pool of public IPs. Only actively-connected machines consume a public IP at any moment. Machines that are idle do not hold a public IP at all.

2. ISP Migration ↗ ~16:48

An organisation can connect to multiple ISPs for reliability (called a multi-homed network). Without NAT, switching ISPs requires reconfiguring the IP address of every internal machine to reflect the new ISP's network prefix. With NAT, only the NAT box's public IP pool changes — all internal machines keep their private IPs unchanged.

IIT KGP is connected to both ERNET and NKN. If ERNET goes down, the NAT device switches its public IP pool to NKN addresses. Internal machines at 10.x.x.x are completely unaffected — zero reconfiguration required.

3. Load Balancing of Servers ↗ ~23:31

Multiple identical servers behind NAT can all be reachable via a single public IP. The NAT device distributes incoming connections across the internal servers based on load. External clients see one IP; internally, the traffic fans out across a cluster. This is conceptually the same as what a reverse proxy / load balancer does.

Example: public IP 128.143.71.21 → NAT distributes requests between web server at 10.0.1.2 and web server at 10.0.1.3. Neither server's private address is ever exposed externally.
NAT Limitation — Inbound Connections ↗ ~23:57
Critical limitation: NAT is fundamentally outbound-initiated. A machine inside the private network can always initiate a connection outward — the NAT creates the mapping dynamically. But an external machine cannot initiate a connection to a private host because it has no way to know the private IP — it can only see the NAT box's public IP, and there is no pre-existing mapping until the internal machine speaks first.

Why this matters

If you want to run a server (web server, game server, SSH endpoint) inside a private network that accepts connections from the public internet, NAT blocks this by default. The outside world can only reach the NAT box's public IP — it has no knowledge of what's behind it. The NAT table has no entry until an outbound connection creates one.

Workarounds

DNS + Static NAT Mapping
Map a DNS name (e.g. www.iitkgp.ac.in) to the NAT box's public IP. Configure a static entry in the NAT table that always forwards requests arriving on a specific port to a specific internal machine. The external world resolves the DNS name, gets the NAT's public IP, and the static mapping handles forwarding. Used for public-facing web servers behind NAT.
Internal Machine Initiates First
If the internal machine establishes an outbound connection first, the NAT creates a mapping entry. The external machine can then use the source IP from that first packet (the NAT's public IP) as a destination for subsequent communication. Works only when there is a mechanism for the internal machine to "announce" itself.
Real-world consequence: This is why your home router (which is NAT) means you can browse freely but people can't connect inward to your machine by default. Port forwarding in router settings = manually adding a static NAT mapping for a specific port.
Concept Summary
why NAT exists
IPv4 address exhaustion

Not enough public IPs for every device. Classes D and E wasted. Network + broadcast addresses eat into each subnet. Device count exploded.

NAT mechanism
Private ↔ Public IP translation at boundary

NAT table maps private IPs to public IPs. Outgoing: replace src. Incoming: reverse lookup, replace dst.

PNAT / masquerading
One public IP → many private hosts

NAT table maps (private IP : port) ↔ (public IP : port). Port number disambiguates multiple hosts sharing one public IP.

private ranges
10.x.x.x / 172.16–32.x.x / 192.168.x.x

Never routed on the public internet. Reusable across any number of organisations simultaneously.

use cases
Conservation, ISP migration, load balancing

Multiple public benefits beyond just conserving addresses. ISP migration = change pool only. Load balancing = one public IP fans out to server cluster.

limitation
Inbound connections blocked by default

No pre-existing NAT entry = no path in. External hosts cannot initiate connections to private hosts. Requires static mappings or DNS workarounds.
