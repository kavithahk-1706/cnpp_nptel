Lecture 16 ⚡ HIGH PRIORITY

Transport Layer Performance

How BDP drives window size decisions, protocol selection, and the mechanics of how the application layer actually hands data to the transport layer — including buffer design.

▶ Watch Lecture
Quick Recap

Last lecture covered connection establishment and ARQ protocols for flow control and reliability — Stop and Wait, Go Back N, and Selective Repeat. This lecture is all about performance implications: what parameters should you tune, and how do you even pick the right protocol in the first place?

Part 1 — Bandwidth Delay Product
01:20

What is BDP?

Bandwidth Delay Product (BDP) = Link Bandwidth × Link (one-way) Delay. It tells you how much data can be "in flight" in the network at any given time — think of it as the volume of the pipe between sender and receiver.

Pipe analogy: Latency = the length of the pipe. Bandwidth = the cross-sectional area of the pipe. BDP = bandwidth × latency = the total volume of data the pipe can hold.

Worked Example

Parameter Value
Link Bandwidth50 Kbps
One-way Delay250 ms
BDP50 Kbps × 250 ms = 12.5 Kbit
Segment Size (assumed)1000 bits
BDP in segments12.5 segments

Round Trip Time (RTT)

When you send a segment and wait for its ACK, the total wait is one Round Trip Time (RTT). RTT = 2 × one-way latency. So in the example above, RTT covers the time to transmit data to receiver and get the ACK back.

During one RTT, the maximum segments that can be outstanding (unacknowledged) in the network = 2 × BDP in segments = 2 × 12.5 = 25 segments.

Why 25 and not 12.5? There are two pipes in a two-way communication — one carrying data from sender to receiver (12.5 segments), and one carrying ACKs back (those ACKs correspond to 12.5 segments sitting in the receiver buffer). Total outstanding = 12.5 + 12.5 = 25.

Part 2 — BDP Implication on Window Size
12:16

Optimal Sender Window Size

Let BD = number of segments equivalent to the BDP (i.e., BDP expressed as a segment count). The sender window size (swnd) that gives maximum link utilization is:

w = 2BD + 1

The +1 accounts for the ACK of the first segment that just arrived at the sender — the link is filled with 2BD segments (data pipe + ACK pipe) and one acknowledgement has just been received but not yet processed. Setting the window to exactly this value ensures the link is always busy transmitting data — maximum utilization with no idle time.

Connecting Window Size to Sequence Number Space

Once you fix w from BDP, you can derive the required number of sequence number bits n. The constraints from the ARQ protocols are:

Protocol Max Window Size Derive n from w
Go Back N 2n−1 Set 2n−1 ≥ w → solve for n
Selective Repeat 2n / 2 Set 2n/2 ≥ w → solve for n

This is the practical workflow: compute BDP → compute optimal w = 2BD + 1 → figure out minimum bits needed for your sequence number field given the ARQ protocol you chose.

Part 3 — BDP and Protocol Design Choice
13:43

The Protocol Selection Problem

BDP doesn't just tell you the window size — it tells you which class of protocol to even bother with. The key comparison is: segment size vs. BDP.

Case Study: When Sliding Window is Pointless

ParameterValue
Link Bandwidth1 Mbps
One-way Delay1 ms
BDP1 Mbps × 1 ms = 1 Kbit = 128 bytes
Segment Size1 KB = 1024 bytes
Segment vs BDP1024 bytes >> 128 bytes (8× larger)
Result: The segment is 8× bigger than the pipe can hold. Even one segment can't fully fit in the link. Sending multiple segments in parallel provides zero advantage — you can't exploit parallelism if you can't even saturate the pipe with one segment. Sliding window overhead (sender buffer, receiver buffer, sequence numbers) is all cost and no benefit here.
Sliding Window ARQ
Higher overhead: must maintain sender window, receiver window, sequence number field
Only useful when segment size < BDP (parallelism helps)
Wrong choice when segment >> BDP
Stop and Wait
Minimal overhead: no sender/receiver windows, trivial sequence number (1 bit)
Send one segment, wait for ACK, repeat
Correct choice when segment >> BDP
Decision rule: If segment size > BDP → use Stop and Wait. If segment size << BDP → use sliding window and set w = 2BD + 1 for max utilization.

Key Design Points

The application may generate data at 10 Mbps while the Transmission Rate Control module determines the optimal send rate as only 2 Mbps (dictated by the flow control algorithm and current window size). These rates are asynchronous — there must be an intermediate buffer to absorb the difference.

Each connection gets its own connection-specific source buffer (Transport Buffer - Sender) because different connections are treated independently — they may have different flow control states, different receivers, different window sizes.

write() is a blocking call. The application calls write()/send() to push data into the transport buffer. This call blocks the port until the complete data is written into the transport buffer. If the buffer fills up (app is generating faster than TRC can drain it), the transport layer blocks the application from writing further — preventing buffer overflow.
Part 5 — Application–Transport Interfacing: Receiver Side
21:39

How the Receiver Side Works

APPLICATION (User Space)
read() / recv()
Transport Buffer - Receiver (Kernel)
per-port queue
CheckBuffer()
sends interrupt when data arrives
TportRecv() ← Data from IP
routes to correct port queue

TportRecv() receives a segment from IP, reads the port number in the transport header, and deposits the data into the correct buffer for that port (each port = one application). CheckBuffer() watches the buffer and fires an interrupt when data becomes available.

read()/recv() is also a blocking call. If the buffer is empty when the application calls read(), the call blocks at the port and waits. The moment data arrives, CheckBuffer() sends an interrupt, the call unblocks, reads the full data from the buffer, and returns to the application.
Symmetry summary: Sender's write() blocks → when buffer is full. Receiver's read() blocks → when buffer is empty.
Part 6 — Organizing the Transport Buffer Pool
24:04

Three Ways to Organize Buffers

The transport layer buffer is a software buffer (in kernel memory). How you organize it matters for memory efficiency and implementation complexity.

(a) Pool of Fixed-Size Buffers
One buffer = one segment. Buffer size = segment size. Simple and fast to allocate/deallocate. Works perfectly when all segments are the same size. Wastes space if segment sizes are widely varied — a small segment wastes most of its fixed-size slot.
(b) Chained Fixed-Size Buffers
Buffers are linked-list nodes of fixed size. Variable-sized segments are spread across multiple chained nodes (like a linked list). More flexible, but space is still wasted if sizes vary widely. Using very small buffers means needing many nodes per segment — adds implementation complexity.
(c) Variable-Size Buffers
Buffer size matches segment size dynamically. Best memory utilization — no wasted space. Disadvantage: dynamic memory allocation is complex; buffer space management must handle arbitrary sizes. Implementation is significantly harder.

Single Large Circular Buffer (Preferred for Heavy Load)

25:41

One large circular buffer per connection. Segments of different sizes are written sequentially into the circular space. Unused space at the tail is available for the next incoming segment. This is a good choice when connections are heavily loaded — the circular structure means memory is reused efficiently without fragmentation.

Practical note: The right buffer design depends on the application. Fixed-size is simple and fast. Variable-size is memory-efficient. Circular buffer is great under heavy sustained load. No universally correct answer — it's a tradeoff.

Interfacing Summary

The application and transport layer interact through blocking system calls and per-connection buffers. write()/send() blocks when the sender buffer is full; read()/recv() blocks when the receiver buffer is empty. A Transmission Rate Control module sits in kernel space and paces the actual sending rate based on the flow control algorithm's window size decision.
