Lecture 17 ★ High Priority

Buffer Management & Congestion Control

Transport layer buffer dynamics, congestion detection, fairness, and the AIMD algorithm. Core TCP behaviour is rooted in this lecture — heavily examinable.

Part 1 — Dynamic Buffer Management

The Rate Mismatch Problem

At the receiver, the transport layer buffer sits between two processes running at independent rates: the network layer delivering incoming segments and the application consuming data via read() system calls. These rates are rarely equal.

If the network delivers at 1 Mbps but the application reads at only 10 Kbps, data accumulates in the transport buffer. Once the buffer is full, any further arriving segment must be dropped — even though the receiver machine is alive and the connection is intact. The solution is to prevent the sender from over-sending in the first place.

Key insight: The sender must dynamically track how much buffer space is actually free at the receiver and limit its window to that value. This is flow control driven by buffer availability, not just ACK timing.

Mechanism — Receiver Advertises Available Space

In window-based (sliding window) flow control, the receiver piggybacks its current available buffer size onto every ACK it sends. The sender then sets its sending window to at most that advertised value. As the application reads data and frees buffer space, the receiver sends updated (larger) advertisements.

Formula: Sender window size ≤ Receiver advertised buffer space

The Deadlock Problem

When B sends buf=0, sender A stops transmitting and waits for a non-zero advertisement. If that follow-up advertisement is lost in the network, A waits indefinitely — a potential deadlock.

Deadlock prevention: B must keep sending duplicate ACKs (with current buf value) after every timeout, even if it has received no new data from A. This ensures the connection recovers even if an advertisement ACK is dropped. ACKs must flow continuously.
Part 2 — Congestion Control

Centralised Baseline: Max-Flow Min-Cut 14:41

In a hypothetical centralised network, every link has a known capacity. To find the maximum achievable throughput between a source S and destination D, one applies the max-flow min-cut theorem: the bottleneck is the minimum cut across all paths. For example, a network whose minimum cut sums to 12 Mbps can never sustain more than 12 Mbps S→D regardless of path choices.

This gives an ideal upper bound. The difficulty is that the real internet is fully decentralised — no router has global network visibility, and flows enter and leave dynamically. Computing a min-cut on-the-fly is impractical.

Congestion Signature: Buffer Overflow 20:59

At any intermediate router, let λ = total incoming rate and μ = total outgoing capacity. If λ > μ the buffer fills over time. Once full, arriving packets are dropped. In wired networks where channel-error losses are negligible, packet loss ≡ congestion signal.

Congestion Control
Prevent congestion from occurring — requires global state. Not feasible at internet scale.
Congestion Avoidance
Detect congestion when it happens and react by reducing rate. Feasible in a decentralised setting. This is what the internet uses.

Revised Sending Rate Formula

Previously, the sender's rate was bounded only by the receiver's flow control window (receiver rate). With congestion avoidance added:

Sending rate = min(network rate, receiver rate)
Where receiver rate comes from the advertised window (flow control), and network rate is the sender's estimate of what the network can sustain — probed incrementally.

Since network rate cannot be known directly, the approach is: gradually increase the rate until packet loss is observed, then reduce, then increase again. The network rate is discovered empirically.

Part 3 — Fairness

Why Fairness Matters 26:49

Congestion control algorithms must be fair — otherwise some flows get starved while others monopolise the bottleneck. In a decentralised network, computing strict (hard) fairness requires full global information — again impractical. Instead, TCP targets max-min fairness.

Max-Min Fairness — Definition

An allocation {λ₁, λ₂, …, λₙ} is max-min fair if the rate of any flow cannot be increased without decreasing the rate of at least one other flow that already has an equal or smaller allocation.

Informally: you cannot make any flow better off without making some other flow worse off. It maximises the minimum allocation — it is fair to the weakest flows first.

Max-Min Fair Allocation — Example 28:46

Consider 4 flows (A, B, C, D) sharing a network with a single bottleneck link shared by all three of B, C, D simultaneously.

Flow Bottleneck link shared with Max-min fair allocation Reasoning
B C, D (3 flows on worst link) 1/3 of capacity Bottleneck link split equally among 3
C B, D 1/3 of capacity Same bottleneck
D B, C 1/3 of capacity Same bottleneck
A B only (on its non-bottleneck link) 2/3 of capacity B used 1/3; remaining 2/3 available for A alone

This allocation is max-min fair: to give B more than 1/3 you must take from A; to give C more you must take from B or D. No improvement is possible without harming another flow.

Part 4 — AIMD Algorithm

Additive Increase Multiplicative Decrease 30:36

Proposed by Chiu and Jain (1989). AIMD is the distributed algorithm that achieves convergence to max-min fairness without any centralised coordination.

Let w(t) = sending rate at time t, a = additive increment (a > 0), b = multiplicative factor (0 < b < 1).

No congestion detected: w(t+1) = w(t) + a   (increase additively)
Congestion detected: w(t+1) = b · w(t)   (decrease multiplicatively)

Why Not AIAD or MIMD? 31:54

The AIMD design rationale is best understood by comparing all four variants on a two-user bandwidth-allocation plane. The axes represent user 1's bandwidth and user 2's bandwidth respectively. Two reference lines matter:

Fairness Line
The 45° diagonal — points where both users receive equal bandwidth. The optimal point lies on this line and on the efficiency line simultaneously.
Efficiency Line
The line where the sum of both allocations equals 100% of link capacity. Points below = under-utilised; points above = congestion.

📊 The AIMD Design Rationale diagram (two flows example) appears at timestamps 31:54 and 34:04. The convergence path — zigzagging diagonally toward the optimal intersection — is best understood visually. Both slides are worth studying carefully.

Algorithm On increase On congestion Trajectory behaviour Converges to optimal?
AIAD +a −a Parallel moves — oscillates along efficiency line ✗ No
MIMD ×c (c>1) ×b (b<1) Radial moves — oscillates on efficiency line (different slope) ✗ No
AIMD +a ×b Diagonal increase (45°), radial decrease (toward origin) — spiral inward toward optimal ✓ Yes
Geometric intuition: Additive increase moves both users' allocations by the same absolute amount (+a each), so it travels at 45° — toward the fairness line. Multiplicative decrease scales both down toward the origin, which also trends toward the fairness line. The combination creates a convergent spiral toward the intersection of the fairness line and efficiency line — the globally optimal point.

AIMD in TCP

TCP uses AIMD to control the size of its congestion window (cwnd) — a second cap on the sender window alongside the receiver-advertised window. The effective sender window is min(cwnd, recv_buf). The AIMD mechanism described here forms the foundation for TCP's slow start and congestion avoidance phases, which are covered in detail in later lectures.

Concept Summary

Dynamic Buffer Management

The transport-layer buffer at the receiver can fill up if the application reads slower than the network delivers. The receiver solves this by advertising its current free buffer space in every ACK. The sender window is capped at the advertised value. If the advertisement drops to 0, the sender blocks; if the follow-up non-zero advertisement is lost, a deadlock results. Prevention: receiver must retransmit buffer advertisements on timeout even without new data.

Congestion Avoidance

The internet cannot enforce hard rate limits from the centre. Instead, each end-host probes the network incrementally, treating packet loss as a congestion signal (λ > μ at some intermediate buffer). The sending rate is constrained to min(network rate, receiver rate). Network rate is estimated by gradual increase until loss, then reduction — an empirical control loop.

Congestion Collapse

Beyond the congestion onset point, goodput drops and delay spikes. Retransmissions of dropped packets compete with new data, further worsening congestion. This is the congestion collapse regime and motivates aggressive early detection.

Max-Min Fairness

A feasible fairness target for distributed networks: an allocation where no flow's rate can be raised without lowering a flow that is already at an equal or lesser allocation. It prioritises the worst-off flows and distributes remaining capacity upward.

AIMD

The mechanism that achieves max-min fairness in a decentralised setting. Rate increases additively (small fixed steps) and decreases multiplicatively (large proportional cut) on congestion. The geometric effect is convergence to the intersection of the fairness line and efficiency line — the globally optimal allocation. TCP uses AIMD to manage its congestion window.
