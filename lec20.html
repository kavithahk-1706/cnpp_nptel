Connection Establishment

Three-Way Handshake

TCP is a bidirectional protocol. To set up a connection between Host A and Host B, both directions must be independently initialised, which requires each side to advertise its own Initial Sequence Number (ISN). The three-way handshake achieves this in three messages.

1 A → B → SYN (seq = x)  ·  SYN flag set. A proposes ISN = x for the A→B stream.
2 B → A → SYN + ACK (seq = y, ack = x+1)  ·  B acknowledges A's SYN and proposes ISN = y for the B→A stream. Both flags set — piggybacked in a single segment.
3 A → B → ACK (seq = x+1, ack = y+1)  ·  A acknowledges B's SYN. Connection is now ESTABLISHED on both sides.
Why three messages and not two? A two-way exchange would only confirm one direction. TCP needs each side to both send and receive an acknowledgement for its own SYN — confirming bidirectional reachability. The SYN+ACK in step 2 is a single piggyback segment that simultaneously acknowledges A's SYN and initiates B's own synchronisation.

After step 3, Host A has confirmed that B acknowledged its ISN (x), and Host B has confirmed that A acknowledged its ISN (y). Both sides can now perform independent sequence-number tracking.

Initial Sequence Number Selection

The Problem: Avoiding Delayed Duplicates

The ISN must be chosen so that packets from a previous, crashed connection are not confused with packets from a newly established connection on the same port pair. Specifically, the new ISN must fall outside the forbidden region of the previous connection's sequence number space.

Two strategies exist to ensure non-overlapping forbidden regions:

Strategy 1 — Time Domain
Wait long enough after a crash before restarting, so all in-flight packets from the old connection have expired (TTL-based die-off). The new connection starts in a safe region automatically.
Strategy 2 — Sequence Number Domain
Choose the new ISN high enough above the last sequence number used by the old connection so that the two forbidden regions do not overlap. TCP uses this strategy.

Original TCP: Clock-Based ISN

The original TCP specification used a hardware clock to generate the ISN:

Clock ticked every 4 microseconds. The ISN was set to the current clock value (range: 0 to 232−1, matching TCP's 32-bit sequence number space). After a crash, the elapsed time means the clock has advanced, producing an ISN that is automatically offset from the previous connection's sequence range.

Problem: SYN Flood Attack (DoS)

The clock-based approach makes ISN generation deterministic. An attacker who observes several connections can compute the clock tick rate and predict future ISNs. This enables the SYN flood attack:

SYN Flood / Denial of Service: The attacker continuously sends spoofed SYN segments with predicted sequence numbers. The victim server accepts each as a legitimate connection request, allocates resources (half-open connection state), and waits for the final ACK that never arrives. Flooded with thousands of such requests from multiple machines, the server's connection queue exhausts — it can no longer process real connections. This is a classic Distributed Denial of Service (DDoS) vector.

Modern TCP: Cryptographic ISN Generation

Current TCP implementations apply a cryptographic hash function on top of the clock value:

1 Clock generates a base value x (ensures non-overlap with previous connection's forbidden region).
2 A cryptographic function transforms x into y, where y > x (preserving the non-overlap guarantee).
3 y is used as the actual ISN. Because y is cryptographically derived, an attacker cannot predict it from observed ISNs, defeating SYN flood prediction.
The cryptographic ISN satisfies both requirements simultaneously: it is high enough above the old forbidden region (safety), and it is unpredictable to an external observer (security).
Connection Release

Four-Step Graceful Release (FIN-based)

TCP connection release is symmetric: each direction of the bidirectional connection is closed independently using a FIN segment. Recalling the Two Army Problem, a perfectly simultaneous close over an unreliable channel is impossible — TCP therefore uses a timeout-assisted approach at the active-close side to tolerate packet reordering and loss.

1 A → B → FIN (seq = n)  ·  Host A initiates active close. A will send no more data, but can still receive.
2 B → A → ACK (ack = n+1)  ·  Host B acknowledges the FIN. B enters passive close but may still send data. The A→B direction is now closed.
3 B → A → FIN (seq = m)  ·  When B is ready to close the B→A direction, it sends its own FIN. Steps 2 and 3 may be combined into a single FIN+ACK if B is ready immediately.
4 A → B → ACK (ack = m+1)  ·  A acknowledges B's FIN, then enters TIME_WAIT and starts a timeout. After the timeout, A fully closes.

Why TIME_WAIT only on the Active-Close Side?

Active Close (Host A)
A initiated the closure but the channel is unreliable — packets may be reordered. A may still receive data from B that was in-flight before B's FIN arrived. The timeout window allows A to continue receiving such stragglers. After the timeout, A discards any further incoming segments.
Passive Close (Host B)
B receives the final ACK from A. At this point B knows that A has already sent its FIN — A will not send any more data. There is nothing left to receive, so no timeout is needed. B closes immediately upon receiving A's last ACK.

Special Cases

Simultaneous Open

Both client and server call connect() at the same time and send SYN simultaneously. Neither receives a SYN+ACK first — both receive a plain SYN. Both independently transition to SYN_RCVD, send SYN+ACK, and upon receiving each other's SYN+ACK transition to ESTABLISHED. This is an unusual but valid scenario handled by the state machine.

Simultaneous Close

Both sides call close() simultaneously and send FIN to each other without first receiving an ACK. Each side, upon receiving the other's FIN, enters the CLOSING state (rather than FIN_WAIT_2). After receiving the final ACK, both sides transition to TIME_WAIT and then to CLOSED.

SYN_RCVD Reset

From SYN_RCVD, the server may issue a RST (reset) to return to the LISTEN state, discarding the half-open connection. This is a defensive measure against SYN flood attacks — the server abandons the pending connection without fully processing it.

Key exam point: TIME_WAIT is exclusive to the active-close side. The passive-close side (LAST_ACK → CLOSED) has no timeout, because once it receives the final ACK, it knows the active-close peer has shut down and will send no more data.
Concept Summary

Three-Way Handshake

SYN (ISN=x) → SYN+ACK (ISN=y, ack=x+1) → ACK (ack=y+1). Three messages are required because TCP is bidirectional; each direction needs an independent SYN acknowledged by the other side.

ISN Selection

Original TCP used a 4 µs clock cycle (range 0–232−1) to produce a monotonically increasing ISN, ensuring the new connection's forbidden region does not overlap with the previous one's. This was vulnerable to SYN flood attacks because the ISN was predictable. Modern TCP applies a cryptographic hash on top of the clock value, preserving the forbidden-region guarantee while making the ISN unpredictable.

Connection Release

Graceful release requires four messages (FIN → ACK → FIN → ACK) because the two directions are closed independently. Steps 2 and 3 may be merged into FIN+ACK. The active-close side enters TIME_WAIT after the last ACK to absorb any straggler segments. The passive-close side does not need a timeout because it knows the active peer will send no further data.

TCP State Machine

Eleven states in total. The three critical ones for exam purposes: ESTABLISHED (normal data transfer), TIME_WAIT (active-close timeout — client only, prevents data loss from delayed packets), and CLOSING (simultaneous close scenario). The notation on every arc is event / action; dashed = server path, solid = client path.
