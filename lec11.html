Lecture 11 HIGH PRIORITY

Transport Layer — 1: Services

Prof. Sandip Chakraborty · IIT Kharagpur · CNPP

Introduces the transport layer — what it is, where it lives in the protocol stack, why it exists on top of an unreliable network layer, and the complete set of services it provides. Lays the groundwork for TCP vs UDP.

1 · Position in the Stack

The TCP/IP stack has 5 layers. End hosts run all 5; intermediate devices only run lower layers:

Application
end hosts only
Transport
end hosts only
Network (IP)
routers / L3 switches
Data Link
L2 switches
Physical
all nodes
The transport layer exists only at the two communicating end hosts. Intermediate routers have no transport layer — they only see up to Layer 3 (the IP header). This is why transport layer services are described as end-to-end.

The transport layer acts as the interface between user applications and the operating system. Data from the application passes into the OS through the transport layer.

2 · Physical Implementation Inside a Host

Inside an individual computer, the layers are implemented across three tiers:

Hardware (NIC)
Physical Layer

Network Interface Card (NIC) handles all physical layer functions. In wireless networks, parts of the data link layer are also pushed into hardware for speed and robustness.

Firmware / Device Driver
Data Link Layer (MAC)

The driver installed for the NIC implements the MAC sub-layer. The upper part of the data link layer (Logical Link Control / LLC) sits in the OS kernel.

OS Kernel
LLC · Network · Transport

In UNIX-type OSes, the kernel implements LLC, the full IP layer, and both TCP and UDP transport layer protocols.

User Space
Application Layer

Browser, chat apps, YouTube, Facebook — all application layer software. Designed by application developers, runs on top of the kernel-provided transport services.

3 · Encapsulation (Headers Added Per Layer)

Each layer adds its own header as data travels down the stack from sender to receiver:

HTTP Data + HTTP Header App layer
+ TCP/UDP Header Transport layer
+ IP Header Network layer
+ MAC Header Data link layer
+ Physical Header [+ Trailer] Physical layer
Some physical layer protocols also add a trailer (appended at the end of the frame) to mark the end of the frame, in addition to the header at the front.

As an application developer, you must specify which transport layer protocol (TCP or UDP) your application will use — this determines which header gets wrapped around your data.

4 · Why the Transport Layer is Needed

The Problem: Network Layer is Unreliable

The network layer provides best-effort datagram delivery — it tries to forward packets hop-by-hop using destination IP addresses, but it provides no guarantee that data actually arrives.

Reasons packets get dropped at the network layer:

Router buffer overflow Physical transmission errors Wireless channel interference Congestion
Routers maintain finite buffers. When the buffer fills up due to high network load, incoming packets are simply dropped. The network layer makes no attempt to recover them.

The Solution: Transport Layer Services

The transport layer sits on top of the unreliable network layer and provides end-to-end services that guarantee applications get what they need — reliability, ordering, flow control, and more.

Key principle: Applications should not be affected by packet loss at intermediate routers. The transport layer hides this unreliability from the application.
5 · Transport Layer Services

Connection Establishment

Before transferring data, the transport layer performs a "handshake" — analogous to saying hello at the start of a phone call. One host sends a hello to verify the other end is alive and ready. Once both ends acknowledge, a logical connection is established and data transfer begins. This ensures the receiver is available before the sender commits to sending data.

Reliable Data Delivery

The transport layer monitors whether each piece of data has been successfully received at the other end. If a packet is lost in transit, the transport layer detects the loss and retransmits the missing data — so the application eventually gets everything it was supposed to receive, regardless of what the network layer dropped.

Flow Control

Flow control addresses a speed mismatch between sender and receiver. Example: sender can transmit at 10 Mbps but receiver can only process at 1 Mbps. Without flow control, the sender floods the network with data the receiver can't handle — the excess is wasted and the network gets unnecessarily loaded.

Flow control forces the transmitter and receiver to agree on a transmission rate — the sender can only push data as fast as the receiver can consume it.

Congestion Control

Distinct from flow control — congestion control handles the network itself being overloaded, not just the receiver. Intermediate routers receive traffic from multiple paths simultaneously. If too many packets converge on one router, it becomes congested (analogous to a traffic junction with vehicles from multiple roads). The transport layer reduces the transmission rate to avoid contributing to this congestion.

Ordered Packet Delivery

In a packet-switched network, packets from the same stream may take different paths through the network and arrive out of order. Example: packet 3 arrives before packet 2 because it took a faster route. The transport layer assigns sequence numbers to packets and reorders them at the receiving end before delivering to the application. Applications always see data in the correct order.

The network layer delivers datagrams — the order they arrive is not guaranteed. The transport layer imposes order on top of this chaos.
6 · TCP vs UDP

The transport layer has two major protocol families. Which one to use depends entirely on what the application needs.

UDP — User Datagram Protocol
Model
Thin wrapper over the network (IP) layer
Services
Minimal — no connection, no reliability, no ordering, no flow/congestion control
Priority
Speed and low latency over correctness
Loss
Acceptable — application handles it
Use cases
Multimedia streaming, DNS, real-time protocols
TCP — Transmission Control Protocol
Model
Full-featured, connection-oriented protocol
Services
Connection establishment, reliable delivery, flow control, congestion control, ordered delivery
Priority
Correctness and completeness over speed
Loss
Not acceptable — retransmission guaranteed
Use cases
HTTP, FTP, file transfer, web browsing

Why UDP Exists — The Delay Trade-off

Every service the transport layer provides comes at a cost: processing delay. Implementing reliability, ordering, and congestion control introduces latency. For applications like video streaming, getting the frame within a deadline matters more than getting every frame perfectly — a missed frame can be approximated by averaging adjacent frames. Adding TCP's overhead would cause unacceptable delays. UDP eliminates this overhead entirely.

Multimedia protocols can tolerate loss up to a point — a missing frame can be interpolated from neighbours. But receiving data within a time deadline is non-negotiable. UDP serves this requirement; TCP does not.

Protocol Usage Summary

Application Layer Protocol Transport Layer Reason
HTTP TCP Web data must be complete and ordered
FTP TCP File transfer must be reliable
DNS UDP Short queries — speed critical, loss tolerable
Multimedia / RTP UDP Real-time, loss-tolerant, latency-sensitive
