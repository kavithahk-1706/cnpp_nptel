Lecture 08

Application Layer — IV
HTTP, HTML & TELNET

HyperText Transfer Protocol internals — architecture, messages, methods, status codes, headers, document types, and connecting via TELNET.

Recap from previous lectures

Client-server model: a server program listens on a specific port; clients on the same or different machine request that service. Every connection is uniquely identified by the 5-tuple:

Server IP + Server Port + Client IP + Client Port + Protocol ID

Change any one element → distinct connection. This is why two tabs in the same browser hitting the same server are separate connections and never bleed into each other.

Transport can be connection-oriented (TCP) or connectionless (UDP). Each layer only concerns itself with peer-to-peer communication at its own layer — reliability, ordering, etc. are handled by whichever layer owns them.
HTTP — Overview

HyperText Transfer Protocol

HTTP is the application-level protocol that allows web documents to be communicated across a network. It is the foundational protocol that makes the World Wide Web possible.

Web Server (HTTP Server)
Runs an HTTPD daemon listening on a port. Responds to requests. Also called: web server / HTTPD.
Web Browser (HTTP Client)
Constructs and sends HTTP requests, renders responses. Examples: Chrome, Firefox, IE. Synonymous with HTTP client.

Versions

VersionRFCKey Feature
HTTP 1.0RFC 1945Non-persistent connections by default
HTTP 1.1RFC 2616Persistent connections by default; multiple req/resp over one TCP connection

Key Properties

Lightweight Distributed Hypermedia support Transport-independent Default port 80 Primarily over TCP
Transport independence: HTTP doesn't mandate TCP in its spec — it's just the de-facto default. The protocol itself can technically run over any transport layer.
WWW Architecture

Distributed Client-Server Service

The World Wide Web is a distributed client-server system. An HTTP client (browser) accesses resources from HTTP servers anywhere on the internet. The parties involved:

Client / Browser
Sends requests. Renders returned documents. Supports protocols: HTTP, FTP, TELNET, SMTP, etc. via an internal controller.
Server (HTTPD)
Listens on a port (default 80). Executes scripts or returns static files. Sends back response with status code.
URL
protocol://host[:port]/path — port omitted if default (80). Identifies the resource to retrieve.
When you type www.iitk.ac.in, the browser: (1) resolves the name to an IP via DNS, (2) connects to port 80 by default, (3) sends an HTTP request. Specifying a different port: xyz.com:7126.

Cookies are also part of the HTTP ecosystem — they maintain session state, since HTTP is inherently stateless (each request-response pair is independent; no memory of previous exchanges).

Web Document Types

Static / Dynamic / Active

Static
Pre-written HTML file on disk. Request it, get the exact same file back every time. No computation at server.
Dynamic (Server-side)
A program runs at the server based on your request and generates the HTML response. Example: send a roll number → get that student's record. Technique: CGI (Common Gateway Interface), server-side scripting.
Active (Client-side)
Server returns code that executes in the browser. Examples: Java Applets, JavaScript. Used for client-side validation, dynamic UI updates without round-trips to server.
CGI (Common Gateway Interface): Classic server-side technique. Client request triggers a CGI program on the server; program dynamically builds an HTML page and returns it. Dynamic content changes based on input (e.g., student roll number → record lookup).
HTTP Message Format

Request & Response Structure

Request Message
Line 1 Request line: Method + URL + HTTP version
Lines 2+ Headers (key: value)
Blank Empty line (separator)
Body Optional — present in POST, PUT, etc.
Response Message
Line 1 Status line: HTTP version + Status code + Phrase
Lines 2+ Headers (key: value)
Blank Empty line (separator)
Body Optional — the requested document / CGI output

Response Status Code Classes

201 Created

5xx Server Error 500 Internal Server Error / 501 Not Implemented / 503 Service Unavailable
404 is the most commonly encountered error — document not found. It means you successfully reached the server but the requested resource doesn't exist there.
HTTP Headers

Header Format

All headers follow the pattern: Header-Name: <space> value

Headers are divided into three types:

General Headers
Apply to both requests and responses: Cache-Control, Connection (keep-alive or close), Date, MIME-Version, Upgrade.
Request Headers
Sent by client: Accept (media types client handles), Accept-Charset, Accept-Encoding, Accept-Language, Authorization, Host, User-Agent.
Response Headers
Sent by server: Accept-Ranges, Age (age of document), Public (supported methods), Retry-After, Server (server name + version).

Entity Headers

Describe the body/content of the message (used in both directions):

Allow Content-Encoding Content-Length Content-Range Content-Type Expires Last-Modified Location
This rich header set is why HTTP can handle heterogeneous media — text, images, video, audio — across a distributed, loosely coupled system with timeout management and caching.

Headers: Content-Type, Content-Length (4 header lines)

RESPONSE
Status line: HTTP/1.0 200 OK
Headers: 4 lines of headers

TELNET as HTTP Client

Accessing HTTP Servers with TELNET

Since HTTP is a plain-text protocol, you can manually drive it using TELNET:

Connect:telnet www.mhhe.com 80
Type:GET /somepath HTTP/1.0
Response:HTTP/1.0 200 OK + headers + body
TELNET works here because HTTP is text-based. What matters is that something sends a valid HTTP request to port 80 — the server doesn't care whether it's a browser or a TELNET session. This demonstrates that the transport channel is protocol-agnostic at the application layer.

TELNET itself is primarily a remote login protocol (covered in a subsequent lecture). Its use here is just to show raw HTTP interaction.

Exam Summary

Core Points to Remember

HTTP is stateless. Each request-response pair is independent. Cookies are the workaround to maintain session state across multiple exchanges.
Default port is 80. Other ports must be explicitly specified in the URL (e.g., xyz.com:8080).
HTTP 1.1 vs 1.0: 1.1 made persistent connections the default — multiple request-response exchanges can happen over a single TCP connection.
Document types: Static (file returned as-is) → Dynamic/CGI (server-side program generates HTML) → Active (client-side script executes in browser). Know which is server-side vs client-side.
Status code series: 1xx = info, 2xx = success, 3xx = redirect, 4xx = client error, 5xx = server error. 404 = Not Found (client error, not server error).
