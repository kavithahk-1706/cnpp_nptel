Protocol Stacks — Layered Services

▶ watch lecture
Lecture Overview

This lecture consolidates and extends the protocol stack concepts from Lectures 1 and 2, introducing new vocabulary that carries significant weight throughout the course: the three defining elements of any protocol (syntax, semantics, timing), the two interfaces every protocol must expose, the IETF standardization process, the hourglass model of internet architecture, and a first formal treatment of sockets. The per-layer function summary at the end is particularly examination-relevant.

Protocols — Refined Definition ↗ ~02:19

A protocol defines two distinct interfaces simultaneously: the interface between layers within the same system (vertical), and the interface between peer layers across different systems (horizontal). It is the fundamental building block of any network architecture. Every protocol object exposes exactly two interfaces:

Service Interface
Defines the set of operations this protocol exposes to the layer immediately above it — what services it provides, and how those services can be invoked by a higher-layer protocol.
Peer-to-Peer Interface
Defines how this protocol communicates with its counterpart on a remote machine — the message format, field sizes, message sequencing, and acknowledgement rules that the two instances must agree on.
A protocol also requires a module that implements these interfaces. The specification describes the required behavior; the module is the actual code or hardware that realizes it.

Protocols are standardized by the IETF (Internet Engineering Task Force). For a protocol to be formally included in the internet architecture, it must satisfy two criteria: (1) a written specification must exist, and (2) at least one — and preferably two — independent working implementations must be demonstrated.

Three Key Elements of Any Protocol ↗ ~05:28
Syntax
The structure and format of messages — data formats, field sizes, encoding schemes, signal levels. Defines the "grammar" of the protocol: what a valid message looks like.
Semantics
The meaning assigned to each message and each field within it — what control information is conveyed, how error conditions are signaled and handled. Defines the "vocabulary": what each message means and what action it requires.
Timing
Speed matching between sender and receiver, and sequencing constraints — ensuring messages are sent and processed in the correct order, and that transmission rates are compatible. Defines the "rhythm" of the exchange.
Every protocol at every layer of the stack can be analyzed through these three dimensions. This is a useful framework for examination questions that ask what a given protocol "defines" or "specifies."
Encapsulation — Revisited ↗ ~13:48

As data moves down the stack, each layer wraps the payload it receives from the layer above with its own header, adding control information relevant to that layer's function. Intermediate network devices (routers, switches) only unwrap headers down to the layer at which they operate — everything above that layer is treated as opaque payload. This is what makes the layered architecture composable: each layer functions independently without requiring knowledge of the protocols above or below it.

A critical distinction: peer layers communicate virtually, not physically. The Transport layer on Machine A does not send data directly to the Transport layer on Machine B. Physically, data travels down through all layers on Machine A, across the network medium, and back up through all layers on Machine B. Logically, however, Transport communicates with Transport, Network with Network, and so on — each layer's header is written for its peer on the other side and is invisible to all other layers.
Internet Architecture — The Hourglass Model ↗ ~24:52

The IETF's internet architecture does not enforce strict layering — an application is technically free to bypass the Transport layer and interact directly with IP if required. The architecture is described as an hourglass due to its characteristic shape when visualized:

many applications (HTTP, FTP, SMTP, DNS...)
TCP  |  UDP
IP — the waist
Ethernet · Wi-Fi · ATM · FDDI...
many physical networks

IP is the narrow waist of the hourglass: every application protocol above it must eventually pass through IP, and every physical network technology below it must be reachable via IP. The wide variety of application protocols at the top and physical network technologies at the bottom are unified by this single, common IP layer in the middle. This design is what allows the internet to accommodate heterogeneous hardware and applications simultaneously.

Each Layer's Function + Data Unit ↗ ~17:05

The Physical, Data Link, and Network layers run on every node in the network (hosts and intermediate routers). The Transport layer and above run only on end hosts — intermediate routers do not implement or process these layers.

L1
Physical
Transmits raw bits over a communication link by converting digital data to analog signals and back. The Physical layer has no concept of message structure or addressing — it is solely responsible for faithfully delivering individual bits from one node to the immediately adjacent one.
unit: bits
L2
Data Link
Collects a stream of raw bits into structured units called frames. Handles hop-to-hop delivery using MAC addresses — responsible for getting a frame from one device to the next directly-connected device. Implemented jointly by the NIC hardware and the device driver in the operating system.
unit: frames
protocols: Ethernet (IEEE 802.3) · Wi-Fi (802.11) · X.25 · ATM · FDDI
L3
Network (Internetwork)
Handles routing between networks — responsible for finding a path from the source to the destination across multiple intermediate hops. IP is connectionless and best-effort: it makes no guarantees about delivery, ordering, or timing. Reliability, flow control, and error recovery are intentionally delegated to higher layers.
unit: packets (also called IP datagrams)
protocols: IP · ICMP · IGMP · ARP · RARP
L4
Transport
Provides process-to-process communication — the ability to address individual applications running on a host, not just the host itself. Port numbers serve as process identifiers. Runs only on end hosts, not on intermediate routing nodes.
unit: messages (segments for TCP, datagrams for UDP)
TCP — connection-oriented, reliable, flow control, congestion control, duplicate suppression · UDP — connectionless, unreliable, best-effort, minimal overhead
L5–7
Application (+ Session + Presentation in OSI)
The layer directly interacted with by end users and applications. In the OSI model: the Session layer manages and terminates connections between cooperating applications; the Presentation layer handles data format translation and encryption; the Application layer is the interface for user-facing network programs. In TCP/IP all three are collapsed into one.
unit: messages / data
HTTP · FTP · SMTP · Telnet · DNS · SNMP
Sockets — First Introduction ↗ ~28:12
A socket is the programming interface between the Application layer and the Transport layer. A networked application is uniquely identified by two components: its IP address (which machine on the network) and its port number (which process on that machine). A socket is the abstraction that combines these two — it is the endpoint through which an application sends and receives data. Full socket programming is covered later in the course; this is the conceptual introduction.

This is why connecting to a website involves more than just an IP address: the browser connects to port 80 (HTTP) or port 443 (HTTPS) on the server's IP. The port number tells the operating system's networking stack which application process should receive the incoming data.

TCP vs. UDP — Quick Reference ↗ ~29:08
TCP
Connection-oriented · Reliable delivery · Duplicate suppression · Congestion control · Flow control. Use when data integrity and completeness are required: file transfers, web pages, email, database queries.
UDP
Connectionless · Unreliable · Best-effort · Minimal overhead · Fast. Use when throughput or latency matters more than guaranteed delivery: video streaming, VoIP, DNS lookups, online gaming, live broadcasts.
Network API ↗ ~26:01

Most network protocols are implemented as software within the operating system kernel. The standardized interface through which applications request network communication services from the OS is called the Network API. The socket API is the most widely used implementation of this interface and is available in virtually every modern operating system. When writing any networked application, the developer interacts with the OS through the network API — the underlying protocol implementation is handled by the kernel.
