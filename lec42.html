Router Architecture — Recap

A router is functionally split into two planes. Understanding this split is essential before examining the hardware.

Control Plane
runs onRoute Processor (CPU + Memory)
jobExecutes routing protocols (Distance Vector, Link State, BGP) to build the routing table
softwareRouter OS (e.g., Cisco IOS) — runs as standard software
outputPopulated routing table
Data Plane (Datapath)
runs onTCAM hardware (dedicated silicon)
jobPer-packet processing — match destination IP → look up FIB → forward to outgoing interface
speedMust handle millions of packets/second; must be hardware-accelerated
inputFIB — Forwarding Information Base (snapshot of routing table loaded into TCAM)
FIB vs Routing Table: The routing table lives in software (control plane). A snapshot of it is pushed into the TCAM hardware as the Forwarding Information Base (FIB). The datapath only reads the FIB — it never touches the routing table directly.

The basic hardware components are: Route Processor (CPU + memory) at the top, connected via an internal interconnection network bus to multiple interface cards which serve as the physical input/output ports of the router.

Ternary Content Addressable Memory (TCAM)

What is TCAM?

TCAM is a specialised high-speed hardware memory that searches its entire contents in a single clock cycle. It is the hardware substrate on which the FIB is implemented in a router's datapath.

The name "ternary" comes from the fact that each bit cell stores one of three values:

0 — matches 0 1 — matches 1 X — don't care / wildcard (matches either)

Searching is based on pattern matching, not address lookup. For example, the pattern 110X matches both 1100 and 1101, because the fourth bit is a don't-care.

CAM vs RAM — Inverse Access Principle

Property RAM CAM / TCAM
Access method Provide address → get content Provide content → get address
Direction Address-in, Data-out Data-in, Address-out
Search speed Sequential / random-access, slower for lookup Entire memory searched in one clock cycle
Router use General-purpose storage FIB lookups in datapath

TCAM for IP Prefix Matching

Routing table entries are stored as prefix + mask pairs. In TCAM, the host bits (masked-out bits) are stored as X (don't care). Only the network prefix bits are matched precisely.

Key insight: At intermediate routers, only the network address part of a destination IP is matched. The host address part is only needed at the last-hop router (the one directly connected to the destination's LAN) for final delivery.

Example with a simplified 4-bit IP address space:

Prefix/Mask TCAM Format Meaning
101 / 3 101X Match anything starting with 101
111 / 3 111X Match anything starting with 111
10 / 2 10XX Match anything starting with 10
0 / 0 XXXX Default route — matches everything

Longest Prefix Match in TCAM

When multiple TCAM entries match a given destination IP, the router applies the longest prefix match rule: the entry with the most specific (longest) prefix wins. An encoder circuit within the TCAM hardware compares all simultaneous matches and returns the address of the longest-matching entry.

Example: Search word 1011 matches XXXX, 10XX, and 101X. Longest match = 101X (3 bits matched, 1 don't-care) → this entry's output (next-hop + interface) is used.

Linux IP & Route Commands

All commands demonstrated on a Linux machine with three interfaces: lo (loopback), eth0 (Ethernet, currently down), wlan0 (wireless, up).

ip address show 19:11

Shows all network interfaces with their full IP configuration.

FieldMeaning
inetIPv4 address with prefix length (CIDR notation)
inet6IPv6 address
link/etherMAC (hardware) address
brdBroadcast address
mtuMaximum Transmission Unit — largest packet size in bytes the interface will accept
lo→127.0.0.1/8Loopback — same-machine comms; MTU 65536; used to test if your protocol stack is live
eth0→10.0.0.11/24Wired Ethernet — currently DOWN (no cable); MTU 1500
wlan0→10.146.58.130/17Wireless — currently UP; MTU 1500; has both IPv4 and IPv6
MTU 1500 bytes is the standard maximum for both Ethernet and Wi-Fi frames. The loopback interface has an unusually large MTU (65536) because it never leaves the machine — no physical medium limitations apply.

ip link show 23:52

Shows link-layer properties only (no IP addresses). Key fields:

FieldMeaning
qdiscQueuing discipline — the queueing algorithm used on this interface
qlenQueue length — max packets that can be buffered before dropping
stateUP / DOWN / UNKNOWN — current link state
pfifoPriority FIFO queue — used on Ethernet; qlen 1000 packets
mqMulti-queue — used on WLAN (IEEE 802.11); supports 4 QoS classes
The loopback has qlen 1 and no queue — packets are delivered immediately within the same machine. Ethernet and WLAN both have qlen 1000 with different queuing mechanisms suited to their medium.

ip -s address show 26:48

Adds per-interface traffic statistics to the output. Reports for both RX (receive) and TX (transmit):

bytes packets errors dropped overrun multicast carrier (TX only) collisions (TX only)

Useful for diagnosing interface health — a high error or drop count indicates link-layer problems.

Displays the kernel IP routing table. Each entry contains:

ColumnMeaning
DestinationTarget network IP
GatewayNext-hop IP address (0.0.0.0 = directly connected)
GenmaskSubnet mask for this route
FlagsU = Up, G = Gateway, H = Host route

💻 Live Demo: The routing table output and the effect of adding/deleting routes is shown at 28:40 – 38:28. Watching this is the fastest way to understand how routing table entries translate to actual forwarding behaviour.

ip route add <dst> dev <iface> via <gateway>

Adds a static route entry to the routing table. Requires root/sudo. If no netmask is provided, defaults to /32 (host route — exact match only). Example:

sudo ip route add 172.16.2.30/24 dev wlan0 via 10.146.0.2
Adds a route to the 172.16.2.0/24 network via gateway 10.146.0.2 on the wlan0 interface.

These manually entered routes are called static routes. They persist until deleted with sudo ip route del <dst>.

traceroute <destination> 34:18

Traces the path (hop by hop) that packets take to reach a destination. Shows each intermediate router's IP and the round-trip time to that hop.

If a hop is unreachable or the gateway is incorrect, all subsequent hops show * * * (no response). This confirms that the routing table entry — specifically the gateway — directly controls packet forwarding path.
💻 Traceroute Demo: Two traceroutes are shown at 34:18 – 38:28 — one to a reachable host (shows 3 intermediate hops with RTTs) and one to an unreachable host with a wrong gateway (all stars). Clear demonstration of gateway misconfiguration impact.
Concept Summary

TCAM

Specialised hardware memory that searches all entries simultaneously in one clock cycle using ternary (0/1/X) pattern matching. Used in router datapaths to implement the FIB. Longer prefix matches take priority via an encoder circuit — this is the hardware implementation of longest prefix match.

Control Plane vs Data Plane

The control plane runs routing protocols in software to build the routing table. The data plane uses TCAM hardware to perform per-packet FIB lookups at line rate (millions of packets/second). The FIB is a hardware-optimised snapshot of the routing table.

Static vs Dynamic Routing

Commands like ip route add create static routes — manually configured entries that don't change unless explicitly modified. Dynamic routing (Distance Vector, Link State, BGP — covered in subsequent lectures) automatically populates and updates the routing table based on network topology.
