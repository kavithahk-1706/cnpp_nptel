Lecture 33 HIGH PRIORITY

Internet QoS — III
Traffic Policing & Traffic Shaping

Algorithms for regulating output rate and enforcing traffic contracts: leaky bucket, token bucket, and the playout buffer extension.

▶ Watch on NPTEL
Quick Recap

Traffic Policing vs Traffic Shaping 00:26

Both mechanisms enforce a traffic contract but differ in what they do with excess traffic:

Traffic Policing
Identifies peaks that violate the QoS contract and drops those packets immediately.
Output is still bursty — peaks are simply cut off.
Implemented via the bucket capacity (τ) in leaky bucket.
Traffic Shaping
Absorbs bursts and smooths the output rate to a steady flow.
Output appears nearly constant over time.
Implemented via the constant drain rate (r) in leaky bucket.
In practice, achieving a perfectly constant bit rate across multiple routers is difficult. Real implementations combine both policing and shaping — the leaky bucket and token bucket are hybrid mechanisms.
Algorithm 1

Leaky Bucket Algorithm 02:23

Analogy: a physical bucket with a small hole at the bottom. Water (packets) pours in at any rate; it drains at a constant rate determined by the hole size. If you pour in more than the bucket can hold, the excess overflows (is dropped).

Mechanics

Incoming packets enter a packet queue (the bucket). A single server services this queue at a constant rate r (packets/sec or bits/sec). The queue has a fixed maximum capacity τ.

Parameter
r — drain rate

Constant rate at which packets exit the queue. Controls traffic shaping.

Parameter
τ — bucket capacity

Maximum queue depth. Packets arriving when queue is full are dropped. Controls traffic policing.

Input
R — incoming rate

Variable; can be bursty. Absorbed by the queue up to capacity τ.

Output
≤ r (capped)

Never exceeds r. Drops to zero if queue empties (application generates nothing).

Dual Role

Shaping → constant drain rate r Policing → drop when queue full (τ exceeded) No burst support — always one-by-one
The output dips below r only when the application itself isn't generating data — not a failure of the algorithm. The output never exceeds r.

Algorithm 2

Token Bucket Algorithm 08:22

Extends the leaky bucket by introducing traffic burst support. Unused bandwidth is "saved" as tokens, which can be spent later to send a burst of packets all at once.

Components

Token Bucket
size = b tokens

Logical container. Tokens are virtual — just a counter in software. Filled at rate r tokens/sec. Capped at b; excess tokens discarded.

Packet Queue
incoming packets

Holds arriving packets waiting to be dispatched. A scheduler monitors both the token bucket and this queue.

Scheduler Rule
token → send packet

One token consumed per packet sent. If no token available, packet waits. If tokens available, multiple packets can be sent at once.

Output Rate
R(t) = min(Pt, b + rt)

Pt = incoming packet rate. b + rt = cumulative tokens available. Output is bounded by whichever is smaller.

How Bursting Works

When no packets arrive, tokens accumulate in the bucket (up to max b). When a burst of packets arrives, all accumulated tokens are spent immediately — sending up to b + (tokens generated since) packets at once rather than one-by-one. This is the key difference from leaky bucket.

Use case: Buffered video streaming. When the network is idle, tokens accumulate. When a burst of video data arrives, it can be sent immediately to fill the player's playout buffer — instead of being artificially throttled one packet at a time.

Maximum Burst Size (MBS) Derivation 16:35

The cross-point between the incoming packet line and the token line gives the maximum burst moment:

Incoming packets → Y = Pt (cumulative packets received)
Token availability → Y = b + rt (start full at b, grow at rate r)
Cross-point (max burst) → Pt₁ = b + rt₁ → t₁ = b / (P − r)
MBS → MBS = Pb / (P − r)

Where P is the peak incoming rate, r is the token generation rate, and b is the bucket size. The MBS is the maximum number of packets that can be transmitted in a single burst.

Extension

Playout Buffer

Both leaky and token bucket can still produce dips in output rate when the application generates less data than r. For applications requiring a perfectly smooth output, a playout buffer is added in front of the traffic shaper.

The playout buffer introduces additional delay to packets that arrive too early, holding them back until they are needed. This "fills in" the dips in output rate. Used in strict real-time applications where jitter must be minimised.
Application generates variable-rate data
Playout Buffer adds delay to early packets
Traffic Shaper leaky / token bucket
Output Queue near-constant rate to network
Summary

Key Takeaways

Leaky Bucket: Single queue drained at constant rate r, capacity τ. Enforces both policing (drop when full) and shaping (constant output). No burst support — packets always exit one by one.
Token Bucket: Virtual tokens generated at rate r, bucket size b. A packet is sent only when a token is available. Allows burst up to MBS = Pb/(P−r). Output rate R(t) = min(Pt, b+rt).
Playout Buffer: Optional addition that holds early-arriving packets to smooth out dips. Used in strict jitter-sensitive applications.
